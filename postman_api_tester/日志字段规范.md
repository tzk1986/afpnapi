# 日志字段规范（v1.3.0）

发布日期：2026-05-13
文档定位：统一结构化日志字段、敏感字段约束与健康告警口径。

## 1. 基础字段

所有结构化日志由 `logging_utils` 输出，建议至少包含以下字段：

1. `timestamp`：UTC 时间戳（JSON 格式日志）。
2. `level`：日志级别（DEBUG/INFO/WARNING/ERROR）。
3. `logger`：模块名（Python logger 名称）。
4. `message`：日志消息。
5. `event`：事件名，必须遵循 `domain.action.state`。

## 2. 公共业务字段

按场景附带以下字段，缺失时允许为空字符串：

1. `job_id`：任务 ID。
2. `report_name`：报告名。
3. `request_id`：请求追踪 ID（来自 header 或服务端生成）。
4. `method`：HTTP 方法。
5. `path`：请求路径。
6. `status_code`：HTTP 状态码。
7. `duration_ms`：请求耗时。
8. `api_name`：接口名。
9. `status`：业务状态（PASSED/FAILED/ERROR 等）。
10. `response_time_ms`：接口响应耗时。

## 3. 敏感字段约束

### 3.1 禁止明文落日志的字段

1. 认证头：`authorization`、`token`、`x-token`、`x-access-token`、`access-token`、`cookie`、`set-cookie`。
2. 密钥类字段：`secret`、`api-key`、`apikey`、`password`。
3. 会话类字段：`session`、`x-csrf-token`。

### 3.2 处理规则

1. 允许记录字段名，不允许记录明文值。
2. 值统一掩码为 `***`。
3. 需要排障时，优先通过请求 ID 回查上游系统，不在本系统日志降级暴露明文。

## 4. 告警字段与 `/health` 扩展

`/health` 新增 `log_alert` 字段，结构如下：

1. `status`：`ok` 或 `warning`。
2. `alert_triggered`：是否触发告警。
3. `metric`：固定为 `error_rate_per_min`。
4. `window_seconds`：统计窗口（秒）。
5. `threshold`：阈值（每分钟 ERROR 条数）。
6. `current`：当前窗口折算后的每分钟 ERROR 速率。
7. `error_count`：窗口内 ERROR 总数。

## 5. 配置项

在 `postman_api_tester/config.py` 中统一配置：

1. `LOG_LEVEL`
2. `LOG_FORMAT`
3. `LOG_SAMPLE_RATE`
4. `LOG_ALERT_ERROR_WINDOW_SECONDS`（默认 300）
5. `LOG_ALERT_ERROR_RATE_THRESHOLD_PER_MIN`（默认 10）

## 6. 告警策略（简版）

1. 统计窗口：最近 `LOG_ALERT_ERROR_WINDOW_SECONDS` 秒。
2. 速率计算：`error_count / window_seconds * 60`。
3. 触发条件：`current >= threshold`。
4. 输出位置：`GET /health` 的 `log_alert` 字段。

该策略用于轻量级运行态探测，不替代外部告警平台。建议生产环境在 ELK/Prometheus 中二次配置告警与通知渠道。
