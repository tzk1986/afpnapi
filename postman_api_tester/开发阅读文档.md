## 1. 开发前必读条件

开始任何开发前，必须先完整阅读以下文件并确认当前改动是否触及其中约束：

1. `postman_api_tester/操作手册.md`
2. `postman_api_tester/快速命令参考.md`
3. `postman_api_tester/config.py`
4. `report_server.py`
5. `postman_api_tester/postman_api_tester.py`

未完成上述阅读，不允许直接修改核心逻辑（token 处理、报告生成、报告服务接口、分页与筛选行为）。

## 2. 开发流程约束

方案讨论与实施分离：

1. 新功能开发分为“方案讨论”和“实施”两个阶段。
2. 方案讨论阶段禁止修改代码文件（包括 `report_server.py`、`templates/*.html`、`postman_api_tester/*.py`）。
3. 只有收到用户明确口令“开始实施”后，才允许进入代码修改、测试与提交阶段。
4. 方案讨论期间仅允许：读取文件、分析代码结构、输出方案文档、更新约束文档。
5. 新增方案文档命名格式：`YYYY-MM-DD-方案名称.md`，放在 `postman_api_tester/方案讨论`。
6. 实施前必须确认：业务范围、涉及页面、对原功能的影响。
7. 实施前需对本次讨论方案进行合并梳理，避免遗漏。
8. 实施后可复用的验证文件保留到 `test_data`，并在操作手册记录复用方式。

## 2.5 目录结构约束（新增文件必须遵守）

新增代码文件前，必须先确认当前模块归属层级，禁止随意乱放：

```text
postman_api_tester/
├─ core/                    # 核心执行层
│   ├─ checkpoint_manager.py     # 断点恢复管理
│   ├─ execution_helpers.py      # 执行辅助（请求发送、断言评估等）
│   ├─ html_reporter.py          # HTML 报告生成
│   ├─ pipeline.py               # 执行管道编排
│   ├─ report_engine.py          # 报告引擎
│   └─ types.py                  # 共享类型定义
├─ handlers/                # 路由编排层（仅允许路由装饰器与参数校验）
│   ├─ *_routes.py               # Flask 路由委托函数（新增路由必须在此创建）
│   ├─ base_handler.py           # 路由基类（错误响应统一封装）
│   └─ report_analytics_handler.py  # analytics 业务聚合（保留）
├─ services/                # 领域服务层（业务逻辑）
│   ├─ base_service.py           # 服务基类
│   ├─ report_*_service.py       # 报告相关业务服务
│   └─ report_analytics_service.py  # analytics 数据服务
├─ utils/                   # 通用工具层（无业务状态）
│   ├─ url_utils.py              # URL 处理
│   ├─ request_builder.py        # 请求体构建
│   ├─ response_parser.py        # 响应解析
│   ├─ collection_utils.py       # Collection 处理
│   ├─ logging_utils.py          # 结构化日志
│   ├─ security.py               # 脱敏与安全
│   └─ ...                       # 其他纯工具函数
├─ tests/                   # pytest 单元测试（按被测模块命名）
│   ├─ test_*_routes.py          # handler 路由测试
│   ├─ test_base_handler.py      # 基类测试
│   ├─ test_base_service.py      # 服务基类测试
│   └─ ...                       # 其他单元测试
├─ postman_api_tester.py    # CLI 执行入口（保持不变）
├─ report_server.py         # 路由装配模块（仅装饰器+委托）
├─ report_server_app.py     # 应用工厂（Flask app 创建与启动）
├─ report_server_config.py  # 配置读取工具
├─ config.py                # 业务配置常量
├─ models.py                # 共享数据模型
├─ exceptions.py            # 自定义异常
├─ assertions.py            # 断言引擎
├─ parser.py                # Postman Collection 解析
├─ executor.py              # 测试执行器
├─ session.py               # HTTP 会话管理
├─ auth.py                  # 认证与 Token 获取
├─ db_feedback.py           # 数据库反馈解析
└─ ...
```

分层规则：

1. **core/**：执行链路核心逻辑（HTML 报告生成、断点恢复、执行管道、类型定义）。禁止引用 Flask 或 handlers/services。
2. **handlers/**：仅允许接收 HTTP 请求、参数校验、调用 services、返回 Response。禁止直接操作文件系统或发 HTTP 请求。
3. **services/**：承担业务逻辑（导出、重试、列表、判定、锁、提交等）。允许调用 utils 和 repositories，禁止直接返回 Flask Response。
4. **utils/**：纯函数工具（URL 合并、请求构建、响应解析、脱敏、缓存）。禁止持有业务状态，禁止引用 handlers/services。
5. **tests/**：按 `test_<被测模块>.py` 命名。handler 测试必须使用 `test_request_context()`，services 测试允许 mock 依赖。
6. **新增路由**：必须先创建 `handlers/<name>_routes.py`，再在 `report_server.py` 中以 `return _handler_name(args)` 形式注册。
7. **新增服务**：必须创建 `services/<name>_service.py`，如有共享逻辑可抽象为 `base_service.py` 子类。

## 3. 当前有效约束清单（去重归并）

### 3.1 运行与环境约束

1. Python 版本要求：3.10+。
2. 依赖安装入口：`requirements.txt`。
3. 建议在项目根目录执行命令：`d:/tangzk/py/seldom-api-testing`。
4. `POSTMAN_TOKEN`、`POSTMAN_BASE_URL` 支持环境变量注入，优先于 `config.py` 默认值。
5. 报告服务端口/主机支持环境变量：`REPORT_SERVER_PORT`（默认 5000）、`REPORT_SERVER_HOST`（默认 0.0.0.0）。
6. 生产环境优先 `waitress`，未安装时降级 Flask 开发服务器；禁止生产 `debug=True`。

### 3.2 执行参数与安全约束

1. CLI 参数顺序：
   `python -m postman_api_tester.postman_api_tester <postman_file> [base_url] [output_dir] [token] [results_per_page]`
2. token 优先级：命令行/代码入参 > `POSTMAN_TOKEN` > `config.py::TOKEN` > 自动登录。
3. base_url 优先级：命令行/代码入参 > `POSTMAN_BASE_URL` > `config.py::BASE_URL` > 从 Collection 解析。
4. output_dir 优先级：命令行/代码入参 > `config.py::REPORT_OUTPUT_DIR` > 项目根目录 `reports/`。
5. `results_per_page` 默认 30，建议范围 1~100。
6. 请求超时由 `config.py` 集中管理（`REQUEST_CONNECT_TIMEOUT`、`REQUEST_READ_TIMEOUT`），执行层禁止硬编码。
7. token 并发隔离必须保持实例级（`_auth_token`），禁止回退为类变量。
8. URL 解析必须兼容 `request.url.path` 缺失时回退 `request.url.raw`，并支持绝对 URL 与 `{{baseUrl}}/...`。
9. SSRF 防护必须保留：`run_postman_tests`、`/api/run-postman`、`/re-request-api`、`/api/proxy-request`、`/api/run-ad-hoc-tests` 的目标 URL/base_url 仅允许 `http/https`。

### 3.3 报告产物约束

每次执行必须生成同名前缀三件套：

1. `postman_report_时间戳.html`
2. `postman_report_时间戳_details.json`
3. `postman_report_时间戳_meta.json`

执行中断（部分成功）时也必须生成三件套，且使用新时间戳，禁止覆盖历史报告。

### 3.4 报告中心核心约束（report_server.py）

1. 默认监听 `0.0.0.0:5000`。
2. 报告目录优先级：`POSTMAN_REPORTS_DIR` > `REPORTS_DIR` > `config.py::REPORT_OUTPUT_DIR` > 项目根目录 `reports/`。
3. 单报告页分页默认 20、上限 100。
4. 支持历史对比（新增/移除/状态变化/成功率变化）。
5. 同报告并发写回必须使用“按报告名粒度”的写锁。
6. 上传文件名入库前必须清洗（仅保留字母/数字/汉字/`-._()（）【】`）。
7. `RUN_JOBS` 上限由 `config.py::RUN_JOBS_MAX` 管理（默认 200），不可移除清理逻辑。
8. `list_reports()` 必须维持 30 秒 TTL 缓存，并在新报告生成时执行 `_invalidate_reports_cache()`。
9. 首页仅加载摘要字段，禁止注入完整 `results/request_info/response_info`。
10. `/health` 必须保留 `status`、`timestamp`，允许扩展 `log_alert` 等运行态字段。
11. 结构化日志必须使用 `logging`，核心路径禁止新增 `print()` 代替 `logger.*`。
12. 页面渲染必须使用 `templates/` 外置模板；不支持内嵌模板回退分支。
13. 报告删除必须限定在 `REPORTS_DIR` 内，并同步清理 `.html`、`_details.json`、`_meta.json`、`_page_*.html`。
14. 报告结果详情允许人工判定（成功/失败/恢复自动），需保留 `manual_judgement/judgement_history` 并同步重算 summary；重试真实回写后取消人工判定激活状态。
15. 日志告警阈值由 `LOG_ALERT_ERROR_WINDOW_SECONDS`、`LOG_ALERT_ERROR_RATE_THRESHOLD_PER_MIN` 集中管理，并通过 `/health.log_alert` 对外可观测。
16. analytics 能力受 `ENABLE_REPORT_ANALYTICS` 集中控制；关闭时页面不注入 analytics 主容器，`/api/report-analytics/<report_name>` 与 `/api/report-analytics-compare` 必须返回 403。
17. analytics 查询参数 `top_n`、`trend_limit` 必须按配置默认值与上限自动钳制；`include_samples` 必须保留布尔兼容解析。
18. analytics compare 必须保持新增接口形态，不得破坏既有 `/api/compare` 语义。
19. analytics 图表可视化受 `ENABLE_REPORT_ANALYTICS_CHARTS` 集中控制，且必须受父开关 `ENABLE_REPORT_ANALYTICS` 联动约束；父开关关闭时，无论子开关为何值，页面均不加载 Chart.js、不渲染图表 canvas。
20. 图表 CDN 加载失败时必须自动降级为纯表格，不得阻断页面其他功能。
21. **路由层提取约束（v1.3.0）**：`postman_api_tester/report_server.py` 仅保留 Flask 路由装饰器与单行委托调用，禁止在路由函数内直接编写业务逻辑；全部业务逻辑必须下沉至 `handlers/*` 对应模块。新增路由必须先创建 handler 函数，再在 report_server.py 中以 `return _handler_name(args)` 形式委托。
22. **类型检查约束（v1.3.0）**：handlers/ 与 services/ 新增代码必须通过 `mypy --strict` 检查；禁止新增 `Any` 类型扩散，禁止新增未标注返回类型的公有函数。
23. **测试覆盖约束（v1.3.0）**：handlers/ 下新增路由必须配套单元测试，异常路径（400/403/404）与成功路径均需覆盖；测试使用 `test_request_context()` 提供请求上下文，禁止在 handler 模块中调用 `configure_run_jobs()` 等全局状态初始化函数。
24. **启动入口约束（v1.3.0）**：`postman_api_tester/report_server.py` 本身不提供命令行入口（无 `if __name__ == '__main__':`），禁止直接以 `python postman_api_tester/report_server.py` 或 `python -m postman_api_tester.report_server` 方式启动。唯一支持的启动方式为根目录 `python report_server.py`（兼容启动器内部调用 `ReportServerApp.run_app(app)`），或显式执行 `python -c "from postman_api_tester.report_server import app; from postman_api_tester.report_server_app import ReportServerApp; ReportServerApp.run_app(app)"`。

### 3.5 导入、导出与脱敏约束

1. 导出 JSON 必须为单一文件，并按规则合成：原始集合（请求参数回填） + `manual_cases` - `manual_exclusions`。
2. 导出范围支持 `full` 与 `report_only`，默认与开关由 `config.py` 统一管理；禁用 `report_only` 时自动降级 `full`，禁止 500。
3. 导出必须保留完整请求语义：`method/url/headers/params/body`，以及 `body_mode/body_data`。
4. 导入链路必须可解析导出结构（含 `request.url.raw/path/query` 与 body 各模式），保证“可预览可执行”。
5. 敏感头规则由 `config.py::SENSITIVE_HEADERS` 与内置集合合并统一生效：
   - 详情写入时掩码 `***`
   - 导出 `include_auth=false` 时按同一规则剔除
   - 禁止不同链路使用不同敏感头集合
6. 流式导出必须通过新增接口 `/api/export-collection-stream` 提供，不得替换 `/api/export-collection`。
7. 导出通道仅允许 `REPORT_EXPORT_CHANNEL_MODE=auto|legacy|stream`；非法值回退 `auto`。
8. `auto` 模式按 `REPORT_EXPORT_STREAM_THRESHOLD` 分流，流式失败自动回退旧接口；`legacy/stream` 为强制通道。

### 3.6 人工用例、筛选与路由一致性约束

1. 人工用例写入 `_meta.json::manual_cases`，默认目录由 `MANUAL_CASE_FOLDER_NAME` 提供且页面可覆盖填写。
2. `manual_cases` 必须有稳定 `id`，更新/删除按 `id` 定位，禁止用 `name/url` 作为唯一键。
3. 排除能力必须支持“置灰 + 显示/隐藏已排除”开关。
4. 路由层新增关键字参数时，必须同步更新被调函数签名、返回字段、模板消费字段。
5. 执行层 `FAILED/ERROR` 结果必须输出 `db_feedback` 结构化字段，不得降级为仅日志可见。

### 3.7 ad-hoc 执行约束

1. 首页通过“新增接口测试”进入独立页执行，能力受 `ENABLE_ADHOC_RUN`、`ADHOC_MAX_ITEMS`、`ADHOC_DEFAULT_COLLECTION_NAME` 控制。
2. `/api/run-ad-hoc-tests` 仅接收 JSON；`cases` 必须为非空数组，数量不超过 `ADHOC_MAX_ITEMS`。
3. 入队前必须规范化为 Postman Collection v2.1 JSON，写入 `uploaded_collections/<job_id>.json`，并复用现有任务状态与报告链路。
4. ad-hoc URL 仅允许三类：绝对 `http/https`、`{{baseUrl}}/...` 或 `{{base_url}}/...`、配合 `base_url` 的相对路径。
5. body 模式统一支持 `none/raw/urlencoded/formdata/graphql/binary`，不得与导入/重试链路语义漂移。

### 3.8 前端一致性与调试回归约束

1. 所有 UI 改动必须在 `templates/*.html` 实现。
2. 同类页面优先复用既有设计语言；如引入新风格，必须同步评估并更新同类旧页面。
3. 声明“页面已统一”的能力（tooltip/复制/折叠/筛选提示）必须核对全部真实渲染入口，禁止只改单分支即宣告收口。
4. `Ctrl+C/Cmd+C` 接管仅允许在浮层可见且非输入态（input/textarea/select/contenteditable）下生效。
5. 调试前必须先检查目标端口占用，必要时先清理旧进程再启动最新服务；验证后必须关闭本次启动的服务进程。
6. 涉及 `report_server.py` 路由/筛选/导出/人工用例改动，提交前至少验证：
   - `/health`
   - `/`
   - `/report-view`
   - `/api/report-results?include_excluded=true/false`
   - `/api/manual-cases` add/update/delete
   - `/api/export-collection`
   - `/api/report-analytics/<report_name>`
   - `/api/report-analytics-compare`
7. 问题排查默认流程：日志定位 -> 最小复现 -> true/false 双态验证 -> CRUD 闭环 -> 回滚清理。

### 3.9 断点恢复与类型检查约束

1. Checkpoint 恢复键必须使用稳定 `item_path`，禁止使用 `name/url` 去重。
2. Checkpoint 写入必须原子写（tmp + replace）；读取前必须结构校验；校验失败降级全量执行，不中断主流程。
3. `ENABLE_CHECKPOINT_RECOVERY` 默认 `False`，未显式开启时保持与 v1.2.0 等价行为。
4. mypy 必须使用：
   `python -m mypy --config-file pyproject.toml --explicit-package-bases`
5. CI 门禁需持续执行同一 mypy 命令。

### 3.10 方案实施治理约束（61-67 归并）

1. 方案进入实施准备后必须冻结迭代范围并标注 Out of Scope。
2. 新增需求/接口结构调整/跨层重构/性能策略调整需变更单审批后实施。
3. 实施按“阶段目标 -> 门禁验证 -> 下一阶段”推进，门禁失败仅允许修复。
4. 单 PR 仅覆盖单一阶段目标，建议后端 <= 6 文件、前端 <= 3 文件，超限需拆分。
5. 既有稳定接口语义（如 `/api/reports`、`/api/report-results`、`/api/report-meta`、`/api/compare`）不得直接破坏，需新增接口灰度迁移。
6. 新增能力必须提供集中配置开关，默认值优先保持旧行为；开关关闭时页面不注入新能力容器、不发新请求。
7. 每阶段合并前必须沉淀“五段式验收记录”：已完成项/未完成项/风险与遗留/回滚验证/回归结果。

### 3.11 局域网访问约束

1. 本机访问：`http://127.0.0.1:5000`。
2. 局域网访问需放通 5000 端口。
3. 推荐脚本：`allow_report_server_firewall.ps1`。

```powershell
$doc = "postman_api_tester/开发阅读文档.md"
$ts = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
$pattern = "优先级|默认|必须|限制|上限|下限|分页|token|base_url|report_output_dir|POSTMAN_REPORTS_DIR|REPORTS_DIR|5000|status|meta|details"
$files = @(
  "postman_api_tester/config.py",
  "postman_api_tester/postman_api_tester.py",
  "report_server.py",
  "postman_api_tester/README.md",
  "postman_api_tester/操作手册.md",
  "postman_api_tester/快速命令参考.md",
  "postman_api_tester/最终交付清单.md"
)
"`n## 约束更新记录（$ts）`n" | Add-Content $doc
rg -n -i $pattern $files | Add-Content $doc
"`n" | Add-Content $doc
```
