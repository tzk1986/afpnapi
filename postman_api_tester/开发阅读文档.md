## 2. 开发流程约束（新增）

**方案讨论与实施分离约束**：
- 新功能开发分为"方案讨论"与"实施"两个阶段
- 在方案讨论阶段，禁止对任何代码文件进行修改（包括 `report_server.py`、`templates/*.html`、`postman_api_tester/*.py` 等）
- 只有收到用户明确的口令 **"开始实施"** 后，才允许进入代码修改、测试与提交阶段
- 方案讨论期间仅允许：读取文件、分析代码结构、输出方案文档、更新约束文档
- 方案讨论后，新增的方案文档名称增加日期，格式为 `YYYY-MM-DD-方案名称.md`，并存储在 `postman_api_tester/方案讨论` 目录下
- 方案讨论后最终实施前需要确认，方案实施涉及的业务范围，修改的页面有哪些，涉及到原有的哪些功能，需要做确认
- 方案实施前，需要把本次讨论生成的所有方案进行整理合并到一起，统一实施，不应该有实施遗漏
- 方案实施后，用于验证的文件，如果可以复用，进行保留，并存储在 `test_data` 目录下，并在操作手册中记录，验证文件如何使用，以便后续复用

## 3. 当前有效约束清单

### 2.1 运行与环境约束

1. Python 版本要求：3.10+
2. 依赖安装入口：requirements.txt
3. 建议在项目根目录执行命令：d:/tangzk/py/seldom-api-testing
4. **TOKEN 支持环境变量注入**：设置 `POSTMAN_TOKEN` 环境变量可覆盖 config.py 的默认 TOKEN；未设置则为空字符串（触发自动登录）
5. **BASE_URL 支持环境变量注入**：设置 `POSTMAN_BASE_URL` 环境变量可覆盖 config.py 的默认 BASE_URL
6. 报告服务端口/主机支持环境变量：`REPORT_SERVER_PORT`（默认 5000）、`REPORT_SERVER_HOST`（默认 0.0.0.0）
7. **生产建议**：安装 `waitress`（`pip install waitress`）后，report_server.py 自动切换为 waitress WSGI 服务器；未安装则降级使用 Flask 开发服务器

### 2.2 测试执行参数约束

1. CLI 参数顺序：
   python -m postman_api_tester.postman_api_tester <postman_file> [base_url] [output_dir] [token] [results_per_page]
2. token 优先级：命令行/代码入参 token > 环境变量 `POSTMAN_TOKEN` > postman_api_tester/config.py 的 TOKEN > 自动登录获取
3. base_url 优先级：命令行/代码入参 base_url > 环境变量 `POSTMAN_BASE_URL` > postman_api_tester/config.py 的 BASE_URL > 从 Postman 集合解析
4. output_dir 优先级：命令行/代码入参 output_dir > postman_api_tester/config.py 的 REPORT_OUTPUT_DIR > 项目根目录 reports/
5. 报告分页参数 results_per_page 默认 30（报告页建议控制在 1~100）
6. **请求超时（集中配置）**：超时参数现由 `postman_api_tester/config.py` 统一提供（`REQUEST_CONNECT_TIMEOUT=10`、`REQUEST_READ_TIMEOUT=30`），可通过同名环境变量覆盖。`execute_test()` 运行时从 config 动态读取，禁止在执行层硬编码数字
7. **token 并发安全**：`_auth_token` 现为实例变量，不同并发任务互相隔离，禁止恢复类变量方案
8. **details.json 安全**：写入详情文件时，请求头中 `authorization`、`token`、`x-token`、`x-access-token`、`access-token` 字段值统一替换为 `***`，不可明文存储
9. **URL 解析兼容约束（导入执行一致性）**：执行层解析 `request.url` 时，若 `path` 缺失必须回退读取 `raw`；当 `raw` 为绝对 URL 或 `{{baseUrl}}/...` 模式时均应可执行，禁止因仅有 `raw` 结构而在执行阶段静默丢失接口

### 2.3 报告产物约束

每次执行必须生成同名前缀的三类文件：

1. postman_report_时间戳.html（主报告）
2. postman_report_时间戳_details.json（详情数据）
3. postman_report_时间戳_meta.json（历史索引/对比元数据）

### 2.4 报告中心约束（report_server.py）

1. 服务默认监听 0.0.0.0:5000
2. 报告目录读取优先级：
   POSTMAN_REPORTS_DIR 环境变量 > REPORTS_DIR 环境变量 > config.py 中 REPORT_OUTPUT_DIR > 项目根目录 reports/
3. 单报告数据页分页规则：
   默认 20 条/页；上限 100 条/页
4. 支持报告历史对比：新增接口、移除接口、状态变化、成功率变化
5. 并发写回同一报告时使用按报告名粒度的写锁，避免并发覆盖
6. **上传文件名安全**：`original_name` 在写入任务状态前已清洗（仅保留字母/数字/汉字/`-._()（）【】`），防止路径穿越与注入攻击
7. **任务内存上限（集中配置）**：`RUN_JOBS` 上限由 `config.py` 的 `RUN_JOBS_MAX`（默认 200）控制，可通过同名环境变量覆盖；不可移除清理逻辑
8. **报告列表缓存**：`list_reports()` 使用 30 秒 TTL 缓存，新报告生成时通过 `_invalidate_reports_cache()` 主动清除；不可在缓存有效期内绕过
9. **base_url SSRF 防护**：`run_postman_tests()` 和 `/api/run-postman` 接口均校验 base_url 仅允许 `http`/`https` 协议；禁止删除此校验
10. **结构化日志**：postman_api_tester.py 与 report_server.py 均使用 `logging` 模块；禁止在核心路径新增 `print()` 替代 `logger.*`
11. **健康检查端点**：`/health` 端点返回 `{"status": "ok", "timestamp": "..."}` 用于监控；不可删除
12. **生产服务器**：优先使用 waitress；禁止在生产环境将 Flask `debug=True`
13. **HTML 模板外置（固定）**：报告中心页面渲染统一使用 `templates/` 目录下的外置模板（Jinja2 `render_template`）。不再支持 `inline` 模式与内嵌模板回退分支，模板缺失应直接按异常暴露并修复源模板
14. **报告删除能力**：报告中心支持删除单份历史报告，并同时清理其主报告 HTML、`_details.json`、`_meta.json` 以及 `_page_*.html` 分页子页。删除逻辑必须限定在 `REPORTS_DIR` 目录内，禁止删除目录外文件
15. **首页轻量化约束**：报告首页仅加载列表摘要字段（报告名、集合名、来源、生成时间、汇总等），禁止再将完整 `results` / `request_info` / `response_info` 批量注入首页模板；详情数据必须继续通过详情接口或详情文件按需读取
16. **数据库迁移反馈约束**：执行层在 `FAILED`/`ERROR` 结果中应输出 `db_feedback` 结构化字段，并基于常见连接异常、认证异常、SQL兼容异常、对象缺失、字符集及类型兼容异常给出建议；禁止删除该字段或将其改为仅日志可见
17. **可选接口执行约束**：上传执行支持“全量执行 / 仅执行已选接口”两种模式。相关参数与边界值需统一由 `postman_api_tester/config.py` 管理（`ENABLE_SELECTIVE_RUN`、`COLLECTION_PREVIEW_MAX_ITEMS`、`RUN_STATUS_POLL_INTERVAL_MS`、`RUN_RESULTS_PER_PAGE_*`），禁止在页面脚本或服务端路由中继续硬编码默认值
18. **导出范围约束**：报告导出支持 `full`（全量）与 `report_only`（仅本次报告接口）两种范围，默认范围与开关由 `config.py` 统一管理（`REPORT_EXPORT_DEFAULT_SCOPE`、`REPORT_EXPORT_ALLOW_REPORT_ONLY`、`REPORT_EXPORT_INCLUDE_AUTH_DEFAULT`）。在禁止 `report_only` 时必须自动降级 `full`，禁止返回 500
19. **文档闭环约束**：每次功能收尾或发布前，`README.md`、`操作手册.md`、`快速命令参考.md`、`最终交付清单.md` 四份文档必须同步到同一版本号与发布日期，并同步新增能力说明（执行范围、导出范围、配置项）；禁止仅更新单份文档
20. **外置模板约束**：所有前端 UI 改动必须在 `templates/` 目录下的 `.html` 文件中进行。`report_server.py` 已移除内嵌模板（`INDEX_TEMPLATE`、`REPORT_VIEW_TEMPLATE`）与 `render_with_fallback()` 兜底分支；新增功能、样式调整、交互优化统一在外置模板实现
21. **人工用例能力约束**：报告页支持人工新增/编辑/删除用例，数据写入 `_meta.json` 的 `manual_cases` 字段；默认目录由 `config.py` 的 `MANUAL_CASE_FOLDER_NAME` 提供，页面允许手工覆盖填写
22. **排除显示约束**：报告结果支持对单条用例打排除标记并置灰展示；页面必须提供“显示/隐藏已排除”开关，隐藏时不展示已排除项，显示时保留置灰态
23. **统一导出约束**：导出 JSON 必须为单一文件并统一合并：原始集合（按最新请求参数回填） + `manual_cases` - `manual_exclusions`；禁止拆分为多个导出文件
24. **路由-函数签名一致性约束**：凡是路由层以关键字参数调用内部函数（如 `include_excluded=...`），必须同步校验被调用函数签名是否接收该参数；新增筛选参数时，必须同时更新函数签名、返回字段与模板消费字段，禁止只改路由不改函数
25. **人工用例可维护性约束**：`manual_cases` 条目必须具备稳定 `id`（新增时生成，更新/删除时按 `id` 定位），禁止依赖 `name/url` 等可变字段做唯一定位
26. **运行时冒烟回归约束（必做）**：涉及 `report_server.py` 路由、筛选、导出、人工用例改动后，提交前至少验证：`/health`、`/`、`/report-view`、`/api/report-results?include_excluded=true/false`、`/api/manual-cases` 的 add/update/delete、`/api/export-collection`；禁止仅以静态检查代替运行验证
27. **最小闭环排查法约束（复用）**：发现问题时按“日志定位 -> 最小复现 -> 双态参数验证（true/false）-> CRUD 闭环验证 -> 数据回滚清理”执行，并在约束更新记录中沉淀根因与验证命令，便于后续复用
28. **导出参数完整性约束（新增）**：导出集合（尤其人工用例）必须保留完整请求参数，至少包含 `method`、`url`、`headers`、`params`、`body`；对多 Body 模式还必须保留 `body_mode/body_data` 语义，禁止导出仅保留 method/url 导致参数丢失
29. **导入可解析性约束（新增）**：导入链路必须可解析导出文件中的全部请求数据结构（含 `request.url.raw/path/query`、headers、params、body 各模式），并保证“可预览的数据应可执行”，禁止出现导入后请求体为空或执行行为退化
