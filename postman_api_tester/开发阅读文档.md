## 2. 开发流程约束

方案讨论与实施分离：

1. 新功能开发分为”方案讨论”和”实施”两个阶段。
2. 方案讨论阶段禁止修改代码文件（包括 `postman_api_tester/report_server.py`、`templates/*.html`、`postman_api_tester/*.py`）。
3. 只有收到用户明确口令”开始实施”后，才允许进入代码修改、测试与提交阶段。
4. 方案讨论期间仅允许：读取文件、分析代码结构、输出方案文档、更新约束文档。
5. 新增方案文档命名格式：`YYYY-MM-DD-方案名称.md`，放在 `postman_api_tester/方案讨论`。
6. 实施前必须确认：业务范围、涉及页面、对原功能的影响。
7. 实施前需对本次讨论方案进行合并梳理，避免遗漏。
8. 实施后可复用的验证文件保留到 `test_data`，并在操作手册记录复用方式。

### 2.1 AI 定时自动升级豁免条件

由 `.claude/scheduled_tasks.json` 中的 cron 任务触发的自动研发升级（每天 20:30），在满足以下全部前提时可跳过约束 2.3（”开始实施”人工确认）：

1. **分支隔离**：必须在 `feature/auto-upgrade-YYYY-MM-DD` 分支上操作，禁止直接在 master 上提交。
2. **只改进不破坏**：改进内容为新增测试、代码重构、bug 修复、性能优化或新功能；禁止删除既有功能或改变对外接口语义。
3. **可回退**：合并前必须保留完整 commit 历史，合并使用 `--no-ff`，确保可 `git revert` 单 commit 回退。
4. **早间汇报**：每次自动升级完成后，次日 08:30 必须向用户汇报改进内容、风险点与回退方法；未经用户确认不得推送 master。
5. **文档同步**：合并 master 前必须完成 6 份文档版本号统一（约束 3.17）。
6. **测试门禁**：合并前必须 pytest 全通过且 mypy 零错误。

违反上述任一前提时，仍需等待用户明确确认后才能继续。

分层规则：

1. **core/**：执行链路核心逻辑（HTML 报告生成、断点恢复、执行管道、类型定义）。禁止引用 Flask 或 handlers/services。
2. **handlers/**：仅允许接收 HTTP 请求、参数校验、调用 services、返回 Response。禁止直接操作文件系统或发 HTTP 请求。
3. **services/**：承担业务逻辑（导出、重试、列表、判定、锁、提交等）。允许调用 utils 和 repositories，禁止直接返回 Flask Response。
4. **utils/**：纯函数工具（URL 合并、请求构建、响应解析、脱敏、缓存）。禁止持有业务状态，禁止引用 handlers/services。
5. **tests/**：按 `test_<被测模块>.py` 命名。handler 测试必须使用 `test_request_context()`，services 测试允许 mock 依赖。
6. **新增路由**：必须先创建 `handlers/<name>_routes.py`，再在 `report_server.py` 中以 `return _handler_name(args)` 形式注册。
7. **新增服务**：必须创建 `services/<name>_service.py`，如有共享逻辑可抽象为 `base_service.py` 子类。

## 3. 当前有效约束清单（去重归并）

### 3.1 运行与环境约束

1. Python 版本要求：3.10+。
2. 依赖安装入口：`requirements.txt`。
3. 建议在项目根目录执行命令：`d:/tangzk/py/seldom-api-testing`。
4. `POSTMAN_TOKEN`、`POSTMAN_BASE_URL` 支持环境变量注入，优先于 `config.py` 默认值。
5. 报告服务端口/主机支持环境变量：`REPORT_SERVER_PORT`（默认 5000）、`REPORT_SERVER_HOST`（默认 0.0.0.0）。
6. 生产环境优先 `waitress`，未安装时降级 Flask 开发服务器；禁止生产 `debug=True`。

### 3.2 执行参数与安全约束

1. CLI 参数顺序：
   `python -m postman_api_tester.postman_api_tester <postman_file> [base_url] [output_dir] [token] [results_per_page]`
2. token 优先级：命令行/代码入参 > `POSTMAN_TOKEN` > `config.py::TOKEN` > 自动登录。
3. base_url 优先级：命令行/代码入参 > `POSTMAN_BASE_URL` > `config.py::BASE_URL` > 从 Collection 解析。
4. output_dir 优先级：命令行/代码入参 > `config.py::REPORT_OUTPUT_DIR` > 项目根目录 `reports/`。
5. `results_per_page` 默认 30，建议范围 1~100。
6. 请求超时由 `config.py` 集中管理（`REQUEST_CONNECT_TIMEOUT`、`REQUEST_READ_TIMEOUT`），执行层禁止硬编码。
7. token 并发隔离必须保持实例级（`_auth_token`），禁止回退为类变量。
8. URL 解析必须兼容 `request.url.path` 缺失时回退 `request.url.raw`，并支持绝对 URL 与 `{{baseUrl}}/...`。
9. SSRF 防护必须保留：`run_postman_tests`、`/api/run-postman`、`/re-request-api`、`/api/proxy-request`、`/api/run-ad-hoc-tests` 的目标 URL/base_url 仅允许 `http/https`。proxy 端点另须校验 `PROXY_ALLOWED_HOSTS` 域名白名单（非空时）。

### 3.3 报告产物约束

每次执行必须生成同名前缀三件套：

1. `postman_report_时间戳.html`
2. `postman_report_时间戳_details.json`
3. `postman_report_时间戳_meta.json`

执行中断（部分成功）时也必须生成三件套，且使用新时间戳，禁止覆盖历史报告。

### 3.4 报告中心核心约束（report_server.py）

1. 默认监听 `0.0.0.0:5000`。
2. 报告目录优先级：`POSTMAN_REPORTS_DIR` > `REPORTS_DIR` > `config.py::REPORT_OUTPUT_DIR` > 项目根目录 `reports/`。
3. 单报告页分页默认 20、上限 100。
4. 支持历史对比（新增/移除/状态变化/成功率变化）。
5. 同报告并发写回必须使用“按报告名粒度”的写锁。
6. 上传文件名入库前必须清洗（仅保留字母/数字/汉字/`-._()（）【】`）。
7. `RUN_JOBS` 上限由 `config.py::RUN_JOBS_MAX` 管理（默认 200），不可移除清理逻辑。
8. `list_reports()` 必须维持 30 秒 TTL 缓存，并在新报告生成时执行 `_invalidate_reports_cache()`。
9. 首页仅加载摘要字段，禁止注入完整 `results/request_info/response_info`。
10. `/health` 必须保留 `status`、`timestamp`，允许扩展 `log_alert` 等运行态字段。
11. 结构化日志必须使用 `logging`，核心路径禁止新增 `print()` 代替 `logger.*`。
12. 页面渲染必须使用 `templates/` 外置模板；不支持内嵌模板回退分支。
13. 报告删除必须限定在 `REPORTS_DIR` 内，并同步清理 `.html`、`_details.json`、`_meta.json`、`_page_*.html`。
14. 报告结果详情允许人工判定（成功/失败/恢复自动），需保留 `manual_judgement/judgement_history` 并同步重算 summary；重试真实回写后取消人工判定激活状态。
15. 日志告警阈值由 `LOG_ALERT_ERROR_WINDOW_SECONDS`、`LOG_ALERT_ERROR_RATE_THRESHOLD_PER_MIN` 集中管理，并通过 `/health.log_alert` 对外可观测。
16. analytics 能力受 `ENABLE_REPORT_ANALYTICS` 集中控制；关闭时页面不注入 analytics 主容器，`/api/report-analytics/<report_name>` 与 `/api/report-analytics-compare` 必须返回 403。
17. analytics 查询参数 `top_n`、`trend_limit` 必须按配置默认值与上限自动钳制；`include_samples` 必须保留布尔兼容解析。
18. analytics compare 必须保持新增接口形态，不得破坏既有 `/api/compare` 语义。
19. analytics 图表可视化受 `ENABLE_REPORT_ANALYTICS_CHARTS` 集中控制，且必须受父开关 `ENABLE_REPORT_ANALYTICS` 联动约束；父开关关闭时，无论子开关为何值，页面均不加载 Chart.js、不渲染图表 canvas。
20. 图表 CDN 加载失败时必须自动降级为纯表格，不得阻断页面其他功能。
21. **路由层提取约束（v1.3.0）**：同约束 2.2 第 6 条。`report_server.py` 仅保留装饰器与单行委托，业务逻辑下沉至 `handlers/*`。
22. **类型检查约束（v1.3.0）**：handlers/ 与 services/ 新增代码必须通过 mypy 检查（实际命令见约束 3.9.4，`pyproject.toml` 中配置了部分 strict 标志但未启用完整 `--strict`）；禁止新增 `Any` 类型扩散，禁止新增未标注返回类型的公有函数。
23. **测试覆盖约束（v1.3.0）**：handlers/ 下新增路由必须配套单元测试，异常路径（400/403/404）与成功路径均需覆盖（测试命名规范见约束 2.2 第 5 条）；禁止在 handler 模块中调用 `configure_run_jobs()` 等全局状态初始化函数。
24. **启动入口约束（v1.4.1）**：报告服务唯一推荐的启动方式为 `python -m postman_api_tester`（通过包级 `__main__.py` 入口调用 `ReportServerApp.run_app(app)`）。禁止直接以 `python postman_api_tester/report_server.py` 或 `python -m postman_api_tester.report_server` 方式启动（`report_server.py` 仅承载路由装配，不提供命令行入口）。根目录兼容启动器 `report_server.py` 已移除。
25. **子目录报告支持约束（v1.3.1）**：报告可生成到 `reports/` 下任意子目录中；`list_reports()` 必须使用 `rglob` 递归扫描，`serve_report()` 必须通过 `meta_file` 推导实际路径并直接返回文件内容。禁止在子目录报告场景下依赖 `send_from_directory`（其内部 `secure_filename` 会将路径分隔符替换为下划线，导致 404）。
26. **报告扫描排除约束（v1.3.1）**：`report_meta_files()` 与 `legacy_postman_html_files()` 扫描时必须按 `REPORT_SCAN_EXCLUDE_DIRS` 过滤排除目录（默认 `"old"`）；排除判断仅检查目录层级（`Path.parts[:-1]`），不得误伤同名文件。
27. **favicon 路由（v1.3.3）**：`/favicon.ico` 必须返回 204 空响应，消除浏览器默认请求导致的 404 日志噪音。

### 3.5 导入、导出与脱敏约束

1. 导出 JSON 必须为单一文件，并按规则合成：原始集合（请求参数回填） + `manual_cases` - `manual_exclusions`。
2. 导出范围支持 `full` 与 `report_only`，默认与开关由 `config.py` 统一管理；禁用 `report_only` 时自动降级 `full`，禁止 500。
3. 导出必须保留完整请求语义：`method/url/headers/params/body`，以及 `body_mode/body_data`。
4. 导入链路必须可解析导出结构（含 `request.url.raw/path/query` 与 body 各模式），保证“可预览可执行”。
5. 敏感头规则由 `config.py::SENSITIVE_HEADERS` 与内置集合合并统一生效：
   - 详情写入时掩码 `***`
   - 导出 `include_auth=false` 时按同一规则剔除
   - 禁止不同链路使用不同敏感头集合
6. 流式导出必须通过新增接口 `/api/export-collection-stream` 提供，不得替换 `/api/export-collection`。
7. 导出通道仅允许 `REPORT_EXPORT_CHANNEL_MODE=auto|legacy|stream`；非法值回退 `auto`。
8. `auto` 模式按 `REPORT_EXPORT_STREAM_THRESHOLD` 分流，流式失败自动回退旧接口；`legacy/stream` 为强制通道。

### 3.6 人工用例、筛选与路由一致性约束

1. 人工用例写入 `_meta.json::manual_cases`，默认目录由 `MANUAL_CASE_FOLDER_NAME` 提供且页面可覆盖填写。
2. `manual_cases` 必须有稳定 `id`，更新/删除按 `id` 定位，禁止用 `name/url` 作为唯一键。
3. 排除能力必须支持“置灰 + 显示/隐藏已排除”开关。
4. 路由层新增关键字参数时，必须同步更新被调函数签名、返回字段、模板消费字段。
5. 执行层 `FAILED/ERROR` 结果必须输出 `db_feedback` 结构化字段，不得降级为仅日志可见。

### 3.7 ad-hoc 执行约束

1. 首页通过“新增接口测试”进入独立页执行，能力受 `ENABLE_ADHOC_RUN`、`ADHOC_MAX_ITEMS`、`ADHOC_DEFAULT_COLLECTION_NAME` 控制。
2. `/api/run-ad-hoc-tests` 仅接收 JSON；`cases` 必须为非空数组，数量不超过 `ADHOC_MAX_ITEMS`。
3. 入队前必须规范化为 Postman Collection v2.1 JSON，写入 `uploaded_collections/<job_id>.json`，并复用现有任务状态与报告链路。
4. ad-hoc URL 仅允许三类：绝对 `http/https`、`{{baseUrl}}/...` 或 `{{base_url}}/...`、配合 `base_url` 的相对路径。
5. body 模式统一支持 `none/raw/urlencoded/formdata/graphql/binary`，不得与导入/重试链路语义漂移。

### 3.8 前端一致性与调试回归约束

1. 所有 UI 改动必须在 `templates/*.html` 实现。
2. 同类页面优先复用既有设计语言；如引入新风格，必须同步评估并更新同类旧页面。

7. **参数传递链路**：`handlers/job_routes.py` → `services/report_job_execution_service.py` → `postman_api_tester.py::run_postman_tests()` → `core/execution_helpers.py` → `executor.py`；修改 `judgment_config` 参数时必须同步所有转发层签名，避免运行时 `TypeError`。

### 3.13 数据驱动测试约束（v1.4.0）

1. **功能开关**：`ENABLE_DATA_DRIVEN`（默认 `False`），支持环境变量覆盖。开关关闭时不加载数据文件、不展开接口、页面不注入上传入口。
2. **数据文件格式**：仅支持 CSV（UTF-8/UTF-8-sig/GBK）和 JSON（数组对象）。不支持 Excel。
3. **数据文件路径安全**：Web 上传的数据文件存入 `uploaded_collections/`；CLI 模式允许任意路径但必须文件存在。禁止路径穿越。
4. **行数限制**：`DATA_FILE_MAX_ROWS`（默认 10000），超限拒绝执行。`DATA_FILE_MAX_SIZE`（默认 10MB）。
5. **按需展开**：仅 URL/headers/body/params 中引用了数据文件变量名的接口才按数据行数展开；无变量引用的接口保持单次执行。`{{baseUrl}}`/`{{base_url}}` 不算数据驱动变量。
6. **变量替换范围**：`utils/variable_substitution.py` 中的 `substitute_variables()` 排除 `baseUrl` 和 `base_url`；替换范围覆盖 URL/headers/body/params 中的 `{{变量名}}`。
7. **checkpoint 兼容**：展开接口（data_index > 0）的 checkpoint 键为 `item_path#data_index` 复合键（如 `0.2.1#3`）；未展开接口保持原键格式（如 `0.2.1`），向后兼容旧 checkpoint。
8. **报告展示**：结果记录含 `data_index` 字段（未展开接口 data_index=0）；报告列表在接口名称后显示数据组编号徽标（`#N`）。
9. **参数传递链路**：`handlers/job_routes.py` → `services/report_job_submission_service.py` → `services/report_job_execution_service.py` → `postman_api_tester.py::run_postman_tests()` → `core/execution_helpers.py::_prepare_execution_apis()`；`data_file` 参数必须同步所有转发层签名。

### 3.14 请求串联与变量提取约束（v1.4.0）

1. **功能开关**：`ENABLE_VARIABLE_EXTRACTION`（默认 `False`），支持环境变量覆盖。开关关闭时不创建 VariableContext、不解析 `x_extract`、页面不注入编辑区。
2. **变量上下文生命周期**：`core/variable_context.py::VariableContext` 仅在 `_execute_api_suite()` 调用期间有效，不持久化到 checkpoint。
3. **x_extract 配置**：放置在 Postman Collection item 的 `request` 对象中，键为变量名，值为 JSONPath 表达式（`$.field.subfield`、`$.array[0].field`）或特殊前缀 `$header.Name`（提取响应头）。
4. **JSONPath 安全**：`utils/extract_utils.py` 仅支持简化路径（字段访问 + 数组索引），禁止 eval/exec/过滤器/通配符/递归下降。使用自研轻量解析器。
5. **变量优先级**：数据行变量（已烘焙到 ApiConfig）> 运行时提取变量（VariableContext）> Collection 预置变量 > initial_variables。
6. **提取失败行为**：提取失败时变量不写入上下文（不覆盖已有值），日志记录警告，报告中 `extracted_variables` 仅记录成功提取的键值对。
7. **选择性执行警告**：选择性执行可能跳过"变量生产"接口，导致后续接口变量缺失。提供 `initial_variables` 参数让用户预填关键变量。
8. **executor 集成**：`executor.py::execute_test()` 在请求前调用 `substitute_in_api_config()` 替换变量，请求后调用 `variable_context.update_from_extract()` 提取变量。
9. **ad-hoc 支持**：`utils/collection_utils.py::normalize_adhoc_case()` 支持 `x_extract` 字段，`build_adhoc_collection()` 将其写入 request 节点。

### 3.21 Pre-request 脚本约束（v1.18.0）

   - **Step B（x_pre_request 表达式）**：需启用 `ENABLE_PRE_REQUEST_SCRIPT`，在请求发出前执行 Python 表达式，结果作为局部变量参与本次请求的变量替换。
3. **沙箱安全策略**（`utils/pre_request_executor.py`）：
   - **限制内置函数**：`__builtins__` 设为安全子集（`int`/`str`/`float`/`len`/`range`/`list`/`dict` 等），禁用 `open`/`exec`/`eval`/`import`。
   - **白名单模块**：仅允许 `hashlib`/`hmac`/`base64`/`time`/`json`/`re`，禁止网络/文件/系统调用。
   - **危险关键字拦截**：正则检查 `import`/`__import__`/`exec`/`eval`/`compile`/`open`/`os`/`sys`/`subprocess`/`__builtins__`/`globals`/`locals` 等关键字。
   - **超时保护**：表达式执行超时 1 秒。Unix 使用 `signal.SIGALRM`，Windows 使用 `ThreadPoolExecutor`。
   - **错误处理**：表达式执行失败返回空字符串，不中断请求流程，日志记录具体错误信息。
4. **变量隔离**：pre-request 设置的变量仅在当前请求的变量替换中使用（局部变量），**不写入全局 `VariableContext`**。避免并发执行时变量互相覆盖，不影响批次调度器的依赖计算。
5. **执行流程**：`executor.py::execute_test()` 中，先执行 `x_pre_request` 表达式得到 `local_vars`，再合并到 `merged_vars = {**variable_context.variables, **local_vars}`，最后调用 `substitute_in_api_config()`。
6. **解析器扩展**：`parser.py` 中 `ApiConfig` TypedDict 新增 `x_pre_request: Optional[Dict[str, str]]` 字段，`_parse_request()` 解析该字段（遵循 `x_*` 扩展模式）。
7. **前端 UI**：`templates/collection_editor.html` 新增 `Pre-request` 标签页，键值对编辑器支持添加/编辑/删除表达式，展示可用模块与变量引用说明。
8. **表达式引用**：可通过 `variables.get('name', 'default')` 引用已有变量，通过 `result.get('var', '')` 引用同批次前置表达式结果（按定义顺序）。
9. **参数传递链路**：`config.py::ENABLE_PRE_REQUEST_SCRIPT` → `executor.py::execute_test()` 条件检查 → `utils/pre_request_executor.py::execute_pre_request()` → 局部变量合并 → `utils/variable_substitution.py::substitute_in_api_config()`。
