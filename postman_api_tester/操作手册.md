## 1. 工具目标

用于读取 Postman/APIFox 导出的 Collection JSON，批量执行接口请求，输出控制台摘要与 HTML 报告。

## 2. 程序目录结构

建议新人先对下面这些目录和文件建立最基本认识：

```text

│  ├─ postman_api_tester.py          # 主执行入口，负责解析 Postman、发请求、生成报告
│  ├─ config.py                      # 地址、Token、超时、报告目录等集中配置
```

理解方式建议如下：

1. `postman_api_tester/` 负责“执行测试”。

4. `reports/` 负责“存放测试产物”。

## 3. 拉取代码后的准备工作

1. Python 3.10+
2. 已安装 requirements.txt 依赖
3. 具备可访问接口环境（内网/测试环境）

### 3.1 拉取或解压代码

如果通过 Git 拉取，建议放到固定目录，例如：

```powershell
git clone <你的仓库地址> d:\tangzk\py\seldom-api-testing
cd d:\tangzk\py\seldom-api-testing
```

如果是压缩包交付，也建议解压到类似目录，再执行后续步骤。

### 3.2 安装依赖

```powershell
cd d:\tangzk\py\seldom-api-testing
pip install -r requirements.txt
```

### 3.3 修改配置文件

编辑文件：`postman_api_tester/config.py`

第一次使用至少关注以下配置：

1. `BASE_URL`：目标环境地址
2. `TOKEN`：认证 token，可留空
3. `REPORT_OUTPUT_DIR`：报告输出目录，可留空
4. `REQUEST_CONNECT_TIMEOUT` / `REQUEST_READ_TIMEOUT`：请求超时
5. `ENABLE_SELECTIVE_RUN`：是否启用“仅执行已选接口”能力
6. `REPORT_EXPORT_DEFAULT_SCOPE`：报告导出默认范围（`full` / `report_only`）

示例：

```python
BASE_URL = "http://10.50.11.130:11000"
TOKEN = ""
REPORT_OUTPUT_DIR = r"D:\api-test-reports"
```

配置建议顺序：

1. 先改 `BASE_URL`
2. 再决定 `TOKEN` 是直接填写还是留空自动登录
3. 最后确认 `REPORT_OUTPUT_DIR` 是否要统一输出到固定目录

如需临时覆盖，不必改代码文件，可直接用环境变量：

```powershell
$env:POSTMAN_BASE_URL = "http://10.50.11.130:11000"
$env:POSTMAN_TOKEN = "your_token"
python -m postman_api_tester.postman_api_tester "sample_api_collection.json"
```

## 4. 标准执行流程

1. 准备 Postman JSON（建议 Collection v2.1）。
2. 检查并按需设置 config.py：BASE_URL、TOKEN。
3. 执行测试命令。
4. 查看控制台汇总。
5. 打开 HTML 报告检查失败详情。
6. 必要时基于新 token 重跑。

推荐新人的最短路径：

1. 先准备一份最小可执行的 `*.postman.json`
2. 先跑命令行测试，确认能出报告
3. 再启动报告中心，看页面和失败详情
4. 最后再切换到大批量 Collection

## 5. 执行方式

### 5.1 命令行执行（推荐）

```bash
python -m postman_api_tester.postman_api_tester <postman_file> [base_url] [output_dir] [token] [results_per_page]
```

- postman_file：必填，Postman JSON 文件路径
- base_url：可选，覆盖文件内 baseUrl
- output_dir：可选，默认项目根目录下 reports/
- token：可选，传入后跳过自动登录
- results_per_page：可选，报告每页条数，默认 30

配置优先级补充：

1. 命令行 `output_dir`
2. `postman_api_tester/config.py` 中 `REPORT_OUTPUT_DIR`
3. 默认 `项目根目录/reports/`

最常用示例：

```powershell
python -m postman_api_tester.postman_api_tester "支付（UPS）.postman.json"
```

如果只是第一次验证环境是否打通，建议先用 `sample_api_collection.json` 或最小 Collection，不要一开始就跑几百个接口。

### 5.2 交互式快速执行

```powershell
python postman_api_tester/run_test_and_open.py
```

适合不熟悉命令行参数的同学，脚本会依次引导输入：

1. Postman 文件
2. base_url
3. 报告输出目录
4. 每页条数

说明：

1. 报告中心首页上传区支持先“解析接口”，再选择“全量执行”或“仅执行已选接口”。
2. 仅执行已选接口模式下，未选择接口会被后端拦截并提示，不会误执行全量。

## 6. 报告说明

每次执行会生成：

- postman_report_时间戳.html：主报告
- postman_report_时间戳_details.json：详情数据
- postman_report_时间戳_meta.json：历史索引与报告比对元数据

报告中可查看：

- 请求方法、URL、状态码
- 请求头/参数/请求体
- 响应头/响应体
- 汇总统计（通过/失败/错误）

特别说明：

1. `_details.json` 用于详情页按需读取，不建议手工修改。
2. `_meta.json` 用于报告中心首页、差异对比、元数据接口读取。
3. 报告中心默认不展示 `_page_*.html` 分页子页。

## 7. 报告中心与历史对比

### 7.1 启动方式

```bash
cd d:\tangzk\py\seldom-api-testing
python report_server.py
```

启动后可使用：

1. 本机访问：`http://127.0.0.1:5000`
2. 局域网访问：`http://你的局域网IP:5000`

### 7.2 页面说明

1. 打开首页后可查看历史总报告列表，分页子报告页不会在首页展示。
2. 点击报告名称或“数据视图”可直接进入单份报告的数据浏览页。
3. 在单报告页可按接口名称或路径搜索，并支持结果状态筛选。
4. 单报告页支持分页显示，默认 20 条，最高 100 条。
5. 点击“详情”可查看请求头、查询参数、请求体、响应头、响应体。
6. 失败或异常结果中，可查看数据库迁移反馈信息，辅助判断是否为数据库切换问题。
7. 点击“原始HTML”可打开生成时的原始静态报告页面。
8. 在首页选择两份历史报告后，可执行结构化差异对比。
9. 若要给其他同事访问，只需保证双方在同一局域网，且本机防火墙已放通 5000 端口。
10. 单报告页“导出最新 JSON”支持两种范围：`full`（全量）和 `report_only`（仅本次报告接口）。
11. 单报告页支持人工用例三段式编辑流程：发送请求 -> 自动回填响应 -> 选择 PASSED/FAILED 后保存。
12. 结果列表操作列支持行内“排除/取消排除”，可直接切换单条结果的排除状态。
13. 人工用例“发送请求”使用后端代理接口，仅允许 `http/https`，请求超时遵循 `config.py` 统一配置。
14. 编辑重试接口 `/re-request-api` 已同步启用 `http/https` 校验，与代理接口统一 SSRF 防护口径。
15. 报告页修复双编辑器 Body 模式隔离（mc 与 er 互不影响），避免切换时互相污染。
16. 报告页 Body 类型选择体验优化：支持 none/raw/urlencoded/formdata/graphql/binary，切换后与重试区编辑器互不串扰。
17. 报告页顶部新增“页面模板更新时间”小标记，用于确认当前页面是否为最新模板版本。
18. 人工用例详情中的响应体展示与报告详情统一为同款展示逻辑（支持预览与展开）。
19. 编辑人工用例时回显历史请求信息（Headers/Params/Body），便于改参后重新发送。
20. 导出结果一致性修复：导出弹窗中的人工用例数量与实际导出内容一致，且已排除接口不会被导出。

### 7.3 报告目录一致性

报告中心读取目录优先级：

1. `POSTMAN_REPORTS_DIR`
2. `REPORTS_DIR`
3. `postman_api_tester/config.py` 中 `REPORT_OUTPUT_DIR`
4. 默认 `项目根目录/reports/`

按此规则可确保“测试执行输出目录”和“报告中心读取目录”一致。

### 7.4 防火墙说明

如果其他机器无法访问，但本机可以访问，请检查 Windows 防火墙是否允许 Python 或 5000 端口的入站访问。

推荐直接使用项目根目录脚本一键放通：

```powershell
cd d:\tangzk\py\seldom-api-testing
powershell -ExecutionPolicy Bypass -File .\allow_report_server_firewall.ps1
```

说明：

1. 脚本会自动申请管理员权限。
2. 默认放通 TCP 5000 端口，规则名为 `Postman Report Server 5000`。
3. 执行完成后，同局域网可通过 `http://你的局域网IP:5000` 访问报告中心。

如需手工执行，也可以使用下面命令：

```powershell
New-NetFirewallRule -DisplayName "Postman Report Server 5000" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 5000
```

## 8. 迁移验证建议（MySQL -> 国产数据库）

该工具适合做业务回归与接口可用性验证：

1. 验证接口状态码与主流程可用性
2. 验证关键响应字段是否完整
3. 对迁移前后同入参做结果对比
4. 优先关注失败详情中的数据库反馈信息，快速判断是连接问题、认证问题，还是 SQL 兼容问题

仅靠接口通过率不能覆盖全部迁移风险，建议补充：

1. 数据一致性抽样核对
2. SQL 兼容性专项验证
3. 并发与性能压测

## 9. 故障排查

- 未登录/401：优先确认 token 来源和优先级
- URL错误：检查 base_url 覆盖逻辑
- 接口超时：确认网络、防火墙、目标服务健康状态
- message 中出现“数据库反馈”：优先按提示检查连接、权限、SQL 方言、schema 或字符集问题
- 报告为空：确认 JSON 是否包含 item/request 结构
- 局域网不可访问：检查 `python report_server.py` 是否已启动，以及防火墙是否放通 5000 端口
- 无法对比历史报告：确认 `reports/` 下同时存在 `postman_report_*.html` 与对应的 `postman_report_*_meta.json`

## 10. 新人第一次使用建议顺序

1. 先看本手册第 2 节目录结构，知道测试执行和报告查看分别在哪。
2. 再改 `postman_api_tester/config.py` 的 `BASE_URL` / `TOKEN` / `REPORT_OUTPUT_DIR`。
3. 执行 `pip install -r requirements.txt` 安装依赖。
4. 先用最小 `*.postman.json` 跑一次命令行测试。
5. 确认目录中生成 `.html`、`_details.json`、`_meta.json` 三件套。
6. 启动 `python report_server.py`，从浏览器进入报告中心查看结果。
7. 若要给其他机器访问，再执行 `allow_report_server_firewall.ps1`。
