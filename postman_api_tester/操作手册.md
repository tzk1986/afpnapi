## 1. 工具目标

用于读取 Postman/APIFox 导出的 Collection JSON，批量执行接口请求，输出控制台摘要与 HTML 报告。

## 2. 程序目录结构

建议新人先对下面这些目录和文件建立最基本认识：

```text

├─ postman_api_tester/               # 核心测试模块与文档目录（包内主实现）

理解方式建议如下：

1. `postman_api_tester/` 负责执行核心与报告服务主实现。
2. 根目录 `report_server.py` 负责兼容启动，不承载业务逻辑。
3. `templates/` 负责页面显示。
4. `reports/` 负责存放测试产物。

### 2.1 调用路径总览（目录迁移后）

1. 服务启动路径：`python report_server.py` -> 根目录兼容启动器 -> `postman_api_tester/report_server.py`。
2. 路由调用路径：`postman_api_tester/report_server.py` 路由 -> `handlers/*`（入参校验/编排）-> `services/*`（业务能力）-> `report_repository.py` / `report_meta_repository.py` / `report_job_store.py`。
3. 通用支撑路径：`handlers/services` 统一调用 `utils/*`，覆盖 URL 处理、请求构建、响应解析、脱敏与缓存。
4. 执行生成路径：任务入队/重试由 `handlers/job_routes.py` 编排，最终调用 `postman_api_tester.py` 的 `run_postman_tests()` 输出报告三件套。

```

### 3.3 修改配置文件

编辑文件：`postman_api_tester/config.py`

第一次使用至少关注以下配置：

1. `BASE_URL`：目标环境地址
2. `TOKEN`：认证 token，可留空
3. `REPORT_OUTPUT_DIR`：报告输出目录，可留空
4. `REQUEST_CONNECT_TIMEOUT` / `REQUEST_READ_TIMEOUT`：请求超时
5. `ENABLE_SELECTIVE_RUN`：是否启用“仅执行已选接口”能力
6. `ENABLE_ADHOC_RUN`：是否启用首页“新增接口测试”独立页面入口
7. `ADHOC_MAX_ITEMS` / `ADHOC_DEFAULT_COLLECTION_NAME`：ad-hoc 单次接口上限与默认集合名
8. `REPORT_EXPORT_DEFAULT_SCOPE`：报告导出默认范围（`full` / `report_only`）
9. `REPORT_EXPORT_CHANNEL_MODE`：导出通道模式（`auto` / `legacy` / `stream`）
10. `REPORT_EXPORT_STREAM_THRESHOLD`：自动分流阈值（总接口数 >= 阈值时优先走流式）
11. `ENABLE_RESPONSE_TIME`：是否记录并展示每接口响应时间（默认 `True`）
12. `ENABLE_RETRY_FAILURES`：是否在报告页启用"一键重试失败用例"按钮（默认 `True`）
13. `ENABLE_JUNIT_EXPORT`：是否启用 JUnit XML 导出链接（默认 `True`）
14. `ENABLE_REPORT_LIST_FILTER`：是否启用首页报告列表筛选面板（默认 `True`）
16. `ENABLE_REPORT_ANALYTICS`：是否启用“分析洞察”页签及 analytics 接口（默认 `True`）
17. `REPORT_ANALYTICS_TOP_N_DEFAULT` / `REPORT_ANALYTICS_TOP_N_MAX`：分析 TopN 默认值与上限
18. `REPORT_ANALYTICS_TREND_LIMIT_DEFAULT` / `REPORT_ANALYTICS_TREND_LIMIT_MAX`：趋势序列默认值与上限
19. `REPORT_ANALYTICS_ENABLE_SAMPLES`：是否默认返回错误样本（默认 `False`）
20. `REPORT_ANALYTICS_HISTOGRAM_BUCKETS`：响应时间直方图桶边界（默认 `0,50,100,200,500,1000,3000,5000`）
21. `QUALITY_SCORE_FAILED_PENALTY` / `QUALITY_SCORE_ERROR_PENALTY` / `QUALITY_SCORE_SLOW_PENALTY` / `QUALITY_SCORE_ASSERTION_MISSING_PENALTY`：质量评分扣分阈值
22. `ENABLE_ASSERTIONS`：是否启用 JSONPath 断言校验（默认 `False`，需显式开启后断言功能才生效）
23. `ASSERTIONS_ENGINE`：断言引擎类型（默认 `jsonpath`）
24. `ENVIRONMENTS`：多环境配置字典（键为环境名，值为 `{base_url, token}`），通过环境变量 `ENVIRONMENTS_JSON` 注入 JSON 字符串
25. `DEFAULT_ENV_NAME`：默认激活的环境名称
26. `ENABLE_CHECKPOINT_RECOVERY`：是否启用断点恢复（默认 `False`）
27. `CHECKPOINT_FLUSH_EVERY_N`：每执行 N 条写入一次 checkpoint（默认 `1`）
28. `CHECKPOINT_DIR`：checkpoint 目录（空则默认 `reports/checkpoints`）
29. `ENABLE_ASSERTION_STRICT_MODE`：断言引擎异常是否严格判定为 FAILED（默认 `False`）
30. `LOG_LEVEL` / `LOG_FORMAT` / `LOG_SAMPLE_RATE`：日志级别、格式与采样率。
31. `LOG_ALERT_ERROR_WINDOW_SECONDS` / `LOG_ALERT_ERROR_RATE_THRESHOLD_PER_MIN`：ERROR 速率告警窗口与阈值。
32. `ENABLE_ERR_CODE_JUDGMENT`：是否启用 errCode 参与结果判定（默认 `False`，保持旧行为；开启后 errCode 必须匹配 `SUCCESS_ERR_CODES` 中任一值才判定为成功）。
33. `SUCCESS_ERR_CODES`：视为成功的 errCode 值列表，逗号分隔（默认 `"0"`）。
34. `ENABLE_MESSAGE_JUDGMENT`：是否启用 message 参与结果判定（默认 `True`；关闭后不检查 message）。
35. `SUCCESS_MESSAGES`：视为成功的 message 值列表，逗号分隔（默认 `"success"`，空字符串始终视为成功）。
示例：

```python
BASE_URL = "http://10.50.11.130:11000"
TOKEN = ""
REPORT_OUTPUT_DIR = r"D:\api-test-reports"
```

配置建议顺序：

1. 先改 `BASE_URL`
2. 再决定 `TOKEN` 是直接填写还是留空自动登录
3. 最后确认 `REPORT_OUTPUT_DIR` 是否要统一输出到固定目录

如需临时覆盖，不必改代码文件，可直接用环境变量：

```powershell
$env:POSTMAN_BASE_URL = "http://10.50.11.130:11000"
$env:POSTMAN_TOKEN = "your_token"
python -m postman_api_tester.postman_api_tester "sample_api_collection.json"
```

## 4. 标准执行流程

1. 准备 Postman JSON（建议 Collection v2.1）。
2. 检查并按需设置 config.py：BASE_URL、TOKEN。
3. 执行测试命令。
4. 查看控制台汇总。
5. 打开 HTML 报告检查失败详情。
6. 必要时基于新 token 重跑。

推荐新人的最短路径：

1. 先准备一份最小可执行的 `*.postman.json`
2. 先跑命令行测试，确认能出报告
3. 再启动报告中心，看页面和失败详情
4. 最后再切换到大批量 Collection

## 5. 执行方式

### 5.1 命令行执行（推荐）

```bash
python -m postman_api_tester.postman_api_tester <postman_file> [base_url] [output_dir] [token] [results_per_page]
```

- postman_file：必填，Postman JSON 文件路径
- base_url：可选，覆盖文件内 baseUrl
- output_dir：可选，默认项目根目录下 reports/
- token：可选，传入后跳过自动登录
- results_per_page：可选，报告每页条数，默认 30

配置优先级补充：

1. 命令行 `output_dir`
2. `postman_api_tester/config.py` 中 `REPORT_OUTPUT_DIR`
3. 默认 `项目根目录/reports/`

最常用示例：

```powershell
python -m postman_api_tester.postman_api_tester "支付（UPS）.postman.json"
```

如果只是第一次验证环境是否打通，建议先用 `sample_api_collection.json` 或最小 Collection，不要一开始就跑几百个接口。

### 5.2 交互式快速执行

```powershell
python postman_api_tester/run_test_and_open.py
```

适合不熟悉命令行参数的同学，脚本会依次引导输入：

1. Postman 文件
2. base_url
3. 报告输出目录
4. 每页条数

说明：

1. 报告中心首页上传区支持先“解析接口”，再选择“全量执行”或“仅执行已选接口”。
2. 仅执行已选接口模式下，未选择接口会被后端拦截并提示，不会误执行全量。

## 6. 报告说明

每次执行会生成：

- postman_report_时间戳.html：主报告
- postman_report_时间戳_details.json：详情数据
- postman_report_时间戳_meta.json：历史索引与报告比对元数据

报告中可查看：

- 请求方法、URL、状态码
- 请求头/参数/请求体
- 响应头/响应体
- 汇总统计（通过/失败/错误）

特别说明：

1. `_details.json` 用于详情页按需读取，不建议手工修改。
2. `_meta.json` 用于报告中心首页、差异对比、元数据接口读取。
3. 报告中心默认不展示 `_page_*.html` 分页子页。

## 7. 报告中心与历史对比

### 7.1 启动方式

```bash
cd d:\tangzk\py\seldom-api-testing
python report_server.py
```

启动后可使用：

1. 本机访问：`http://127.0.0.1:5000`
2. 局域网访问：`http://你的局域网IP:5000`

### 7.1.1 服务进程管理（查找、停止与重启）

当页面返回 404、行为不符合预期，或修改代码后需要生效时，通常需要停止旧进程并重启服务。

#### 1. 确认服务是否在运行

打开浏览器访问 `http://127.0.0.1:5000`，如果返回 `Not Found` 或页面内容与预期不符，可能是旧进程仍在运行。

也可用命令行快速验证：

```bash
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:5000/
```

返回 `200` 表示服务正常响应；`404` 或其他状态码需进一步排查。

#### 2. 查找占用 5000 端口的进程

```bash
# Windows（Git Bash / MSYS2 环境）
netstat -ano | grep ":5000.*LISTEN"
```

输出示例：

```
TCP    0.0.0.0:5000    0.0.0.0:0    LISTENING    48500
```

最后一列即为进程 PID（上例为 `48500`）。

如果 `grep` 不方便，也可以用 PowerShell（需管理员权限或调整执行策略）：

```powershell
Get-NetTCPConnection -LocalPort 5000 -State Listen | Select-Object LocalPort, OwningProcess, State
```

#### 3. 停止旧进程

查到 PID 后，逐个终止：

```bash
taskkill //F //PID <PID>
```

示例：

```bash
taskkill //F //PID 48500
```

如果同一端口有多个 Python 进程（常见于多次启动未正常退出），需要逐个终止，直到 `netstat` 不再显示 `LISTENING` 状态：

```bash
netstat -ano | grep ":5000.*LISTEN"
# 如果输出为空，说明端口已释放
```

#### 4. 重新启动服务

```bash
cd d:\tangzk\py\seldom-api-testing
python report_server.py
```

启动成功后终端会显示：

```
 * Running on http://127.0.0.1:5000/
```

此时浏览器刷新 `http://127.0.0.1:5000` 即可正常访问。

#### 5. 常见问题

| 现象 | 原因 | 处理方式 |
|------|------|---------|
| 访问首页返回 404 | 旧进程仍在运行，未加载最新代码 | 按上述步骤终止旧进程后重启 |
| 启动时报 `Address already in use` | 端口被其他进程占用 | 先查找并终止占用端口的进程 |
| 终端无输出且浏览器无响应 | 服务未成功启动 | 检查 `requirements.txt` 依赖是否已安装 |
| 局域网其他机器无法访问 | 防火墙未放通 5000 端口 | 执行 `allow_report_server_firewall.ps1` |

### 7.2 页面说明

1. 打开首页后可查看历史总报告列表，分页子报告页不会在首页展示。
2. 首页支持“新增接口测试”入口按钮，点击后进入独立 ad-hoc 页面，可不依赖上传文件直接录入接口并执行。
3. 点击报告名称或“数据视图”可直接进入单份报告的数据浏览页。
4. 在单报告页可按接口名称或路径搜索，并支持结果状态筛选。
5. 单报告页支持分页显示，默认 20 条，最高 100 条。
6. 点击“详情”可查看请求头、查询参数、请求体、响应头、响应体。
7. 失败或异常结果中，可查看数据库迁移反馈信息，辅助判断是否为数据库切换问题。
8. 点击“原始HTML”可打开生成时的原始静态报告页面。
9. 在首页选择两份历史报告后，可执行结构化差异对比。
10. 若要给其他同事访问，只需保证双方在同一局域网，且本机防火墙已放通 5000 端口。
11. 单报告页“导出最新 JSON”支持两种范围：`full`（全量）和 `report_only`（仅本次报告接口）。
12. 当源集合接口与本次报告接口完全一致时，`report_only` 与 `full` 的导出内容会一致，导出弹窗会显示提示说明。
13. 报告结果与人工用例的“排除/取消排除”使用统一排除键规范，旧格式排除键会自动兼容，`include_excluded=false` 可正确隐藏已排除项。
14. 单报告页支持人工用例三段式编辑流程：发送请求 -> 自动回填响应 -> 选择 PASSED/FAILED 后保存。
15. 结果列表操作列支持行内“排除/取消排除”，可直接切换单条结果的排除状态。
16. 人工用例“发送请求”使用后端代理接口，仅允许 `http/https`，请求超时遵循 `config.py` 统一配置。
17. 编辑重试接口 `/re-request-api` 已同步启用 `http/https` 校验，与代理接口统一 SSRF 防护口径。
18. 报告页修复双编辑器 Body 模式隔离（mc 与 er 互不影响），避免切换时互相污染。
19. 报告页 Body 类型选择体验优化：支持 none/raw/urlencoded/formdata/graphql/binary，切换后与重试区编辑器互不串扰。
20. 报告页顶部新增“页面模板更新时间”小标记，用于确认当前页面是否为最新模板版本。
21. 人工用例详情中的响应体展示与报告详情统一为同款展示逻辑（支持预览与展开）。
22. 编辑人工用例时回显历史请求信息（Headers/Params/Body），便于改参后重新发送。
23. 导出结果一致性修复：导出弹窗中的人工用例数量与实际导出内容一致，且已排除接口不会被导出。
24. 导入执行一致性修复：导出后二次导入时，若 `request.url` 仅含 `raw`（无 `path`），执行层仍可正确解析并执行，避免“解析 2 条、执行 1 条”。
25. 导出参数完整性修复：人工用例导出时同步保留 `request_info` 的 headers/params/body（含 body_mode/body_data），导入后可完整解析并执行。
26. 结果详情支持人工判定状态：可直接“标记成功/标记失败”，并可“恢复自动结果”；状态变更会同步回写报告统计。
27. GET 实际请求 URL 去重修复：当配置 URL 与 params 同时带 query 参数时，执行与展示链路统一去重合并，不再出现参数重复。
28. **响应时间展示**：数据视图结果列表新增"耗时(ms)"列；底部汇总区显示 avg（平均）、max（最大）、p95（第 95 百分位）响应时间，受配置项 `ENABLE_RESPONSE_TIME` 控制。

### 7.2.2 full 与 report_only 的区别和使用方法（重点）

两种导出范围都遵循同一总规则：

1. 先基于源 Collection 生成导出内容。
2. 再合并人工用例（`manual_cases`）。
3. 最后剔除已排除项（`manual_exclusions`）。

核心差异仅在“步骤 1 的基底范围”不同：

| 维度 | `full`（全量导出） | `report_only`（导出本次报告接口） |
|---|---|---|
| 基底接口范围 | 源 Collection 全部接口 | 仅本次报告实际执行到的接口（按 `item_path` 裁剪） |
| 是否保留未执行接口 | 保留 | 不保留 |
| 导出文件典型用途 | 保留完整集合做后续回归 | 产出最小复现集合用于问题定位/复测 |
| 与 `full` 何时相同 | - | 当“本次报告接口集合 == 源 Collection 接口集合”时，两者内容会相同 |

什么时候两者会不同：

1. 本次执行是“仅执行已选接口”或其它非全量场景。
2. 源 Collection 后续新增了接口，但本次报告未覆盖到。
3. 本次报告来自局部执行（例如临时挑选接口）而非全量回归。

什么时候两者看起来一样（这是正常现象）：

1. 本次报告本身就是对该源 Collection 的全量执行。
2. 源 Collection 仅包含本次报告那批接口。

如何选择（实操建议）：

1. 需要“完整回归包”或希望继续覆盖未执行接口：选 `full`。
2. 需要“最小复现包”给开发/联调快速复测：选 `report_only`。
3. 不确定时先选 `full`；若目标是复现本次问题再切 `report_only`。

页面使用步骤：

1. 进入单报告“数据视图”页面，点击“导出最新 JSON”。
2. 在下拉中选择导出范围：`全量导出` 或 `导出本次报告接口`。
3. 如需保留认证头再勾选“导出包含 token/authorization”。
4. 点击导出后，弹窗会显示导出范围与数量信息；若两种范围等价会明确提示原因。

导出通道策略说明：

1. `auto`：总接口数达到阈值时优先走 `/api/export-collection-stream`；若流式失败自动回退 `/api/export-collection`。
2. `legacy`：强制走 `/api/export-collection`，适合对比历史链路或排查流式问题。
3. `stream`：强制走 `/api/export-collection-stream`，适合验证流式链路。

### 7.2.1 直接新增接口测试（ad-hoc）使用说明

适用场景：

1. 只想快速验证 1~N 条接口，不想先整理完整 Collection。
2. 联调期需要临时录入 headers/params/body 并直接生成标准报告。

操作步骤：

1. 启动 `python report_server.py` 并打开首页。
2. 点击首页“打开新增接口测试页”，进入独立页面后填写 `collection_name`、`base_url`、`token`、`report_name`、`results_per_page`（均可按需填写）。
3. 新增接口行并录入 `name`、`folder`、`method`、`url`、`headers`、`params`、`body_mode`、`body_data`。
4. 点击执行后，前端调用 `/api/run-ad-hoc-tests`，后端会生成临时 Collection 并复用现有任务队列执行。
5. 继续沿用现有任务状态轮询与报告查看流程，执行完成后进入新报告页。

输入约束：

1. `cases` 至少 1 条，且总数不得超过 `postman_api_tester/config.py` 中 `ADHOC_MAX_ITEMS`。
2. `url` 支持三种形态：绝对 `http/https`、`{{baseUrl}}/...` 或 `{{base_url}}/...`、以及配合 `base_url` 的相对路径。
3. `body_mode` 支持 `none/raw/urlencoded/formdata/graphql/binary`，与既有导出/重试/代理请求语义保持一致。
4. 使用变量 URL 或相对路径时，必须提供合法 `base_url`；`base_url` 与绝对 URL 均只允许 `http/https`。
5. ad-hoc 任务会在 `uploaded_collections/` 中生成临时 JSON 文件，报告产物仍按标准三件套落在报告目录。

### 7.3 报告目录一致性

报告中心读取目录优先级：

1. `POSTMAN_REPORTS_DIR`
2. `REPORTS_DIR`
3. `postman_api_tester/config.py` 中 `REPORT_OUTPUT_DIR`
4. 默认 `项目根目录/reports/`

健康检查补充：

1. `GET /health` 仍保留 `status` 与 `timestamp`。
2. 当启用日志告警策略时，返回体扩展 `log_alert` 字段（ERROR 速率快照与阈值对比结果）。

按此规则可确保“测试执行输出目录”和“报告中心读取目录”一致。

### 7.4 防火墙说明

如果其他机器无法访问，但本机可以访问，请检查 Windows 防火墙是否允许 Python 或 5000 端口的入站访问。

## 9. 故障排查

- 未登录/401：优先确认 token 来源和优先级
- URL错误：检查 base_url 覆盖逻辑
- 接口超时：确认网络、防火墙、目标服务健康状态
- message 中出现“数据库反馈”：优先按提示检查连接、权限、SQL 方言、schema 或字符集问题
- 导出后二次导入出现“预览 2 条但执行 1 条”：升级至当前版本后已修复；根因是旧逻辑仅按 `request.url.path` 解析，未覆盖仅 `raw` 的 URL 结构
- 导出后二次导入后人工用例请求体显示“无”或执行失败：当前版本已修复；根因是旧导出链路写入人工用例时只保留 method/url，未带出 headers/params/body
- 人工用例新增后显示在列表最后：当前版本已修复；读取人工用例时不再为缺失 `created_at` 的旧数据临时补当前时间，避免排序漂移
- 数据视图搜索对人工用例无效：当前版本已修复；搜索词/状态/排除开关已同步作用于人工用例列表
- 报告为空：确认 JSON 是否包含 item/request 结构
- 局域网不可访问：检查 `python report_server.py` 是否已启动，以及防火墙是否放通 5000 端口
- 无法对比历史报告：确认 `reports/` 下同时存在 `postman_report_*.html` 与对应的 `postman_report_*_meta.json`

2. 进入项目目录：`cd d:\tangzk\py\seldom-api-testing`
3. 安装依赖：`pip install -r requirements.txt`
4. 修改 `postman_api_tester/config.py`：`BASE_URL`、`TOKEN`、`REPORT_OUTPUT_DIR`

### 13.3 执行命令

1. 执行测试：`python -m postman_api_tester.postman_api_tester "支付（UPS）.postman.json"`
2. 启动报告中心：`python report_server.py`
3. 浏览器访问：`http://127.0.0.1:5000`

### 13.4 交付建议

首次交付时建议对方先按本手册第 3、4、7 节执行；如果只需要快速上手，可先阅读 [README.md](d:/tangzk/py/seldom-api-testing/postman_api_tester/README.md)。
