8. 导出支持 `full`（全量）与 `report_only`（本次报告接口）双模式。

## 4. 交付后对方应按什么顺序使用

建议告知对方按以下顺序执行：

1. 先阅读 `postman_api_tester/README.md` 了解目录和上手顺序。
2. 安装 Python 3.10+。
3. 在项目根目录执行 `pip install -r requirements.txt`。
4. 修改 `postman_api_tester/config.py` 中的 `BASE_URL`、`TOKEN`、`REPORT_OUTPUT_DIR`。
5. 先用 `sample_api_collection.json` 或最小 Collection 验证环境打通。
6. 再执行正式业务 `*.postman.json`。
7. 通过 `python report_server.py` 启动报告中心。
8. 进入 `http://127.0.0.1:5000` 查看历史报告、失败详情和数据库反馈。

## 5. 本次交付的关键配置点

交付时至少需要提醒对方以下配置：

1. `BASE_URL`：目标系统地址。
2. `TOKEN`：认证 token，可为空。
3. `REPORT_OUTPUT_DIR`：报告输出目录。
4. `REQUEST_CONNECT_TIMEOUT` / `REQUEST_READ_TIMEOUT`：请求超时。
5. `RUN_JOBS_MAX`：报告中心任务记录上限。
6. `ENABLE_SELECTIVE_RUN`：是否启用仅执行已选接口。
7. `REPORT_EXPORT_DEFAULT_SCOPE`：导出默认范围（`full` / `report_only`）。

如不希望对方改代码文件，也可以提示使用环境变量：

```powershell
$env:POSTMAN_BASE_URL = "http://10.50.11.130:11000"
$env:POSTMAN_TOKEN = "your_token"
python -m postman_api_tester.postman_api_tester "sample_api_collection.json"
```

## 6. 交付后的验证要求

应至少覆盖以下验证项：

1. 模块导入成功

4. 报告中心可读取自定义报告目录

7. 关键接口状态码与历史基线一致

4. 阶段性回归或发版前接口健康度检查。
