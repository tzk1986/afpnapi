适用范围：postman_api_tester/report_server.py 及其 handlers/services/utils 分层

## 1. 现状与约束基线

### 1.1 当前已具备能力（可复用）

1. 报告基础数据完整：主报告 summary、results、details、meta 三件套齐全。
2. 已有报告对比能力：/api/compare 可提供新增/移除/状态变化的基础数据。
3. 已有结果筛选能力：/api/report-results 支持状态、关键字、错误码筛选。
4. 已有响应时间字段：result 级 response_time_ms，summary 级 avg/max/p95。
5. 已有服务分层：路由薄层 + services 聚合 + repository 读写，可最小侵入扩展。

### 1.2 必须遵守的稳定性约束

1. 不改变现有接口语义：/api/reports、/api/report-results、/api/report-meta、/api/compare 保持兼容。
2. 首页轻量化约束必须保持：不在首页注入完整 results/request_info/response_info。
3. 配置集中化：新开关与阈值统一放入 config.py，避免页面与路由硬编码。
4. 外置模板约束：所有前端展示改动仅落在 templates/*.html。
5. 冒烟回归约束：健康、首页、数据视图、report-results true/false、manual-cases CRUD、export 需保底通过。

## 2. 目标能力拆解（2.1）

### 2.1 分析仪表板

新增分析接口，返回以下聚合结果：

1. 基础分布
- 状态分布：PASSED/FAILED/ERROR
- 方法分布：GET/POST/PUT/PATCH/DELETE/其他
- 目录分布：folder 维度 TopN

2. 响应时间分布
- 直方图桶分布（默认桶）：[0, 50, 100, 200, 500, 1000, 3000, 5000, +Inf] ms
- 分位数回显：avg、p50、p95、p99、max

3. 错误类型排序
- 连接类错误、认证类错误、业务类错误、断言类错误、数据库兼容类错误、未知错误
- TopN 错误消息（按归一化 message 聚合）

4. 覆盖率视图
- 执行覆盖率：本次执行结果数 / 源集合接口数
- 人工覆盖率：人工用例数 / 源集合接口数
- 未覆盖清单（支持返回 TopN，默认 200）

### 2.2 故障诊断洞察

在 analytics 返回结构中增加 diagnostics：

1. category_summary：按错误类别计数与占比。
2. frequent_errors：高频错误消息及受影响接口样本。
3. suggestions：按类别自动给出建议（复用 db_feedback 思路，扩展到非数据库场景）。

### 2.3 接口质量评分

定义可配置评分模型（默认）：

1. 基础分 100。
2. 扣分项
- FAILED: 每条 -10
- ERROR: 每条 -15
- 慢请求惩罚：response_time_ms > p95 时每条 -5
- 缺少断言惩罚：无 assertion_results 且启用断言场景下每条 -2
3. 最终分数下限 0，上限 100。
4. 输出总分 + 维度分（稳定性/性能/断言完整性）。

### 2.4 趋势与对比

1. 单报告对比：通过已有 /api/compare 输出 success_rate_delta。
2. 历史趋势：按最近 N 份同集合报告（默认 20）输出
- success_rate trend
- avg_response_ms trend
- failed_count trend

## 3. 设计原则（保证现有功能稳定）

1. 新增而非替换：新增 analytics 路由，不改已有路由出参结构。
2. 读路径优先：第一阶段只读 report/meta/details，不引入写路径副作用。
3. 纯函数聚合：聚合逻辑放 services，避免路由层复杂化。
4. 分级开关：通过配置开关可即时禁用 2.1 页面与接口。
5. 渐进发布：后端先行、前端后挂载，默认不影响现有页面主流程。

### 3.1 可实施性确认（基于当前代码基线）

结论：当前方案可实施，且可按“最小侵入”路径落地。

支撑依据：

1. 代码分层已稳定：report_server.py 路由层 + handlers/services/utils 可承载新增只读聚合能力。
2. 数据源已具备：report/meta/details 已包含 2.1 所需关键字段（status/method/folder/message/err_code/response_time_ms/manual_cases）。
3. 兼容策略可执行：可通过新增路由与新增配置开关落地，不触碰现有接口返回结构。
4. 风险可控：已有 mypy 与 smoke 基线，可作为实施门禁。

实施前提：

1. 仅按本方案的阶段 0-3 执行，不跨阶段并行塞入新需求。
2. 先后端只读能力，后前端挂载；未达门禁不得进入下一阶段。
3. 所有新增配置默认保持“关闭或兼容旧行为优先”。

## 5. 接口契约（建议）

### 5.1 新增接口 A：单报告分析

路由：GET /api/report-analytics/<report_name>

请求参数：

1. top_n（默认 10，范围 1-100）
2. trend_limit（默认 20，范围 1-100）
3. include_samples（默认 false）

响应结构（核心字段）：

1. report_name
2. summary_snapshot
3. distributions
- status
- method
- folder_top
4. performance
- histogram
- quantiles (avg/p50/p95/p99/max)
5. diagnostics
- category_summary
- frequent_errors
- suggestions
6. quality_score
- total_score
- stability_score
- performance_score
- assertion_score
- penalties
7. coverage
- source_total
- executed_total
- manual_cases_total
- execution_coverage
- manual_coverage
- uncovered_top
8. trend
- success_rate
- avg_response_ms
- failed_count

### 5.2 新增接口 B：分析对比（可选，第二阶段）

路由：GET /api/report-analytics-compare

请求参数：left, right

响应结构：

1. left_snapshot, right_snapshot
2. delta
- success_rate_delta
- avg_response_delta_ms
- failed_delta
- error_delta
3. score_delta

说明：若不新增 B，可先复用已有 /api/compare + A 接口前端拼接。

## 7.1 防偏离开发约束（范围冻结与变更控制）

目标：防止实施过程中“边做边加需求”，导致方案失控、质量下滑和延期。

### A. 范围冻结规则（Scope Freeze）

1. 本次迭代只交付 2.1 范围：分析仪表板、故障诊断、质量评分、覆盖率、趋势。
2. 以下内容明确排除（Out of Scope）：
- 批量任务与优先级（1.2）
- 环境管理后台（2.2 UI）
- 性能基准管理与告警联动（2.3 全量）
- 测试用例模板/版本/DSL（2.4）
- 新鉴权体系、用户隔离体系、消息通知集成
3. 任何新增需求默认不进入当前迭代，先记录到“下一迭代候选池”。

### B. 变更控制流程（Change Control）

1. 变更单必填字段：
- 变更原因
- 影响范围（接口/页面/配置/测试）
- 风险等级（低/中/高）
- 是否破坏兼容
- 回滚方案
2. 变更审批门槛：
- 低风险：负责人审批
- 中/高风险：负责人 + 测试双签
3. 未审批变更不得进入开发分支。

### C. 需求截断规则（No Feature Creep）

1. 每阶段只允许修复阻塞性缺陷，不允许顺手扩展功能。
2. 发现“看起来顺便可以做”的需求，统一登记，不在当前 PR 实施。
3. 单 PR 变更文件上限建议：
- 后端阶段：<= 6 个文件
- 前端阶段：<= 3 个文件
4. 超出上限需拆分 PR，并重新走门禁。

### D. PR 与验收约束（Definition of Done）

1. 每个阶段必须提交“阶段验收记录”：
- 已完成项
- 未完成项
- 风险与遗留
- 回滚验证结果
2. PR 必须附：
- 接口样例（请求/响应）
- 回归结果（必跑用例）
- 性能对比（若涉及性能）
3. 任一门禁失败，禁止合并。

### E. 稳定性红线（Hard Stop）

触发以下任一条件，立即停止新增开发，仅允许修复：

1. 既有接口契约被破坏。
2. 既有 smoke 用例任意失败。
3. 开关关闭后行为与当前版本不一致。
4. 无法提供可执行回滚路径。

### F. 文档同步纪律

1. 每阶段完成后仅更新对应文档，不提前写“已完成”结论。
2. 文档版本、发布日期、能力描述必须四份同步（README/操作手册/快速命令参考/最终交付清单）。
3. 开发阅读文档的约束记录必须按时间升序追加，禁止插入历史时间段。

### G. 角色职责与决策边界

1. 产品/需求方：确认范围与优先级，不直接改动技术门禁。
2. 开发负责人：控制范围、拆分阶段、批准低风险变更。
3. 测试负责人：把关回归门禁与兼容性，拥有阻断合并权。
4. 任何人无权绕过门禁直接上线。

## 8. 风险清单与规避策略

### 风险 1：大报告聚合耗时高，拖慢页面

规避：

1. 分层缓存：对 analytics 结果加短 TTL 缓存（建议 30s，沿用报告缓存节奏）。
2. 按需字段：include_samples 默认 false，避免大样本传输。
3. 限流参数：top_n、trend_limit 强制上限。

### 风险 2：错误分类误判，导致建议不可信

规避：

1. 分类结果输出 confidence 或匹配规则来源。
2. 未命中统一归类 unknown，不强行归因。
3. 规则字典放可维护配置区，支持后续扩展。

### 风险 3：修改现有 payload 导致前端回归

规避：

1. 只新增路由，不修改既有接口字段。
2. 现有模板 JS 不做侵入式改写，analytics 独立挂载。
3. 增量发布：先后端开关关闭发布，再灰度开启。

### 风险 4：旧报告缺字段导致聚合异常

规避：

1. 所有字段读取走归一化默认值。
2. response_time_ms 缺失按 null 处理，不参与分位数。
3. assertion_results 缺失按空列表处理。

### 风险 5：与现有缓存机制冲突

规避：

1. analytics 缓存独立命名空间，不复用列表缓存对象。
2. 报告写回后统一失效：沿用 invalidate_reports_cache，并补 analytics 缓存失效钩子。

## 9. 稳定性保障与回归门禁

### 9.1 必跑回归（不低于现有约束）

1. /health
2. /
3. /report-view
4. /api/report-results?include_excluded=true
5. /api/report-results?include_excluded=false
6. /api/manual-cases add/update/delete
7. /api/export-collection
8. 新增 /api/report-analytics/<report_name>

### 9.2 兼容性校验

1. 旧报告（缺 response_time_ms、缺 assertion_results）可正常返回 analytics。
2. 大报告（1000+）接口耗时不超过基线阈值。
3. 开关关闭后页面和接口行为与当前版本一致。

### 9.3 发布策略

1. 第一步：代码上线，ENABLE_REPORT_ANALYTICS=false。
2. 第二步：预发打开开关，跑全链路 smoke + 人工验收。
3. 第三步：生产灰度开启，观察 24-48 小时。
4. 第四步：全量开启。

### 9.4 回滚策略

1. 快速回滚：仅关闭 ENABLE_REPORT_ANALYTICS。
2. 路由保留但返回 403，不影响其余能力。
3. 不回滚已有 report/meta/details 数据结构。
