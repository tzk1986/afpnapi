## 1. 方案 1.1：错误恢复与容错（Error Recovery & Fault Tolerance）

## 1.1 子能力 A：任务 Checkpoint 与断点恢复

### 1.1.1 目标

当执行过程中出现网络波动、进程中断或任务异常时，可基于稳定标识恢复，避免重复执行已完成接口。

### 1.1.2 技术方案

1. 新增配置项（`postman_api_tester/config.py`）
- `ENABLE_CHECKPOINT_RECOVERY`（默认 False）
- `CHECKPOINT_FLUSH_EVERY_N`（默认 1，表示每条结果落盘）
- `CHECKPOINT_DIR`（默认 `reports/checkpoints`）

2. 新增 Checkpoint 数据模型（建议 JSON）
- `job_id`
- `collection_fingerprint`（输入集合内容摘要）
- `base_url`
- `created_at` / `updated_at`
- `executed_item_paths`（使用稳定 key：`item_path_text`，如 `0.2.5`）
- `completed`（bool）
- `last_error`（可选）

3. 写入策略
- 原子写：先写临时文件，再 rename 覆盖
- 读取校验：字段完整性 + JSON 格式校验
- 校验失败：自动降级为“从头执行”，并记录告警日志

4. 恢复策略
- 恢复前先校验 `collection_fingerprint` 与 `base_url` 是否匹配
- 匹配：跳过已执行 `item_path`
- 不匹配：拒绝恢复并走全量执行

### 1.1.3 风险与缓解

1. 风险：Checkpoint 文件损坏导致恢复失败
- 缓解：原子写 + 读取校验 + 降级全量执行

2. 风险：断点恢复重复执行接口
- 缓解：仅使用 `item_path` 作为稳定键，不使用 name/url

3. 风险：恢复到错误集合版本
- 缓解：引入 `collection_fingerprint` 强校验

### 1.1.4 回滚设计

1. 关闭 `ENABLE_CHECKPOINT_RECOVERY` 即恢复原执行路径
2. Checkpoint 文件保留但不参与执行
3. 不影响原 `run_postman_tests()` 入参与返回结构

---

## 1.2 子能力 B：部分成功报告（Partial Success Report）

### 1.2.1 目标

即使执行中发生局部异常，也必须产出完整报告三件套，避免整批结果丢失。

### 1.2.2 技术方案

1. 报告落盘策略调整
- 执行阶段持续积累已完成结果
- 出现中断异常时，进入“收尾生成”分支
- 强制生成 html/details/meta 三件套

2. 元数据增强（增量字段）
- `meta.execution_mode`: `full` / `partial`
- `meta.interrupted`: true/false
- `meta.interrupt_reason`: string

3. 报告命名策略
- 部分成功报告必须使用新时间戳命名
- 禁止覆盖既有报告

### 1.2.3 风险与缓解

1. 风险：部分报告覆盖原报告
- 缓解：强制新 report_name（时间戳唯一）

2. 风险：前端无法识别部分报告
- 缓解：只做字段新增，不改变现有 summary 结构

3. 风险：用户误解“部分成功”为“全部成功”
- 缓解：在 meta 增加中断标识，前端可选显示提示（后续可单独 UI 小改）

### 1.2.4 回滚设计

1. 如出现兼容问题，保留三件套强制生成能力但隐藏新增字段消费
2. 前端仅按既有字段渲染，不受影响

## 1.3 子能力 C：断言降级与严格模式

### 1.3.1 目标

断言引擎异常不应导致整条接口执行崩溃；通过可配置严格模式控制行为。

### 1.3.2 技术方案

1. 新增配置项（`postman_api_tester/config.py`）
- `ENABLE_ASSERTION_STRICT_MODE`（默认 False）

2. 行为定义
- 严格模式 True：断言执行异常 -> 当前用例 FAILED
- 严格模式 False：断言执行异常 -> 记录 `assertion_error`，接口状态按 HTTP 结果判定

3. 结果结构增强
- 新增 `assertion_engine_error`（string，可空）
- 保留现有 `assertion_results` 字段

### 1.3.3 风险与缓解

1. 风险：非严格模式掩盖断言脚本问题
- 缓解：日志中强告警 + 报告中显式字段

2. 风险：严格/非严格口径混淆
- 缓解：在 meta 中记录 `assertion_strict_mode` 实际值

### 1.3.4 回滚设计

1. 默认 False，本质不改变当前主流程
2. 出现争议可先冻结配置开关，不改代码路径

## 2. 方案 1.3：响应时间性能优化（Response Time Performance Optimization）

## 2.1 子能力 A：报告列表懒加载（只读 meta 摘要）

### 2.1.1 目标

`list_reports()` 仅加载列表摘要，避免在首页链路读取完整 `results` 大数组。

### 2.1.2 技术方案

1. 后端策略
- `list_reports()` 仅从 `_meta.json` 读取：报告名、集合名、来源、时间、summary
- 明确禁止读取 `results` 全量内容进入首页内存

2. 缓存策略
- 保持现有 30 秒 TTL
- 保持 `_invalidate_reports_cache()` 触发机制不变

3. 数据结构
- 缓存对象新增 `summary_only=true` 标识（内部字段）
- 对外接口字段保持兼容

### 2.1.3 风险与缓解

1. 风险：破坏现有缓存行为
- 缓解：不更改缓存生命周期，只更改缓存内容粒度

2. 风险：首页筛选依赖字段不足
- 缓解：保证 `summary` 与筛选依赖字段完整保留

### 2.1.4 回滚设计

1. 保留旧读取函数（仅内部），可通过快速切换回旧策略
2. 发现问题时仅回滚列表读取逻辑，不影响详情页

---

## 2.2 子能力 B：详情页虚拟渲染（前端分片绘制）

### 2.2.1 目标

详情页面对 1000+ 行结果时避免主线程长时间阻塞和页面卡顿。

### 2.2.2 技术方案

1. 模板修改位置
- `templates/report_view.html`

2. 渲染策略
- 数据仍保留在前端数组
- 初始只渲染可视区域 + 缓冲区
- 滚动时按窗口更新 DOM

3. 功能兼容
- 搜索/筛选逻辑在完整数组上执行
- 展示层仅虚拟化，不改变筛选语义

### 2.2.3 风险与缓解

1. 风险：虚拟渲染导致搜索结果错位
- 缓解：搜索先产出 filteredData，再交给虚拟渲染器

2. 风险：展开详情状态在滚动中丢失
- 缓解：维护 `expandedRowKeys` 状态映射

3. 风险：键盘操作体验退化
- 缓解：保留现有交互快捷键，并补充可访问性测试

### 2.2.4 回滚设计

1. 增加前端开关 `useVirtualRender`（默认 true，仅前端变量）
2. 异常时可退回分页全量渲染模式

---

## 2.3 子能力 C：流式导出（Streaming Export）

### 2.3.1 目标

大文件导出时降低峰值内存占用并提升首字节响应时间。

### 2.3.2 技术方案

1. 新增接口
- 新增 `/api/export-collection-stream`
- 保留 `/api/export-collection` 原行为不变

2. 输出方式
- 采用 chunked 流式响应
- 保持导出 JSON 结构与原接口一致（统一导出约束不变）

3. 兼容策略
- 前端默认仍可调用旧接口
- 新入口可在高级场景按需启用

### 2.3.3 风险与缓解

1. 风险：老客户端不支持流式
- 缓解：旧接口不动，流式为新增能力

2. 风险：流式中断产生不完整 JSON
- 缓解：增加流式写异常捕获和错误日志；客户端失败时可回退旧接口

### 2.3.4 回滚设计

1. 下线新接口即可，旧接口无感
2. 不影响既有导出链路

## 3. 新增约束（拟追加到开发阅读文档）

建议新增为约束 45~52：

45. **Checkpoint 恢复键约束**：断点恢复必须使用稳定 `item_path` 作为已执行唯一键，禁止使用 `name/url` 作为去重键。  
46. **Checkpoint 安全写约束**：checkpoint 文件写入必须采用原子写（tmp + rename），读取前必须进行结构校验；校验失败必须降级为全量执行，不得中断主流程。  
47. **断点恢复默认关闭约束**：`ENABLE_CHECKPOINT_RECOVERY` 默认 `False`，未显式开启时执行行为必须与 v1.2.0 一致。  
48. **部分成功报告三件套约束**：执行中断时仍必须生成 html/details/meta 三件套，并以新时间戳命名，禁止覆盖历史报告。  
49. **断言严格模式约束**：新增 `ENABLE_ASSERTION_STRICT_MODE`，严格模式下断言引擎异常判定 FAILED，非严格模式下仅记录 `assertion_engine_error` 不得使整条接口 ERROR。  
50. **报告列表内存约束**：`list_reports()` 首页链路不得加载完整 `results` 数组到内存，仅允许读取 meta 摘要字段；并继续遵守 30 秒缓存与 `_invalidate_reports_cache()` 失效机制。  
51. **详情页大数据渲染约束**：详情页在 1000+ 结果场景下不得出现明显卡死（可操作阈值：主线程无连续 >2s 阻塞），搜索/筛选语义必须保持与原逻辑一致。  
52. **导出兼容约束**：流式导出能力必须以新增接口提供，不得替换或改变 `/api/export-collection` 既有返回行为与调用方式。

## 4. 进度监控与验收检查点

## 4.1 1.1 检查点（3 个里程碑）

### M1：Checkpoint 基础可用

1. 开启开关后，首次执行生成 checkpoint 文件
2. 人工中断后可恢复执行

### M3：断言模式双态通过

1. 严格模式：断言引擎异常 -> FAILED

## 4.2 1.3 检查点（3 个里程碑）

### P1：列表懒加载达标

1. 100 份报告场景下首页接口响应时间明显下降
2. 服务进程内存峰值低于改造前
3. 缓存命中行为与失效行为保持一致

### P2：详情页渲染达标

1. 1000+ 行数据首屏可在 2 秒内出现可交互内容（目标）
2. 滚动无明显掉帧卡顿

### P3：流式导出可用

1. 新接口可下载与旧接口等价结构 JSON
