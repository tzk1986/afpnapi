## 二、P0 — 必须修复

### 2.1 异常处理精细化（session.py）

**现状**：`session.py` 第 35、46、57 行存在三处 `except Exception:` 宽泛捕获，将 `ConnectionError`、`SSLError`、`TimeoutError` 等具体异常统一吞没，调试时无法区分网络超时、SSL 证书错误和连接拒绝等不同故障。

**目标**：按具体异常类型分别处理，至少区分：
- `requests.exceptions.ConnectionError` → 连接失败
- `requests.exceptions.SSLError` → SSL 证书问题
- `requests.exceptions.Timeout` → 请求超时
- `requests.exceptions.HTTPError` → HTTP 错误码
- `Exception` → 兜底（仅保留一处，记录完整堆栈）

**涉及文件**：
- `postman_api_tester/session.py`（3 处）

**风险评估**：低风险，仅拆分 except 分支，不改变业务逻辑。

---

### 2.2 print() 替换为 logger

**现状**：`core/html_reporter.py` 第 464-481 行有 12 处 `print()` 调用，用于控制台输出报告摘要；`core/execution_helpers.py` 第 682 行有 1 处 `print("生成测试报告...")`。违反开发阅读文档约束 3.4 第 11 条："核心路径禁止新增 `print()` 代替 `logger.*`"。

**目标**：
- 报告摘要输出改为 `logger.info()`，保持格式一致
- "生成测试报告..." 改为 `logger.info("生成测试报告...")`
- 确保 `LOG_SAMPLE_RATE` 等配置仍可控制输出频率

**涉及文件**：
- `postman_api_tester/core/html_reporter.py`（12 处）
- `postman_api_tester/core/execution_helpers.py`（1 处）

**风险评估**：低风险，仅替换输出方式。需注意控制台用户的现有使用习惯。

## 四、P2 — 中优先级

### 4.1 配置访问统一

**现状**：多个模块绕过 `report_server_config.py` 直接从 `config.py` 读取配置，导致配置访问路径不统一：

| 模块 | 行号 | 直接读取的配置项 |
|------|------|----------------|
| `handlers/http_handler.py` | 66 | `REQUEST_CONNECT_TIMEOUT`、`REQUEST_READ_TIMEOUT` |
| `handlers/server_routes.py` | 85 | 多项 |
| `handlers/test_proxy_routes.py` | 53 | `PROXY_ALLOWED_HOSTS` |
| `utils/logging_utils.py` | 162, 173, 205 | `LOG_LEVEL`、`LOG_FORMAT`、`LOG_SAMPLE_RATE` 等 |
| `utils/security.py` | 41 | `SENSITIVE_HEADERS` |

**目标**：
- 将上述配置项统一在 `report_server_config.py` 中暴露
- 各模块改为从 `report_server_config.py` 读取
- `utils/` 模块通过函数参数接收配置值，而非直接引用全局配置

**涉及文件**：
- `postman_api_tester/report_server_config.py`（新增配置暴露）
- 上表 5 个模块（修改导入路径）

**风险评估**：中风险，需确保环境变量和默认值行为不变。

### 4.2 Handler 间直接引用治理

**现状**：部分 handler 模块直接引用其他 handler 的内部函数，违反分层原则（handler 应只调用 service，不应互相调用）：

| 调用方 | 被调用方 | 引用内容 |
|--------|---------|---------|
| `server_routes.py:129` | `admin_handler.py` | `delete_report_artifacts` |
| `job_routes.py:13` | `collection_handler.py` | 3 个函数 |
| `job_routes.py:18` | `job_handler.py` | 2 个函数 |
| `retry_routes.py:10` | `job_handler.py` | 2 个函数 |
| `report_result_routes.py:24` | `report_handler.py` | 多个函数 |
| `report_result_routes.py:28` | `report_analytics_handler.py` | 多个函数 |
| `test_proxy_routes.py:11` | `http_handler.py` | `execute_http_request` |

**说明**：当前 handler 间引用分为两类：
1. **薄导出层调用**（如 `job_routes` → `job_handler`）：这是 v1.3.0 分层后的正常中间层，`*_handler.py` 承担"中间转发"角色，`*_routes.py` 承担"路由装饰"角色。这类引用是合理的。
2. **跨业务域调用**（如 `server_routes` → `admin_handler`）：这类调用绕过了 service 层，应改为通过 service 调用。

**目标**：
- 第二类（跨业务域调用）改为经 service 层中转
- 第一类（薄导出层调用）保持现状，但考虑后续是否需要合并中间层

**涉及文件**：
- `postman_api_tester/handlers/server_routes.py`（修改 `delete_report_artifacts` 调用路径）

**风险评估**：中风险，需验证 service 层已提供等效接口。

---

### 4.3 Utils 模块解耦全局配置

**现状**：`utils/security.py` 和 `utils/logging_utils.py` 通过延迟导入（函数内 `import config`）方式引用全局配置，违反 utils 层"纯函数工具、不持有业务状态"的约束。

**目标**：
- `utils/security.py`：敏感头规则通过参数传入，不再内部读取 config
- `utils/logging_utils.py`：日志配置通过参数传入（`configure_logging_from_config` 已是此模式，但内部 3 处延迟导入可进一步清理）

**涉及文件**：
- `postman_api_tester/utils/security.py`
- `postman_api_tester/utils/logging_utils.py`
- 对应调用方（修改传参方式）

**风险评估**：中风险，涉及接口签名变更。

## 五、P3 — 低优先级

### 5.1 服务层文件 I/O 缓存

**现状**：多个 service 模块在每次请求时执行完整的 JSON 文件读取和写入，无缓存机制：

| 模块 | 操作 |
|------|------|
| `report_judgement_service.py` | 每次判定更新读写 meta JSON |
| `report_meta_update_service.py` | 每次 meta 更新读写 JSON |
| `report_patch_service.py` | 单次操作链中多次嵌套读写 JSON |
| `report_export_service.py` | 全量 JSON 加载/转储 |

**目标**：评估是否可以复用 `report_repository.py` 的 TTL 缓存机制，减少高频读写。

**风险评估**：需评估缓存一致性风险（并发写回场景）。

---

### 5.2 公有函数 Docstring 补齐

- `handlers/job_handler.py`：4 个公有函数

**风险评估**：无风险。

## 七、Out of Scope

以下不在本次优化范围内：

1. 新功能开发（性能优化仅针对既有代码质量）
2. Flask → ASGI 框架迁移（影响范围过大）
3. 数据库持久化（当前内存存储满足需求）
4. 前端模板引擎替换（Jinja2 已满足需求）
5. Python 版本升级适配（当前 3.10+ 已足够）
