## 一、需求概述

在现有 UI 录制/编辑能力基础上，新增 **回放执行引擎**，实现：

1. **浏览器内回放** — 在 iframe 中逐步执行录制步骤，实时显示进度和结果
2. **Headless 无界面执行** — 后台 Playwright 自动执行，支持定时/批量
3. **执行报告** — 每步通过/失败、耗时、截图，生成可视化报告
4. **设置管理** — Web 端配置执行模式、浏览器类型、超时参数

## 三、Step 数据格式（共享契约）

录制器和两种执行引擎使用统一的 step JSON 格式：

```json
{
  "action": "click",
  "selector": {
    "primary": "[data-testid=\"submit-btn\"]",
    "strategy": "test-id",
    "fallback_css": "#submit-btn",
    "fallback_xpath": "//button[1]"
  },
  "value": "typed text",
  "element_info": {
    "tag": "input",
    "text": "",
    "type": "text",
    "name": "username",
    "placeholder": "请输入用户名",
    "visible": true
  },
  "page_url": "https://example.com/login"
}
```

### Action 类型全集

| Action | 浏览器内回放 | Headless | 说明 |
|--------|-------------|----------|------|
| `click` | ✅ | ✅ | 单击 |
| `dblclick` | ✅ | ✅ | 双击 |
| `type` | ✅ | ✅ | 输入文本 |
| `select` | ✅ | ✅ | 下拉选择 |
| `check` / `uncheck` | ✅ | ✅ | 复选框 |
| `select_radio` | ✅ | ✅ | 单选按钮 |
| `submit` | ✅ | ✅ | 表单提交 |
| `keypress` | ✅ | ✅ | 特殊按键 |
| `navigate` | ✅ | ✅ | 页面导航 |
| `new_tab` | ⚠️ | ✅ | 新标签页（浏览器内有限制） |
| `api_call` | ️ | ✅ | API 调用录制（仅记录，不重放） |
| `wait` | ✅ | ✅ | 等待毫秒数 |
| `wait_for` | ✅ | ✅ | 等待元素出现 |
| `assert_visible` | ✅ | ✅ | 断言元素可见 |
| `assert_text` | ✅ | ✅ | 断言文本内容 |
| `assert_url` | ✅ | ✅ | 断言当前 URL |
| `assert_title` | ✅ | ✅ | 断言页面标题 |

### 选择器解析策略

两种执行引擎共享相同的选择器解析逻辑：

```
primary selector（首选策略）
    ↓ 失败
fallback_css（CSS 回退）
    ↓ 失败
fallback_xpath（XPath 最终回退）
```

解析时按 selector 字符串前缀自动识别类型：
- `//` 开头 → XPath
- `text=` 开头 → 文本选择器
- `role=` 开头 → 角色选择器
- `#` 或 `.` 开头 → CSS
- 其他 → 尝试 CSS selector

## 六、Phase 1：浏览器内回放（MVP）

### 6.1 目标

实现最小可行产品：用户在编辑器中点击"执行"，在浏览器内 iframe 中逐步回放录制步骤，实时查看进度和每步结果。

### 6.2 实施步骤

#### Step 1.1：前端回放 JS 引擎

**文件**：`services/ui_recorder_inject.py`

新增 `get_replayer_js(origin)` 函数，生成内联 JS 代码注入到 iframe 中。

**核心功能**：
- 复用 `SelectorEngine`（已有代码）
- `ReplayEngine.init(steps, options)` — 初始化
- `ReplayEngine.start()` — 开始执行
- `ReplayEngine._executeAction(step)` — 执行单个 action
- `ReplayEngine._findElement(selectorStr, selectorObj)` — 查找元素（含 fallback 链）
- `ReplayEngine._highlightElement(step)` — 高亮当前操作元素
- `ReplayEngine._notifyParent(eventType, data)` — 通过 postMessage 通知父页面

**支持的操作**：
```javascript
switch (step.action) {
  case 'click':     el.click();
  case 'dblclick':  el.dispatchEvent(new MouseEvent('dblclick', {bubbles:true}));
  case 'type':      el.value = text; dispatch input + change events;
  case 'select':    el.value = value; dispatch change event;
  case 'check':     el.checked = true/false; dispatch change event;
  case 'submit':    form.submit();
  case 'navigate':  parent.postMessage({type:'ui-replay-navigate', url: ...}, '*');
  case 'wait':      await sleep(ms);
  case 'assert_visible': return el ? el.offsetParent !== null : false;
  case 'assert_text': return (el.textContent || '').indexOf(expected) >= 0;
  case 'assert_url': return location.href === expected;
  case 'assert_title': return document.title.indexOf(expected) >= 0;
}
```

**通信协议**：
```
父页面 → iframe（postMessage）:
  { type: 'ui-replay-control', action: 'init' | 'start' | 'pause' | 'stop' | 'next', steps?: [], options?: {} }

iframe → 父页面（postMessage）:
  { type: 'ui-replay-step_complete', data: { index, action, status, duration_ms, error, current_url } }
  { type: 'ui-replay-all_complete', data: { status: 'passed' | 'failed', total_steps, passed, failed, duration_ms } }
  { type: 'ui-replay-navigate', url: ... }
  { type: 'ui-replay-ready' }
```

**风险与规避**：

| 风险 | 影响 | 规避措施 |
|------|------|----------|
| iframe 内 JS 执行上下文受限 | 某些操作（如 window.parent）可能被 CSP 阻止 | iframe 设置 `sandbox="allow-same-origin allow-scripts allow-modals"` |
| SPA 页面元素未渲染就执行 | 选择器找不到元素导致失败 | 每步执行前加 `waitForElement` 轮询（默认 5 秒超时） |
| 密码字段暴露明文 | 安全风险 | 录制阶段已用 `{{password}}` 替代，回放时不显示原始值 |
| 页面导航后 iframe 内容丢失 | 导航到新页面后注入的 JS 被清除 | 导航后重新注入回放引擎 JS（通过 proxy 的 beforeRequest 钩子） |
| 跨域 iframe 加载失败 | 某些网站设置 X-Frame-Options 拒绝嵌入 | 已有 ui_proxy_service 改写 HTML 移除 frame-busting 脚本 |

#### Step 1.2：执行结果存储

**文件**：`services/ui_execution_store.py`（新增）

```python
class UiExecutionStore:
    """执行结果文件存储。

    目录结构：
      ui_testing_cases/
        exec_{job_id}/
          result.json      ← 完整报告
          screenshots/     ← 截图（Phase 2）
    """
    def create_job(self, case_id: str, mode: str, case_name: str) -> str:
        """创建执行任务记录，返回 job_id。"""

    def update_step(self, job_id: str, step_result: dict) -> None:
        """更新单个步骤结果（追加写入）。"""

    def finalize_job(self, job_id: str, status: str, summary: dict) -> str:
        """完成执行，写入最终报告。返回目录路径。"""

    def get_result(self, job_id: str) -> Optional[dict]:
        """读取完整执行报告。"""

    def list_results(self, case_id: Optional[str] = None, limit: int = 20) -> List[dict]:
        """列出执行历史（可选按 case_id 过滤）。"""
```

**设计要点**：
- 每次写入都追加到 `result.json`，不依赖内存
- 使用文件锁（`threading.Lock`）保证并发安全
- 服务重启后状态自动恢复（从文件读取）

**BrowserReplayEngine 轻量封装**：

```python
class BrowserReplayEngine:
    """浏览器内回放编排器。

    不直接操作浏览器，而是返回 replay URL 供前端加载。
    真正的执行发生在前端 JS 中。
    """
    def __init__(self, job_id: str, case_data: dict, options: dict):
        self.job_id = job_id
        self.case_data = case_data
        self.options = options

    def get_replay_url(self) -> str:
        """返回回放页面 URL。"""
        return f"/ui-testing/replay/{self.job_id}"
```

**文件**：`services/ui_headless_engine.py`（新增）

```python
from playwright.async_api import async_playwright

class HeadlessPlaywrightEngine:
    """使用 Playwright 在无头浏览器中执行 UI 测试。

    特性：
    - 多浏览器支持（chromium/firefox/webkit）
    - 自定义分辨率
    - 自动截图（每步执行前后）
    - 网络请求拦截和日志
    """

    async def execute(self, case_data: dict, job_id: str, options: dict) -> ExecutionReport:
        """执行完整用例。"""

    async def _execute_step(self, step: dict, index: int) -> StepResult:
        """执行单个步骤。"""

    async def _find_element(self, selector: dict | str):
        """查找元素（含 fallback 链）。"""

    async def _screenshot(self, name: str) -> str:
        """截图并返回文件路径。"""
```

**设计要点**：
- 使用 `asyncio` 异步执行，不阻塞 Flask 主线程
- 每个 job 独立的 browser context，避免状态污染
- 每步执行前后各截图一次，便于调试
- 网络请求日志记录（用于分析 API 调用）

#### Step 2.3：异步执行管理器

**文件**：`services/ui_execution_manager.py`（新增）

```python
class UiExecutionManager:
    """管理所有执行任务的生命周期。

    支持浏览器内回放和 Headless 两种模式。
    """

    def create_job(self, case_id: str, mode: str, options: dict) -> str:
        """创建执行任务。"""

    async def run_headless_job(self, job_id: str) -> None:
        """在后台启动 Headless 执行。"""

    def get_job_status(self, job_id: str) -> Optional[ExecutionRecord]:
        """查询执行状态。"""

    def cancel_job(self, job_id: str) -> bool:
        """取消执行。"""

    def list_jobs(self, case_id: Optional[str] = None, limit: int = 20) -> List[ExecutionRecord]:
        """列出执行历史。"""
```

**设计要点**：
- 使用 `asyncio.create_task()` 在后台运行 Headless 任务

### 11.3 日志级别策略

| 级别 | 使用场景 | 示例 |
|------|----------|------|
| **DEBUG** | 单步执行详情、postMessage 通信、选择器回退、截图、存储读写 | 开发调试时开启，生产环境关闭 |
| **INFO** | 任务生命周期事件（创建/开始/完成/取消）、批量进度、报告生成、设置变更、清理 | 生产环境默认级别 |

### 11.4 调试排查指引

#### 11.4.1 回放失败排查流程

```
1. 检查执行日志中的 ui.execution.step_failed 事件
   → 查看 error 字段定位失败原因
   → 查看 selector 字段确认选择器是否正确

2. 如果 error 包含 "Element not found"
   → 查看 ui.replay.selector_fallback 确认是否使用了 fallback
   → 查看 ui.replay.element_not_found 获取所有尝试过的选择器
   → 在编辑器中打开该步骤，检查选择器是否仍然有效

3. 如果 error 包含 "timeout"
   → 查看 ui.replay.iframe_load_timeout 确认 iframe 加载时间
   → 检查目标页面是否可正常访问
   → 考虑增大 UI_EXECUTION_DEFAULT_TIMEOUT_MS 配置

4. 如果整体执行中断
   → 查看 ui.execution.error 获取异常堆栈
   → 检查 ui.execution.cancelled 确认是否用户主动取消
   → 检查 ui.execution.timeout 确认是否超时自动终止
```

#### 11.4.2 Headless 执行排查流程

```
1. Playwright 启动失败
   → 查看 ui.headless.browser_start 是否有 ERROR 日志
   → 运行 `playwright install chromium` 确认浏览器已安装
   → 检查系统依赖（Linux 需要额外安装系统库）

2. 页面加载异常
   → 查看 ui.headless.network_request 检查 API 调用是否正常
   → 查看 ui.headless.resource_error 检查资源加载是否失败
   → 查看截图确认页面渲染状态

3. 执行超时
   → 查看 ui.execution.timeout 确认哪个步骤超时
   → 检查目标页面是否响应缓慢
   → 调整 UI_EXECUTION_DEFAULT_TIMEOUT_MS 或单步超时配置
```

#### 11.4.3 存储与恢复排查

```
1. 执行报告丢失
   → 查看 ui.store.write 确认写入是否成功
   → 检查 ui_testing_cases/exec_{job_id}/ 目录是否存在
   → 检查 result.json 文件内容是否完整

2. 服务重启后历史为空
   → 查看 ui.store.recovery 确认恢复过程是否正常
   → 检查 exec_* 目录是否存在且包含 result.json
   → 查看 ERROR 日志确认是否有文件读取异常

3. 磁盘空间不足
   → 查看 ui.cleanup.expired 确认清理是否正常运行
   → 调整 UI_EXECUTION_RETENTION_DAYS 配置
   → 手动清理旧的 exec_* 目录
```

#### 11.4.4 常用调试命令

```bash
# 查看最近的执行日志（按事件过滤）
grep "ui.execution" logs/app.log | tail -50

# 查看失败的执行详情
grep "ui.execution.step_failed\|ui.execution.error" logs/app.log

# 查看 Headless 浏览器日志
grep "ui.headless" logs/app.log

# 查看存储写入情况
grep "ui.store" logs/app.log

# 查看 webhook 发送情况
grep "ui.settings.webhook" logs/app.log

# 查看所有 WARNING 及以上日志
grep -E '"level": "(WARNING|ERROR)"' logs/app.log | grep "ui\."
```

### 11.5 日志采样策略

高频事件（单步执行详情、postMessage 通信、截图）在生产环境建议开启采样：

```python
# config.py 新增配置
UI_EXECUTION_LOG_SAMPLE_RATE = _env_float("UI_EXECUTION_LOG_SAMPLE_RATE", 0.1, lo=0.0, hi=1.0)
```

- 默认采样率 `0.1`（10%），避免大量步骤执行时日志爆炸
- `ui.execution.step_complete` 和 `ui.replay.postmessage` 受采样控制
- WARNING 及以上级别日志不受采样限制，始终输出
