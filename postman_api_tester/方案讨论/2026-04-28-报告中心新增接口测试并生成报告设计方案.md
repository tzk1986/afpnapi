# 2026-04-28 报告中心新增接口测试并生成报告设计方案

## 1. 背景与目标

当前报告中心已支持：
- 上传 Collection 并执行（全量/已选接口）
- 单接口重试与人工用例管理
- 导出最新 Collection（含人工用例与排除合成）

但缺少“在报告中心直接新建接口测试任务并生成完整报告”的原生入口。

本方案目标：
1. 在报告中心首页提供“新增接口测试”入口按钮，并跳转到独立页面完成后续操作。
2. 用户无需先准备完整 Collection 文件，也能创建一个或多个接口并执行。
3. 仍复用现有执行引擎，产出标准三件套报告（html/details/meta）。
4. 与现有功能兼容（上传执行、人工用例、导出、筛选、排除、历史对比）。
5. 明确风险与回归策略，降低上线风险。

## 2. 约束对齐

已对齐现有约束：
- 外置模板约束：仅在 templates/*.html 承载前端改动。
- 报告三件套约束：必须继续生成 html/details/meta。
- SSRF 防护：仅允许 http/https。
- 超时配置：统一读取 config.py 的 REQUEST_CONNECT_TIMEOUT/REQUEST_READ_TIMEOUT。
- token/base_url/output_dir 优先级：沿用现有 run_postman_tests 逻辑。
- 运行冒烟约束：提交前验证 /health、/、/report-view、/api/report-results(true/false)、/api/manual-cases CRUD、/api/export-collection。

## 3. 总体设计

### 3.1 设计原则

1. 不新增第二套执行引擎：通过“动态生成临时 Collection JSON + 调用现有 /api/run-postman”完成执行。
2. 不破坏既有上传执行入口：首页只增加按钮入口，新增能力在独立页面承载，互不影响。
3. 数据结构尽量复用现有 request_info/request body 模式语义（none/raw/urlencoded/formdata/graphql/binary）。

### 3.2 用户路径

新增首页流程：
1. 打开报告中心首页。
2. 点击“新增接口测试”按钮，进入独立 ad-hoc 页面。
3. 在独立页面的可视化编辑器中新增 1..N 条接口（名称、目录、方法、URL、Headers、Params、Body、期望状态码）。
4. 点击“执行并生成报告”。
5. 系统后台生成临时 Collection，走现有 run-postman 任务队列执行。
6. 执行完成后进入新报告的 report-view。

## 4. 方案细节

### 4.1 前端改造（index.html）

新增模块：
- 名称：新增接口测试独立页面（临时 Collection）
- 入口位置：首页上传执行区域下方的“新增接口测试”按钮

核心能力：
1. 接口列表编辑
   - 新增/复制/删除接口行
   - 拖拽排序（可选，若实现复杂可二期）
2. 请求参数编辑
   - method/url/headers/params/body_mode/body_data
   - body 支持 none/raw/urlencoded/formdata/graphql/binary
3. 任务参数
   - base_url（可选）
   - token（可选）
   - output_dir（可选）
   - report_name（可选）
   - results_per_page（复用配置边界）
4. 输入校验
   - URL 允许绝对 URL 或 {{baseUrl}}/path
   - 至少 1 条有效接口
   - method 非空，name/url 非空

前端提交：
- 调用新接口 POST /api/run-ad-hoc-tests
- payload 中包含接口定义数组与运行参数

### 4.2 后端新增接口（report_server.py）

新增路由：POST /api/run-ad-hoc-tests

职责：
1. 校验请求体结构与字段边界。
2. 将 ad-hoc 接口定义转为 Postman Collection v2.1 JSON。
3. 将 Collection 保存到 UPLOADS_DIR（文件名使用 job_id，防穿越）。
4. 内部复用 run_postman_job（或与 /api/run-postman 对齐的入队逻辑）创建任务。
5. 返回 job_id，前端沿用现有 /api/run-postman-status/{job_id} 轮询。

Collection 生成规则：
- 顶层：info/schema/item
- item 结构与现有导入解析兼容：
  - name
  - request.method
  - request.url.raw/path/query
  - request.header
  - request.body（按 mode 输出）
- 每条接口附带 expected_status（可通过 event test 注释或扩展字段保存；执行层默认 200，不破坏兼容）

重要：
- 不改动 run_postman_tests 的主流程。
- 不改动 /api/run-postman 现有行为。

### 4.3 代码复用点

优先复用现有函数语义：
- _set_request_url
- _set_request_headers
- _set_request_body
- _build_request_kwargs（用于前端输入验证与模式统一）
- run_postman_job + run_postman_tests

这样可避免：
- body 模式实现两套导致行为漂移
- 导出/导入语义不一致

### 4.4 配置项建议（config.py）

新增（建议）：
- ENABLE_ADHOC_RUN = true
- ADHOC_MAX_ITEMS = 300
- ADHOC_DEFAULT_COLLECTION_NAME = "临时接口测试"

说明：
- 边界值集中在 config.py，避免页面和路由硬编码。
- 默认开启，便于平滑上线。

## 5. 兼容性设计

### 5.1 与上传执行兼容

- 新增入口不替换旧入口。
- 仍走同一任务队列与状态接口。
- 报告产物目录与命名规则不变。

### 5.2 与人工用例兼容

- 报告生成后，report-view 的 manual_cases 逻辑不变。
- ad-hoc 生成报告仍可新增/编辑/排除人工用例。
- 导出时仍走“原始回填 + manual_cases - exclusions”统一逻辑。

### 5.3 与历史对比兼容

- 新报告照常写 meta.results。
- /api/compare 无需改造。

## 6. 风险清单与规避

1. 风险：ad-hoc body 模式与现有执行语义不一致
- 规避：后端统一复用 _set_request_body；前端 body_mode/body_data 按既有字段协议提交。

2. 风险：URL 仅 raw 时执行丢失
- 规避：生成 Collection 时同时写 raw，并保证 path/query 可解析；继续依赖执行层 raw 回退机制。

3. 风险：SSRF 与内网探测
- 规避：沿用现有 base_url/url 的 http/https + netloc 校验；拒绝 file://、gopher:// 等协议。

4. 风险：超大 ad-hoc 任务拖垮页面
- 规避：ADHOC_MAX_ITEMS 限制、前端提示与后端硬拦截。

5. 风险：新增接口破坏现有路由
- 规避：新增独立路由，不修改既有请求结构；保留现有 run-postman 全链路。

6. 风险：回归遗漏（路由签名/参数不一致）
- 规避：执行最小闭环排查法，覆盖 true/false 双态、CRUD 与导出。

## 8. 测试与验收建议

功能验收最小集：
1. 首页点击“新增接口测试”按钮可成功进入独立页面。
2. 独立页面新增 1 条 GET 接口 -> 生成报告成功。
3. ad-hoc 新增 1 条 POST(raw json) -> 请求体在详情可见。
4. ad-hoc 报告可进入 report-view，并可执行人工用例 CRUD。
5. ad-hoc 报告支持排除开关与导出 full/report_only。
6. 历史对比可选择 ad-hoc 报告。
7. 负例：非法 URL 协议被拦截；空接口列表被拦截。

回归必测（沿用约束 26）：
- /health
- /
- /report-view
- /api/report-results?include_excluded=true
- /api/report-results?include_excluded=false
- /api/manual-cases add/update/delete
- /api/export-collection

## 9. 非目标（本期不做）

1. 不在本期实现复杂断言编排（如脚本化断言 DSL）。
2. 不在本期实现跨任务模板库管理（可二期扩展）。
3. 不在本期替换现有 run-postman 上传入口。

## 10. 结论

本方案通过“ad-hoc 输入 -> 临时 Collection -> 复用现有执行链路”实现最小改动上线：
- 能满足“报告中心直接新增接口测试并生成报告”。
- 能与现有上传执行、人工用例、导出、对比能力兼容。
- 风险可控，且回归路径明确。
