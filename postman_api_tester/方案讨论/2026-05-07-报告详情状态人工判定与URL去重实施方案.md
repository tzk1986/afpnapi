# 2026-05-07 报告详情状态人工判定与URL去重实施方案

## 1. 背景

本次实施聚焦两个问题：

1. 报告详情中 GET 实际请求 URL 可能出现查询参数重复显示。
2. 报告详情缺少“人工标记成功/失败”能力，无法在不重发请求时快速修正状态。

## 2. 目标

1. 保证请求发送链路中 URL query 与 params 不重复叠加。
2. 在报告详情中新增人工判定按钮：标记成功、标记失败、恢复自动结果。
3. 保持现有排除、导出、分页、重试、人工用例能力兼容。

## 3. 方案设计

### 3.1 URL 与 params 去重策略

统一策略：

1. 先解析 URL 自带 query。
2. 再与 params 合并（params 覆盖同名 query）。
3. 最终请求一律使用“无 query 的 URL + 合并后的 params”。

落地点：

1. postman_api_tester/postman_api_tester.py
2. report_server.py（/re-request-api、/api/proxy-request）
3. templates/report_view.html（actual_request_url 兜底拼接）

### 3.2 人工判定状态能力

后端新增路由：

- POST /api/report-result-judgement

支持动作：

1. override：覆盖为 PASSED 或 FAILED。
2. restore：恢复自动判定结果。

数据策略：

1. 在 meta.results[i] 中保存 manual_judgement（当前人工判定状态）。
2. 保存 judgement_history（人工判定与恢复历史）。
3. 每次变更后重算 summary，并失效报告缓存。

前端改造：

1. 详情弹窗新增按钮：标记成功、标记失败、恢复自动结果。
2. 状态变更后即时刷新 summary、列表和详情元信息。
3. 手工用例详情维持原行为，不展示上述按钮。

## 4. 风险与规避

1. 风险：人工判定后与重试结果冲突。
   - 规避：重试回写时将 manual_judgement.active 置为 false，保证“真实重试结果”优先。

2. 风险：旧报告缺少 actual_request_url 字段。
   - 规避：前端保留兜底拼接，但改为去重合并策略。

3. 风险：影响现有筛选与排除链路。
   - 规避：仅新增 judgement_source 字段，不改原筛选参数与排除键逻辑。

## 5. 验证建议

1. 运行 Python 语法检查（postman_api_tester.py / report_server.py）。
2. 验证报告详情中实际 URL 不再重复 query。
3. 验证标记成功/失败/恢复后 summary 与列表同步更新。
4. 验证 /health、/、/report-view、/api/report-results、/api/export-collection 基础链路可用。
