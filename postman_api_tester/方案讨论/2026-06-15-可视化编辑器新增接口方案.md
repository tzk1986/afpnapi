**日期**：2026-06-15  
**版本基线**：v1.7.0 → v1.8.0（已实施）  
**状态**：✅ 已实施完成（2026-06-15）

---

## 一、需求背景

当前 Collection 可视化编辑器（`/collection-editor`）支持导入完整 Collection、编辑、删除、拖拽排序、导出，但缺少以下能力：

1. **手动新增接口**：不依赖外部文件，直接在编辑器中创建新接口
2. **接口复制**：快速复制现有接口作为新接口的起点，支持跨分组、批量复制
3. **cURL 导入**：从 cURL 命令粘贴生成接口（开发者常用场景），支持 `--cookie`/`--proxy`/`--cert` 等高级参数
4. **HAR 导入**：从浏览器开发者工具导出的 HAR 格式导入，可将响应 body 自动写入 x_extract
5. **Postman snippet 导入**：解析 Postman 生成的 Python/JS/Go 等代码片段，提取请求信息
6. **新建文件夹**：在侧边栏创建分组/文件夹，支持拖拽移动、深层嵌套
7. **接口跨文件夹拖拽**：接口可在不同文件夹间拖拽移动，支持多选后批量拖拽到目标文件夹或根目录

## 三、功能设计

### 3.1 统一入口设计

在侧边栏操作区统一增加入口按钮，点击后弹出对应功能的模态框：

```
┌──────────────────────────────────────────┐
│ 接口列表          12 个接口                │
│ ☑ 全选  [🗑 删除选中]                    │
│ [+ 新增 ▼]  [📁 新建文件夹]              │
│   ├─ ✚ 空白接口                           │
│   ├─ ⧉ 复制当前接口                       │
│   ├─ ⟲ 从 cURL 导入                      │
│   ├─ ⟲ 从 HAR 导入                       │
│   └─ ⟲ 从 Postman 代码片段导入            │
├──────────────────────────────────────────┤
│ 📁 用户模块  [⇅]                          │
│   POST 登录                              │
│   GET  用户列表                           │
│ 📁 订单模块  [⇅]                          │
│   📁 退款  [⇅]                            │
│     POST 创建退款                          │
│   POST 创建订单                           │
│ ...                                      │
└──────────────────────────────────────────┘
```

**设计理由**：
- 新增接口（5 种方式）用下拉菜单统一入口，避免按钮过多
- "新建文件夹"是组织层面的操作，与新增接口性质不同，独立按钮
- "复制当前接口"在无选中接口时置灰禁用
- 文件夹标题旁显示拖拽手柄 `[⇅]`，支持拖拽移动到其他文件夹或根目录

### 3.2 功能一：空白接口新增

#### 交互

1. 点击"+ 新增" → "✚ 空白接口"
2. 弹出模态框，填写核心字段：

| 字段 | 类型 | 默认值 | 必填 |
|------|------|--------|------|
| 接口名称 | text | "新接口" | ✅ |
| 请求方法 | select | GET | ✅ |
| URL | text | "" | ✅ |
| 描述 | textarea | "" | ❌ |

3. 确认后进入编辑状态，可补充 headers/params/body/x_extract

#### 目标分组逻辑

- 有选中接口 → 添加到同一分组
- 无选中接口 → 添加到根分组（`group_name: ""`）
- 空状态（未导入任何 Collection）→ 自动初始化空 Collection

#### 新增后行为

- 自动选中新接口，进入编辑状态
- 新接口插入到目标分组的末尾

### 3.3 功能二：接口复制（含跨分组/批量）

#### 交互 — 单个复制

1. 先选中一个接口
2. 点击"+ 新增" → "⧉ 复制当前接口"
3. 弹出确认框，显示：
   - 原接口名称
   - 新接口名称（默认追加 "_副本"）
   - **目标分组**（下拉框，默认当前分组，可选择其他分组）
4. 确认后立即复制，新接口出现在目标分组中

#### 交互 — 批量复制

1. 勾选多个接口（使用复选框）
2. 点击"+ 新增" → "⧉ 复制已选接口"（菜单项根据选中数量动态显示）
3. 弹出确认框，显示：
   - 已选接口数量
   - **目标分组**（下拉框，默认当前分组）
4. 确认后批量复制，每个接口名称自动追加 "_副本"

#### 交互 — 右键菜单

在接口项上右键，显示上下文菜单：
- "复制接口"（复制到同分组）
- "复制到..."（弹出分组选择器）

#### 复制规则

| 字段 | 复制行为 |
|------|---------|
| id | 重新生成（uuid） |
| name | 追加 "_副本"（如已存在则 "_副本2"、"_副本3"...） |
| method | 原样复制 |
| url | 原样复制 |
| headers | 深拷贝 |
| params | 深拷贝 |
| body_mode | 原样复制 |
| body_data | 深拷贝 |
| x_extract | 深拷贝 |
| description | 原样复制 |

#### 禁用条件

- 无选中接口时，"复制当前接口"和"复制已选接口"均置灰

### 3.4 功能三：cURL 导入（含高级参数）

#### 交互

1. 点击"+ 新增" → "⟲ 从 cURL 导入"
2. 弹出模态框，包含：
   - **cURL 命令输入区**（textarea，大文本框，placeholder 提示粘贴 cURL 命令）
   - **目标分组选择**（下拉框，列出所有分组；默认当前选中接口所在分组）
3. 点击"解析预览"，显示解析结果：
   - 自动提取的 URL、Method、Headers、Body
   - 可手动修正解析错误
4. 点击"确认导入"，新增接口到目标分组

#### cURL 解析规则

支持标准 cURL 命令格式，覆盖以下参数：

| cURL 参数 | 映射字段 | 说明 |
|-----------|---------|------|
| URL（位置参数） | url | 最后一个非选项参数 |
| `-X` / `--request` | method | HTTP 方法 |
| `-H` / `--header` | headers | 可多次出现，格式 `Key: Value` |
| `-d` / `--data` / `--data-raw` | body_data.content | 请求体，自动推断 body_mode=raw |
| `--data-urlencode` | body_data.content | URL 编码数据，推断 body_mode=urlencoded |
| `-F` / `--form` | body_data.data | multipart 表单，推断 body_mode=formdata |
| `-G` / `--get` + `-d` | params + method=GET | 数据作为查询参数 |
| `-u` / `--user` | headers[Authorization] | 转换为 Basic Auth header（`Basic base64(user:pass)`） |
| `-A` / `--user-agent` | headers[User-Agent] | User-Agent header |
| `-e` / `--referer` | headers[Referer] | Referer header |
| `--compressed` | headers[Accept-Encoding] | 添加 Accept-Encoding: gzip, deflate |
| `-b` / `--cookie` | headers[Cookie] | Cookie 字符串，直接写入 Cookie header |
| `--cookie-jar` | description | 记录到描述中（执行时不处理 jar 文件） |
| `-x` / `--proxy` | description | 代理地址记录到描述中（供用户参考） |
| `--cert` / `--cacert` | description | 证书路径记录到描述中（供用户参考） |
| `-k` / `--insecure` | description | 记录"跳过 SSL 验证"到描述中 |
| `-L` / `--location` | description | 记录"跟随重定向"到描述中 |
| `--max-time` / `-m` | description | 超时时间记录到描述中 |
| `--connect-timeout` | description | 连接超时记录到描述中 |
| `-I` / `--head` | method=HEAD | 等同于 `-X HEAD` |
| `-T` / `--upload-file` | body_data.content + method=PUT | 文件上传，推断 PUT 方法 |

#### 解析实现

纯前端 JavaScript 解析，不依赖后端。解析步骤：

1. 去除换行续行符（`\` 结尾的行）
2. 提取命令名 `curl`（兼容 `curl.exe`、`/usr/bin/curl` 等路径）
3. 按空格分词（尊重引号内的空格）
4. 逐个解析选项，映射到请求字段
5. 自动从 Content-Type header 推断 body_mode

#### 接口名称生成

- 优先使用 URL 的最后一个路径段（如 `/api/users/list` → `users/list`）
- 如果无法提取，使用 "cURL 导入接口"

### 3.5 功能四：HAR 导入（含响应→x_extract）

#### 交互

1. 点击"+ 新增" → "⟲ 从 HAR 导入"
2. 弹出模态框，包含：
   - **HAR 输入区**（textarea，粘贴 HAR JSON 或上传 `.har` 文件）
   - **文件上传按钮**（可选，支持 `.har` 文件）
   - **接口预览列表**（解析后显示所有可导入的接口）
   - **全选/反选** 复选框
   - **目标分组选择**
3. 勾选要导入的接口
4. 点击"确认导入"，批量新增到目标分组

#### HAR 格式说明

HAR (HTTP Archive) 是浏览器开发者工具导出的标准格式，结构：

```json
{
  "log": {
    "entries": [
      {
        "request": {
          "method": "POST",
          "url": "http://example.com/api/data",
          "headers": [{"name": "Content-Type", "value": "application/json"}],
          "queryString": [{"name": "page", "value": "1"}],
          "postData": {
            "mimeType": "application/json",
            "text": "{\"key\": \"value\"}"
          }
        },
        "response": { ... }
      }
    ]
  }
}
```

#### HAR 解析规则

| HAR 字段 | 映射字段 | 说明 |
|---------|---------|------|
| `request.method` | method | HTTP 方法 |
| `request.url` | url | 完整 URL |
| `request.headers` | headers | 过滤掉浏览器内部 header（Host、Connection 等） |
| `request.queryString` | params | 查询参数 |
| `request.postData.text` | body_data.content | 请求体文本 |
| `request.postData.mimeType` | body_mode | 根据 MIME 类型推断 |
| `response.content.text` | x_extract（可选） | 见下方"响应提取"规则 |

#### 过滤规则

以下 header 默认不导入（浏览器内部 header）：

- `Host`、`Connection`、`Content-Length`
- `Accept-Encoding`、`Accept-Language`（可选保留）
- `:authority`、`:method`、`:path`、`:scheme`（HTTP/2 伪 header）

#### 响应提取（response → x_extract）

HAR 文件通常包含响应数据。导入时提供选项"是否从响应中提取变量"：

- 若勾选，解析 `response.content.text`（响应 body）
- 对响应 JSON 的**顶层字段**自动生成 x_extract 规则
- 生成规则：变量名 = 字段名，JSONPath = `$.字段名`
- 嵌套对象/数组：仅生成第一层路径（如 `$.data`，不展开 `$.data.list[0].id`）

示例：

```json
// HAR 响应 body
{"errCode": 0, "data": {"token": "abc123", "userId": 1001}}

// 生成的 x_extract
{
  "errCode": "$.errCode",
  "data": "$.data"
}
```

用户在导入预览界面可手动调整：取消不需要的提取规则、修改变量名、添加深层路径。

#### 接口名称生成

- 使用 URL 的最后一个路径段 + 方法前缀（如 `POST /api/users` → `POST users`）
- 如果 URL 无路径，使用域名

### 3.6 功能五：Postman 代码片段导入

#### 交互

1. 点击"+ 新增" → "⟲ 从 Postman 代码片段导入"
2. 弹出模态框，包含：
   - **代码片段输入区**（textarea，粘贴代码）
   - **语言选择**（下拉框：Python/JavaScript/Go/cURL/Java，默认自动检测）
   - **目标分组选择**
3. 点击"解析预览"，显示提取结果
4. 确认后导入

#### 支持的语言与提取规则

| 语言 | 提取方式 |
|------|---------|
| **Python (requests)** | 正则匹配 `requests.get/post/put/delete(url, ...)` → 提取 url、headers、json/data、params |
| **JavaScript (fetch)** | 正则匹配 `fetch(url, {method, headers, body})` → 提取 url、method、headers、body |
| **JavaScript (axios)** | 正则匹配 `axios.get/post(url, ...)` → 提取 url、headers、params/data |
| **Go (net/http)** | 正则匹配 `http.NewRequest("METHOD", url, body)` + `req.Header.Set()` → 提取 method、url、headers、body |
| **Java (OkHttp/HttpClient)** | 正则匹配 `Request.Builder().url(...)` → 提取 url、method、headers、body |
| **cURL** | 复用 cURL 解析器（与功能三相同） |

#### 自动检测逻辑

1. 检查代码是否包含 `curl` 关键字 → cURL
2. 检查是否包含 `requests.` → Python
3. 检查是否包含 `fetch(` → JavaScript (fetch)
4. 检查是否包含 `axios.` → JavaScript (axios)
5. 检查是否包含 `http.NewRequest` → Go
6. 检查是否包含 `Request.Builder` → Java
7. 无法识别时提示用户手动选择语言

#### 限制

- 仅提取请求信息，不处理响应解析/断言
- 不支持动态变量（如 Python 的 f-string 变量），保留原始 `{{...}}` 或 `{variable}` 占位符
- 复杂的认证逻辑（如 OAuth 签名）仅保留最终生成的 header

### 3.7 功能六：新建文件夹（分组）

#### 交互

1. 点击侧边栏"📁 新建文件夹"按钮
2. 弹出简单模态框：
   - **文件夹名称**（text 输入框，默认"新建文件夹"）
   - **父级分组**（select 下拉框，列出所有现有分组；默认"根目录"）
3. 点击"确认创建"，新文件夹出现在侧边栏

#### 数据结构

文件夹对应 `state.collection.groups` 中的一个 group 对象：

```javascript
{
  group_name: "用户模块",
  requests: [],      // 新建时为空
  subgroups: []      // 新建时为空（暂不支持嵌套子文件夹）
}
```

#### 目标位置逻辑

| 父级分组选择 | 行为 |
|-------------|------|
| 根目录 | 添加到 `state.collection.groups` 顶层 |
| 已有文件夹 | 添加到该文件夹的 `subgroups` 中 |

#### 重命名与删除

- **重命名**：双击侧边栏文件夹名称，进入编辑状态，回车确认
- **删除**：右键菜单或文件夹标题旁的删除按钮（仅删除空文件夹；非空文件夹需先移出或清空接口）

#### 嵌套层级

- **不限制嵌套深度**：根 → A → B → C → ... 均可
- 侧边栏按缩进展示层级关系
- 导出时按嵌套结构生成 Postman Collection 的 folder 树

#### 文件夹拖拽移动

- 文件夹标题旁显示拖拽手柄图标
- 拖拽文件夹到：
  - **根目录**：移动到顶层
  - **其他文件夹**：成为其子文件夹
- 防护：不允许拖入自己的子文件夹（循环引用检测）

### 3.8 功能七：接口跨文件夹拖拽（含多选）

#### 交互

1. **单接口跨文件夹拖拽**：
   - 拖拽接口项到目标文件夹标题上
   - 释放后接口从原分组移动到目标文件夹
   - 拖拽到根目录区域则移回根分组

2. **多接口批量拖拽**：
   - 勾选多个接口（复选框）
   - 拖拽任意一个已选接口
   - 所有已选接口一起移动到目标位置
   - 拖拽过程中显示"移动 N 个接口"提示

#### 拖拽视觉反馈

| 拖拽目标 | 视觉反馈 |
|---------|---------|
| 文件夹标题 | 文件夹高亮 + "放入"提示 |
| 根目录空白区域 | 根目录区域高亮 |
| 接口项之间 | 显示插入线（同组内排序） |
| 非法目标（如自身子文件夹） | 红色禁止图标 |

#### 实现要点

- 文件夹标题增加 `ondragover`/`ondrop` 事件监听
- 多选拖拽：从 `state.selectedIds` 获取所有选中接口
- 移动逻辑：从原分组 `requests` 中移除 → 插入目标分组 `requests`
- 排序场景（同组内拖拽）：保持现有的排序逻辑不变

### 3.9 目标分组逻辑（通用）

所有新增方式共用以下目标分组逻辑：

```
1. 若模态框中有"目标分组"选择器，以用户选择为准
2. 否则：
   a. 有选中接口 → 添加到同一分组
   b. 无选中接口 → 添加到根分组
   c. 空 Collection → 创建根分组并添加
```

## 四、数据流

### 4.1 空白接口 / 复制接口

```
用户操作 → 前端构造请求对象 → 插入 state.collection.groups → 同步 state.allRequests → renderTree() → selectRequest(新 id)
```

### 4.2 cURL 导入

```
粘贴 cURL → 前端解析 → 构造请求对象 → 插入 → 同步 → 渲染
```

### 4.3 HAR 导入

```
粘贴/上传 HAR → 前端解析 log.entries → 过滤 → 批量构造请求对象 → 插入 → 同步 → 渲染
```

### 4.4 Postman 代码片段导入

```
粘贴代码 → 自动检测语言 → 正则提取请求信息 → 构造请求对象 → 插入 → 同步 → 渲染
```

### 4.5 新建文件夹

```
输入文件夹名称 + 选择父级 → 构造 group 对象 → 插入 groups 或 subgroups → renderTree()
```

### 4.6 接口跨文件夹拖拽

```
拖拽接口（单选或多选）到文件夹标题 → 从原分组移除 → 插入目标分组 → renderTree()
```

### 4.7 导出兼容性

所有新增方式最终都写入 `state.collection.groups`，导出时通过 `build_collection_json()` 自动包含，无需额外处理。

## 五、后端变更

### 5.1 是否需要后端变更？

**不需要**。所有解析和构造逻辑均在前端 JavaScript 中完成：

- cURL 解析：前端 JS 字符串解析
- HAR 解析：前端 JS JSON 解析
- 数据写入：直接操作 `state.collection`
- 导出：复用现有 `/api/collection-editor/save` 接口

### 5.2 预留扩展

如果未来 cURL/HAR 解析逻辑过于复杂（如支持更多 cURL 参数、HAR 响应提取），可将解析逻辑下沉到后端：

- 新增 `POST /api/collection-editor/import-curl`
- 新增 `POST /api/collection-editor/import-har`

当前方案优先前端实现，保持简单。
