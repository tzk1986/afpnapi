## 二、代码质量重构（优先级：中）

### 2.1 runtime_utils 与 execution_helpers 职责统一
**现状**：
- `runtime_utils.py` 有 checkpoint 相关函数
- `execution_helpers.py` 也有 checkpoint 逻辑
- 存在重复的 checkpoint_key 计算

**改进**：
- 将 checkpoint 逻辑统一收口到 `core/checkpoint_manager.py`
- `runtime_utils.py` 仅保留纯工具函数（无状态）
- `execution_helpers.py` 调用 `checkpoint_manager` 而非直接操作

**验证**：
- 现有 checkpoint 测试全通过
- 执行链路回归测试

### 2.2 services 层错误处理模式提取
**现状**：
```python
# 多个 service 重复的模式
try:
    result = do_something()
    logger.info("success", extra={...})
    return result
except ValueError as e:
    logger.warning("validation error: %s", e)
    raise
except Exception as e:
    logger.exception("unexpected error")
    raise
```

**改进**：
- 在 `base_service.py` 提取 `@service_operation` 装饰器
- 自动处理日志 + 异常分类
- 减少 30+ 处重复代码

### 2.3 handlers 参数校验统一
**现状**：
- 部分 handler 手动校验 `if not report_name: return _json_error(...)`
- 校验逻辑分散，错误码不统一

**改进**：
- 提取 `validate_required_fields(payload, fields, error_code_prefix)` 工具函数
- 统一返回格式和错误码生成
- 减少 50+ 处重复校验代码

## 三、安全加固（优先级：高）

### 3.1 输入长度限制审计
**检查清单**：
- [ ] `/api/run-postman` — cases 数组长度限制
- [ ] `/api/run-ad-hoc-tests` — cases 数组长度限制（已有 ADHOC_MAX_ITEMS）
- [ ] `/api/manual-cases` — case name/description 长度限制
- [ ] `/api/export-collection` — report_name 长度限制
- [ ] `/api/report-analytics` — report_name 长度限制
- [ ] 所有接收 JSON body 的端点 — body 大小限制（Flask MAX_CONTENT_LENGTH）

**改进**：
- 添加统一的输入长度校验中间件
- 超限返回 413 Payload Too Large 或 400 Bad Request
- 记录审计日志

### 3.2 并发竞态条件检查
**检查点**：
- `report_lock_service.py` — 同一报告的并发写回
- `report_job_store.py` — 任务状态的并发更新
- `global_variables_service.py` — 变量的并发读写

**改进**：
- 添加并发测试（多线程同时操作同一资源）
- 必要时引入更细粒度的锁（文件锁 vs 内存锁）
- 添加超时机制防止死锁

### 3.3 敏感信息日志脱敏审计
**检查方法**：
```bash
# 搜索可能的敏感信息泄露
grep -r "token\|password\|secret\|key" postman_api_tester/ | grep "logger\."
```

**改进**：
- 敏感字段自动脱敏（`***` 或 `[REDACTED]`）
- 添加日志脱敏中间件
- 更新开发文档约束

## 四、性能优化（优先级：低）

### 4.1 报告列表缓存失效策略
**现状**：
- 30s TTL 固定缓存
- 新报告生成时手动调用 `_invalidate_reports_cache()`

**改进**：
- 改为事件驱动：监听报告目录变化自动失效
- 或使用版本号：每次写入递增版本号，缓存键含版本号
- 预期提升：减少 50% 的缓存穿透

### 4.2 大报告分页加载优化
**现状**：
- 1000+ 接口的报告首页加载 2-3s
- 原因：一次性读取全部 meta + results

**改进**：
- 首页仅加载 summary + 前 20 条 results
- 滚动加载后续页（前端 IntersectionObserver）
- 后端提供 `/api/report-results-page?name=X&page=N` 接口
- 预期提升：首页加载 < 500ms
