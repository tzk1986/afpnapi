# P0 功能实施方案：数据驱动测试 + 请求串联与变量提取

**日期：** 2026-06-12
**状态：** 方案讨论（待审批）
**关联文档：** `2026-06-12-同类项目调研对比与功能演进方案.md`

## Context

基于 2026-06-12 同类项目调研（见 `方案讨论/2026-06-12-同类项目调研对比与功能演进方案.md`），数据驱动测试与请求串联/变量提取被定义为 P0 级功能——这两项是 Newman/HttpRunner 的基础能力，直接影响日常测试效率。当前项目缺少这两项能力，导致：

1. **数据驱动缺失**：同一接口需要用不同参数验证（如不同用户角色、边界值），只能手动复制多份 Collection 或在 Postman 中逐一修改，无法自动化批量验证。
2. **请求串联缺失**：典型工作流（如 登录→获取token→调用业务接口→查询结果）无法自动化，必须手动将上一步响应中的 token/id 复制到下一步请求中。

**目标**：在现有 Postman Collection 执行架构上，以最小侵入方式新增这两项能力，保持向后兼容（不配置时行为与 v1.3.4 完全一致）。

---

## 一、数据驱动测试

### 1.1 功能概述与核心场景

**核心场景**：单接口（或少量接口）大数据量压测。用 CSV/JSON 数据文件驱动同一接口反复请求，观察接口在大数据量下的响应时间、成功率、错误分布等表现。

**典型用例**：

用例 1 — 单接口静态数据压测：
```
数据文件: 1000 条 orderId
接口: GET /api/orders/{{orderId}}
→ 1000 次查询，观察响应时间趋势
```

用例 2 — 数据驱动 + 请求串联（创建→查询完整链路）：
```
数据文件（1000 行）:
  productName,price,quantity
  商品A,99.9,5
  商品B,199.0,1
  ...

接口 1：创建订单 POST /api/orders
  body: {"name":"{{productName}}","price":{{price}},"qty":{{quantity}}}
  x_extract: {"orderId": "$.data.orderId"}

接口 2：查询订单 GET /api/orders/{{orderId}}
  （orderId 来自接口 1 的响应提取，非静态数据文件）

→ 1000 行 × 2 接口 = 2000 次请求
→ 每次查询的 orderId 都是上一步创建的真实返回值
→ 可观察：创建接口的写入性能 + 查询接口的读取性能
```

用例 3 — 认证 + 业务操作串联：
```
数据文件（500 行）:
  username,password
  user1,pass1
  user2,pass2
  ...

接口 1：登录 POST /api/login
  body: {"username":"{{username}}","password":"{{password}}"}
  x_extract: {"token": "$.data.token", "userId": "$.data.userId"}

接口 2：获取用户信息 GET /api/users/{{userId}}
  headers: {"Authorization": "Bearer {{token}}"}

接口 3：更新昵称 PUT /api/users/{{userId}}/profile
  headers: {"Authorization": "Bearer {{token}}"}
  body: {"nickname": "test_{{username}}"}

→ 500 行 × 3 接口 = 1500 次请求
→ token 和 userId 在每行数据内跨接口传递
→ 每行数据的变量上下文独立（下一行数据重新开始）
```

**非目标场景**（v1 不涉及）：
- 不同接口使用完全不同的数据文件（建议拆分为不同 Collection）
- 复杂的条件分支（如"如果创建成功则查询，否则跳过"）

### 1.2 数据文件格式

**CSV 格式**（首行为变量名）：
```csv
orderId,expected_status
1001,200
1002,200
9999,404
abc,400
```

**JSON 格式**（数组对象）：
```json
[
  {"orderId": "1001", "expected_status": 200},
  {"orderId": "1002", "expected_status": 200},
  {"orderId": "9999", "expected_status": 404}
]
```

### 1.3 配置方式

**CLI 参数**：`--data-file <path>` （可选）
**Web UI**：首页上传区增加"数据文件"上传入口（可选，与 Collection 文件并列）
**Collection 内声明**（高级用法）：在 Collection `variable` 块中声明数据文件路径
```json
{
  "variable": [
    {"key": "baseUrl", "value": "https://api.example.com"},
    {"key": "dataFile", "value": "test_data/orders.csv"}
  ]
}
```

### 1.4 执行模型（按需替换）

```
原始 apis (N 个接口)

判定：每个 api 的 URL/headers/body/params 中是否包含数据文件中的变量名
  - 不包含 → 保持 1 份（正常执行一次）
  - 包含 → 展开为 M 份（每行数据一份）

典型场景（单接口压测）：
  Collection 只有 1 个接口 "查询订单"
  URL: /api/orders/{{orderId}}
  数据文件: 1000 行

  → 该接口展开为 1000 份，每份使用不同的 orderId
  → 总执行次数 = 1000
  → 报告中按 data_index 顺序展示，可观察响应时间趋势

混合场景（多接口 + 部分使用数据）：
  数据文件有 orderId 列，100 行
  - 接口 A（URL 含 {{orderId}}）→ 展开为 100 份
  - 接口 B（无变量引用）→ 保持 1 份
  
  总执行次数 = 100 + 1 = 101

展开规则：
- 仅包含数据文件中实际存在的变量名的接口才展开
- {{baseUrl}}/{{base_url}} 不算数据驱动变量（由 base_url 逻辑处理）
- 结果记录增加 data_index 字段标识第几组数据（未展开的接口 data_index=0）
- 报告中按原始接口分组展示
```

### 1.7 执行链路集成点

展开步骤插入位置（`execution_helpers.py`）：

```
_prepare_execution_apis()
    → _parse_collection_apis()
    → _filter_selected_apis()        # 先筛选
    → _apply_base_url_override()     # 再应用 base_url
    → _expand_apis_with_data()       # 【新增】数据驱动展开
    → return apis (可能从 N 变为展开后数量)
```

`_expand_apis_with_data()` 实现要点：
- 如果 `data_file` 为空或 `ENABLE_DATA_DRIVEN=False`，直接返回原始 apis（零开销）
- 遍历每个 api，检测 URL/headers/body/params 中是否引用了数据文件中的变量名
- **不包含数据变量的 api**：保持 1 份，`data_index=0`，`data_row={}`
- **包含数据变量的 api**：复制 M 份，每份 `data_index=0..M-1`，`data_row=该行数据`
- 对每个展开副本调用 `substitute_in_api_config()` 替换变量
- 展开后重新计算 `full_url`（因为 url 可能已被替换）
- 日志记录展开前后数量（如"原始 10 个接口，数据驱动展开后 25 个"）

变量检测方法：
```python
def _api_uses_data_variables(api: ApiConfig, data_columns: set[str]) -> bool:
    """检测 api 的 URL/headers/body/params 是否引用了数据文件中的变量名。
    
    扫描 api 中所有字符串值，查找 {{var}} 模式，
    如果任一 var 在 data_columns 中则返回 True。
    排除 baseUrl/base_url（这两个由 base_url 逻辑处理）。
    """
```

### 1.9 与现有功能的兼容性

| 现有功能 | 兼容性分析 | 处理方式 |
|---------|-----------|---------|
| 选择性执行 (selected_item_paths) | 筛选在展开前执行，筛选的是原始接口；展开后只有被选中的接口（含其数据变体）会执行 | 筛选 → 展开，符合直觉 |
| 断点恢复 (checkpoint) | 展开接口的 item_path 键需要扩展为 `(item_path, data_index)` 复合键；未展开接口保持原键 | checkpoint 键格式升级（仅对 data_index > 0 的记录） |
| 进度回调 (progress_callback) | total 变为展开后的实际数量（非 N×M，因为无变量接口不展开） | 自然适配 |
| 报告生成 | TestResultRecord 增加 `data_index` 字段（默认 0），HTML 模板按原始接口分组 + 性能摘要 | 向后兼容（字段可选，data_index=0 时不显示数据列标识） |
| 结果判定 (judgment) | 判定逻辑与数据驱动无关，无需修改 | 无需改动 |
| SSRF 防护 | 替换后的 URL 仍需通过协议白名单校验 | 替换后校验 |
| 变量提取（请求串联） | 数据驱动展开的接口也可以使用 x_extract 提取变量；提取结果写入共享上下文 | 两个功能正交，自然组合 |

## 二、请求串联与变量提取

### 2.1 功能概述

支持在请求间动态传递**任意业务变量**。前一个接口的响应中任意字段（orderId、userId、token、sessionId 等）可被提取为变量，后续接口的 URL/headers/body/params 中通过 `{{变量名}}` 引用。

**不限于 token 传递**——任何从响应中提取的值都可以跨接口使用：
- 创建资源后提取 ID → 后续查询/更新/删除该资源
- 登录后提取 token + userId → 后续接口同时使用两者
- 查询配置后提取参数值 → 后续接口使用该参数

### 2.2 变量来源（三层优先级）

```
接口级 extract 配置 > Collection 级 variables 声明 > 运行时提取（前序接口响应）
```

1. **Collection 级 variables**：Postman Collection JSON 的 `variable` 块（已有解析，仅用于 baseUrl）
2. **运行时提取**：前序接口响应中通过 `x_extract` 配置提取的变量
3. **内置变量**：`{{baseUrl}}`、`{{base_url}}`（已有逻辑处理，不走通用替换）

### 2.3 Collection 扩展字段

在 `request` 对象中新增 `x_extract` 字段：

**示例 1 — 登录场景（提取 token + userId）**：
```json
{
  "request": {
    "method": "POST",
    "url": "/api/login",
    "body": {"mode": "raw", "raw": "{\"username\":\"admin\",\"password\":\"123\"}"},
    "x_extract": {
      "token": "$.data.token",
      "userId": "$.data.userId",
      "sessionId": "$header.session-id"
    }
  }
}
```

**示例 2 — 创建资源场景（提取业务 ID）**：
```json
{
  "request": {
    "method": "POST",
    "url": "/api/orders",
    "body": {"mode": "raw", "raw": "{\"product\":\"{{productName}}\",\"qty\":{{quantity}}}"},
    "x_extract": {
      "orderId": "$.data.orderId",
      "orderNo": "$.data.orderNo",
      "createdAt": "$.data.createdAt"
    }
  }
}
```

后续接口可通过 `{{token}}`、`{{orderId}}`、`{{orderNo}}` 等引用这些提取值。

`x_extract` 规则：
- 键：变量名（将写入变量上下文）
- 值：JSONPath 表达式（以 `$.` 开头），或特殊前缀 `$header.名称` 提取响应头
- 提取失败时变量不写入（不覆盖已有值），日志记录警告

### 2.4 变量上下文生命周期

```
变量上下文 (VariableContext) — 贯穿整个 _execute_api_suite() 调用

初始化：
  - 从 Collection.variable 解析预置变量
  - 从 Web UI 传入的 initial_variables（可选）

执行过程中：
  - 每个接口执行前：substitute_in_api_config(api, context.variables)
  - 每个接口执行后：根据 x_extract 提取响应值，更新 context.variables
  - context 是 mutable dict，在同一 suite 内共享

销毁：
  - _execute_api_suite() 结束后 context 自然释放
  - 不持久化（不写入 checkpoint）

数据驱动 + 变量提取的组合行为：
  展开后的 apis 顺序为：
    [API_A_row0, API_B_row0, API_A_row1, API_B_row1, ...]
  
  执行顺序保证同一行数据的接口相邻执行：
    API_A_row0: 用 row0 数据创建订单 → 提取 orderId_0
    API_B_row0: 用 orderId_0 查询 → 使用刚提取的值 ✓
    API_A_row1: 用 row1 数据创建订单 → 提取 orderId_1（覆盖 orderId_0）
    API_B_row1: 用 orderId_1 查询 ✓
  
  无需手动重置上下文，展开顺序天然保证正确性。
```

### 2.6 核心模块设计

#### `core/variable_context.py`

```python
class VariableContext:
    """请求串联变量上下文，在 _execute_api_suite 生命周期内共享。"""
    
    def __init__(self, initial_variables: Optional[Dict[str, str]] = None) -> None:
        self._variables: Dict[str, str] = dict(initial_variables or {})
    
    @property
    def variables(self) -> Dict[str, str]:
        return dict(self._variables)  # 返回副本，防止外部直接修改
    
    def get(self, name: str, default: str = "") -> str:
        return self._variables.get(name, default)
    
    def set(self, name: str, value: str) -> None:
        self._variables[name] = value
    
    def update_from_extract(self, extracts: Dict[str, str], response_data: object, response_headers: Dict[str, str]) -> Dict[str, str]:
        """根据 x_extract 配置从响应中提取变量并更新上下文。
        
        返回实际成功提取的变量字典（用于报告记录）。
        """
```

```python
def extract_by_jsonpath(data: object, expression: str) -> Optional[str]:
```


`executor.py` 内部修改：

```python
def execute_test(self) -> TestResultRecord:
    # 【新增】请求前变量替换（如果 variable_context 存在）
    if self.variable_context:
        self.api_config = substitute_in_api_config(self.api_config, self.variable_context.variables)
    
    # ... 现有执行逻辑不变 ...
    
    # 【新增】请求后变量提取
    extracted_vars = {}
    if self.variable_context:
        raw_extract = self.api_config.get('x_extract')
        if isinstance(raw_extract, dict) and raw_extract:
            extracted_vars = self.variable_context.update_from_extract(
                raw_extract, self.response_data, dict(response.headers)
            )
    
    # 将提取的变量记录到 result 中（便于报告展示）
    result['extracted_variables'] = extracted_vars
```

### 2.8 与现有功能的兼容性

| 现有功能 | 兼容性分析 | 处理方式 |
|---------|-----------|---------|
| 选择性执行 | 变量提取依赖执行顺序；选择性执行可能跳过"生产变量"的接口 | 文档警告 + 预置变量兜底 |
| 断点恢复 | 变量上下文不持久化到 checkpoint；恢复后变量丢失 | 文档说明；后续可扩展 checkpoint 记录变量快照 |
| Ad-hoc 执行 | 需要支持在 ad-hoc 页面配置 x_extract | normalize_adhoc_case 支持新字段 |
| 重试失败 | 重试时变量上下文应保持；但由于重新执行，提取值可能被覆盖 | 自然行为，无需特殊处理 |
| `{{baseUrl}}` 处理 | 已有专门逻辑（parser.py + execution_helpers.py），不走通用替换 | substitute 时跳过 baseUrl/base_url |
| SSRF 防护 | 替换后的 URL 仍需校验协议 | 替换后、请求前校验 |

## 三、两个功能的交叉关系

### 3.1 共享组件

两个功能共享 `utils/variable_substitution.py`（`{{变量}}` 替换引擎）。区别在于变量来源：

| | 数据驱动 | 请求串联 |
|---|---------|---------|
| 变量来源 | 外部数据文件（CSV/JSON） | 前序接口响应提取 |
| 变量生命周期 | 单行数据（每个 api 变体独立） | 整个 suite（跨接口共享） |
| 替换时机 | 展开时（执行前批量替换） | 执行时（每个接口执行前实时替换） |
| 展开接口数量 | 是（仅含变量的接口展开） | 否（保持 N 个） |

### 3.2 可同时使用

两个功能可同时启用。优先级：数据行变量 > 变量上下文 > Collection 预置变量。

实现方式：`substitute_in_api_config()` 接受合并后的变量字典，调用方负责合并优先级。

## 四、潜在问题与预防策略

### 4.1 数据驱动相关

| 问题 | 风险等级 | 预防策略 |
|------|---------|---------|
| 大数据文件导致内存溢出/执行过久 | 中 | `DATA_FILE_MAX_ROWS` 限制（默认 10000），超限报错拒绝执行；大数据量时进度回调尤为重要 |
| CSV 编码问题（GBK vs UTF-8） | 中 | 优先 UTF-8，失败后尝试 UTF-8-sig（带 BOM），再失败尝试 GBK |
| 变量名冲突（数据文件变量名与 baseUrl 冲突） | 低 | 替换时 baseUrl/base_url 走专用逻辑，不受数据文件覆盖 |
| 展开后接口数量过大导致报告臃肿 | 中 | 报告中按原始接口分组展示，配置 `DATA_DRIVEN_GROUP_IN_REPORT`（默认 True） |
| 数据文件路径安全性（路径穿越） | 高 | 数据文件仅允许从 `uploaded_collections/` 或 `test_data/` 目录读取；CLI 模式下允许绝对路径但校验文件存在 |

### 4.2 请求串联相关

| 问题 | 风险等级 | 预防策略 |
|------|---------|---------|
| 循环依赖（A 提取 x，B 使用 x 又提取 y，A 使用 y） | 低 | 变量上下文为"只追加"模式，已提取的变量不会被删除；但可被新的提取覆盖 |
| JSONPath 表达式注入（恶意 Collection） | 中 | 仅支持简化的 JSONPath（字段访问 + 数组索引），不支持表达式/过滤器；使用自研轻量解析器而非 eval |
| 变量值包含特殊字符导致 URL 编码问题 | 中 | 替换后由 `normalize_url_and_params()` 自然处理；不额外编码 |
| 提取失败时后续接口使用空变量 | 中 | 提取失败记录日志警告 + 在报告中标记；后续接口使用空字符串替换，请求大概率失败但不会崩溃 |
| 选择性执行跳过"生产变量"的接口 | 高 | 文档明确警告；提供 `initial_variables` 参数让用户预填关键变量；报告中展示未解析变量列表 |
| 断点恢复后变量丢失 | 中 | v1 不持久化变量到 checkpoint（复杂度过高），文档说明；未来可扩展 checkpoint 记录最近一次提取的变量快照 |

### 4.3 通用风险

| 问题 | 风险等级 | 预防策略 |
|------|---------|---------|
| 替换逻辑影响现有 `{{baseUrl}}` 处理 | 高 | `substitute_variables()` 中排除 `baseUrl` 和 `base_url`；单元测试覆盖 |
| 新字段污染现有 ApiConfig 类型 | 低 | ApiConfig 为 TypedDict(total=False)，新字段天然可选 |
| 功能开关未关闭时性能退化 | 低 | 开关关闭时零开销（不创建 VariableContext，不加载数据文件） |
| mypy CI 报错 | 中 | 新增文件立即加入 `pyproject.toml` files 列表；遵循 strict 类型标注 |

## 五、约束变更分析（交叉比对开发阅读文档）

### 5.1 需要新增的约束

#### 约束 31：数据驱动测试约束

1. **功能开关**：`ENABLE_DATA_DRIVEN`（默认 `False`），支持环境变量覆盖。开关关闭时不加载数据文件、不展开接口。
2. **核心场景**：单接口（或少量接口）大数据量压测。用大数据文件驱动同一接口反复请求，观察响应时间、成功率等性能指标。
3. **数据文件格式**：仅支持 CSV（UTF-8/UTF-8-sig/GBK）和 JSON（数组对象）。不支持 Excel（v1 范围）。
4. **数据文件路径安全**：Web 上传的数据文件存入 `uploaded_collections/`；CLI 模式允许任意路径但必须文件存在。禁止路径穿越（`..` 检查）。
5. **行数限制**：`DATA_FILE_MAX_ROWS`（默认 10000），超限拒绝执行并返回明确错误。
6. **按需展开**：仅 URL/headers/body/params 中引用了数据文件变量名的接口才按数据行数展开；无变量引用的接口保持单次执行。`{{baseUrl}}`/`{{base_url}}` 不算数据驱动变量。
7. **变量替换范围**：仅替换 URL/headers/body/params 中的 `{{变量名}}`，不替换 `{{baseUrl}}`/`{{base_url}}`（后者由 base_url 逻辑处理）。
8. **checkpoint 兼容**：展开接口（data_index > 0）的 checkpoint 键为 `(item_path_text, data_index)` 复合键；未展开接口保持原键格式。
9. **性能摘要**：数据驱动模式下，报告中针对展开接口自动生成性能统计摘要（成功率、响应时间 P50/P95/P99、分段趋势、错误分布）。
10. **报告展示**：展开结果增加 `data_index` 字段（未展开接口 data_index=0）；报告中默认按原始接口分组展示。

#### 约束 32：请求串联与变量提取约束

1. **功能开关**：`ENABLE_VARIABLE_EXTRACTION`（默认 `False`），支持环境变量覆盖。
2. **变量上下文生命周期**：仅在 `_execute_api_suite()` 调用期间有效，不持久化到 checkpoint。
3. **x_extract 配置**：放置在 Postman Collection item 的 `request` 对象中，键为变量名，值为 JSONPath 表达式或 `$header.名称`。
4. **JSONPath 安全**：仅支持简化路径（`$.field.subfield`、`$.array[0].field`），禁止 eval/exec/过滤器/通配符。使用自研轻量解析器。
5. **变量优先级**：数据行变量 > 运行时提取变量 > Collection 预置变量 > initial_variables。
6. **选择性执行警告**：选择性执行可能跳过"变量生产"接口，导致后续接口变量缺失。文档必须明确说明。
7. **提取失败行为**：提取失败时变量不写入上下文（不覆盖已有值），日志记录警告，报告中该字段标记为提取失败。

### 5.2 需要修改的现有约束

| 约束编号 | 当前内容 | 需要的修改 |
|---------|---------|-----------|
| 约束 8（URL 解析） | 支持 `{{baseUrl}}` 和绝对 URL | 追加：通用 `{{变量}}` 替换在 base_url 处理之后执行，不干扰 baseUrl 逻辑 |
| 约束 9（SSRF 防护） | 目标 URL 仅允许 http/https | 追加：变量替换后的 URL 同样需要协议校验；校验时机在替换之后、请求之前 |
| 约束 2.5（目录结构） | 列出现有目录 | 新增 `core/variable_context.py`、`utils/data_driver.py`、`utils/extract_utils.py`、`utils/variable_substitution.py` |
| 约束 3.3（报告产物） | 三件套 | 追加：details.json 中结果记录可增加 `data_index`、`extracted_variables` 可选字段 |

以下约束经确认不受影响：
