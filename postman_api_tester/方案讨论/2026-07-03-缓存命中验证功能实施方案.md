# 缓存命中验证功能实施方案

**版本基线**：v1.20.15  
**目标版本**：v1.21.0（MINOR 升级，新增缓存命中验证模式）  
**预计工时**：2-3 小时  
**优先级**：中（新增验证工具，来源于用户测试需求）

---

## 背景

用户需要在服务端缓存场景下验证缓存是否生效。现有工具记录每接口 `response_time_ms`，但每次只执行一次，无法展示缓存效果。

**验证目标**：对同一查询接口发送 N 次请求，比较首次响应时间（miss）与后续响应时间（hit），判断缓存是否生效。

**判断标准**：首次响应时间 ≥ 200ms，后续响应时间 ≤ 20ms，说明缓存生效。

---

## 技术方案

### 方案 A：ad-hoc 页面新增"重复执行"配置（推荐）

在 ad-hoc 页面（`templates/adhoc_run.html`）为每个接口添加**重复次数**输入框，执行时对同一接口发送 N 次请求，报告中展示每次的响应时间对比。

**优势**：
- ad-hoc 页面使用场景更符合"临时验证缓存"的需求
- 改动集中，风险低
- 不破坏现有单接口执行语义

**改动文件**：

| 文件 | 改动 |
|------|------|
| `templates/adhoc_run.html` | 新增"重复次数"输入框（默认 1，范围 1~10） |
| `postman_api_tester/handlers/job_routes.py` | 解析 repeat 字段，展开为多次执行项 |
| `postman_api_tester/executor.py` | `TestResultRecord` 新增 `repeat_index`、`repeat_total` 字段 |
| `templates/report_view.html` | 重复请求分组展示（可展开/收起） |
| `templates/report_details.html` | 单个接口卡片新增"缓存命中验证"对比表格 |
| `postman_api_tester/操作手册.md` | 新增"缓存命中验证"使用章节 |

### 数据流

```
ad-hoc 表单: repeat=3
  → POST /api/run-ad-hoc-tests
  → job_routes 解析 repeat=3，将接口展开为 3 个执行项
  → executor.execute_test() 每次标记 repeat_index (0/1/2)、repeat_total (3)
  → HtmlReporter 渲染时按 repeat_group 分组
  → 报告展示: 第1次 203ms (miss) | 第2次 18ms (hit) | 第3次 15ms (hit)
```

### 报告展示形式

**列表页**：
- 重复接口显示为可展开行（类似分组接口）
- 收起时显示：`接口名 [×3] avg: 79ms p95: 203ms 加速比: 13.5x`
- 展开后显示每次执行明细

**详情页**：
- 单个接口卡片顶部新增"缓存命中验证"区域
- 表格展示：执行序号 | 状态码 | 耗时(ms) | 缓存状态(miss/hit)
- 加速比 = 首次耗时 / 后续平均耗时

---

## 实施步骤

### Step 1：后端数据模型扩展
- `executor.py` 中 `TestResultRecord` 新增 `repeat_index: int`、`repeat_total: int`、`repeat_group: str` 字段
- `postman_api_tester.py` 执行循环中，接口配置 `repeat > 1` 时展开为多次执行，生成 `repeat_group` ID

### Step 2：ad-hoc 前端表单
- `templates/adhoc_run.html` 中每个接口的 Method/URL 行旁新增"重复次数"输入框
- 提交时将 repeat 值写入接口配置

### Step 3：报告页展示
- `templates/report_view.html` 列表页：重复接口分组展示，默认收起
- `templates/report_details.html` 详情页：单个接口卡片顶部新增"缓存命中验证"区域

### Step 4：文档更新
- `操作手册.md` 新增"缓存命中验证"使用章节
- `快速命令参考.md` 新增重复执行参数说明

---

## 验证方式

1. 启动报告服务 `python -m postman_api_tester`
2. 进入 ad-hoc 页面，添加一个组织查询接口
3. 设置重复次数为 5
4. 执行后打开报告，验证：
   - 列表中该接口显示为可展开的重复请求组
   - 展开后 5 次请求各自展示 status_code + response_time_ms
   - 详情页"缓存命中验证"表格正确展示首次 vs 后续耗时对比
