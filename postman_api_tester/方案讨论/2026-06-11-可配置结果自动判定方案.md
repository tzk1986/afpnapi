# 可配置结果自动判定方案 — errCode & message

**提案日期**：2026-06-11
**状态**：方案设计
**基线版本**：v1.3.1

---

## 一、问题背景

当前测试结果判定逻辑为**硬编码**，存在以下限制：

| 维度 | 当前逻辑 | 局限 |
|------|---------|------|
| HTTP 状态码 | `status_code == expected_status`（默认 200） | 合理，无需改 |
| message | 为空或 `"success"`（忽略大小写） | 很多项目 message 返回 `"操作成功"`、`"ok"`、`"提交成功"` 等 |
| errCode | 已从响应提取，**不参与判定** | 很多项目用 `errCode=0` 表示成功，当前完全忽略 |

**直接后果**：测试执行后，大量实际成功的接口因 message 不为 "success" 被标记为 FAILED，用户必须逐条**手动标记成功**，效率低下。

---

## 二、设计目标

1. 新增可配置的 `errCode` 和 `message` 判定条件
2. 执行时**自动判断**结果成功/失败，消除手动标记
3. 支持全局配置 + 集合内接口级覆盖（两层优先级）
4. 不配置时**保持当前默认行为**，完全向后兼容

---

## 三、配置体系

### 3.1 三层优先级

```
集合内接口级配置(x_* 扩展字段) > 全局配置文件(config.py) > 内置默认值
```

### 3.2 全局配置（`config.py` 新增）

```python
# ==============================================================
# 可配置结果自动判定条件
# ENABLE_ERR_CODE_JUDGMENT: 是否启用 errCode 参与结果判定
#   False（默认）= 保持当前行为，不检查 errCode
#   True = errCode 必须匹配 SUCCESS_ERR_CODES 中任一值才判定为成功
# SUCCESS_ERR_CODES: 视为成功的 errCode 值列表（逗号分隔）
#   默认值: "0"
#   示例: SUCCESS_ERR_CODES = "0,200,"  （空字符串也算成功）
# ENABLE_MESSAGE_JUDGMENT: 是否启用 message 参与结果判定
#   True（默认）= 保持当前行为，message 为空或匹配 SUCCESS_MESSAGES 时通过
#   False = 不检查 message
# SUCCESS_MESSAGES: 视为成功的 message 值列表（逗号分隔，忽略大小写）
#   默认值: "success"  （保持当前行为：空字符串或 success 视为成功）
#   示例: SUCCESS_MESSAGES = "success,ok,操作成功"
# ==============================================================
ENABLE_ERR_CODE_JUDGMENT = False
SUCCESS_ERR_CODES = "0"
ENABLE_MESSAGE_JUDGMENT = True
SUCCESS_MESSAGES = "success"
```

### 3.3 集合接口级覆盖

在 Postman Collection item 的 `request` 对象中新增扩展字段：

```json
{
  "request": {
    "method": "POST",
    "url": { "raw": "/api/user/info" },
    "x_expected_status": 200,
    "x_success_err_codes": "0,200",
    "x_success_messages": "success,ok",
    "x_enable_err_code_judgment": true,
    "x_enable_message_judgment": true
  }
}
```

| 扩展字段 | 类型 | 说明 |
|---------|------|------|
| `x_success_err_codes` | string | 逗号分隔的成功 errCode 列表，覆盖全局 `SUCCESS_ERR_CODES` |
| `x_success_messages` | string | 逗号分隔的成功 message 列表，覆盖全局 `SUCCESS_MESSAGES` |
| `x_enable_err_code_judgment` | bool | 是否启用 errCode 判定，覆盖全局 |
| `x_enable_message_judgment` | bool | 是否启用 message 判定，覆盖全局 |

### 3.4 内置默认值（无任何配置时）

| 参数 | 默认值 | 行为 |
|------|-------|------|
| errCode 判定 | **关闭** | 不检查 errCode（保持当前行为） |
| message 判定 | **开启** | message 为空或 `"success"` 时通过（保持当前行为） |
| 状态码判定 | **开启** | HTTP status == expected_status（默认 200，保持当前行为） |

## 四、判定逻辑变更

### 4.1 当前逻辑（`executor.py:241-243`）

```python
status_code_ok = self.resp_status_code == expected_status
normalized_message = str(response_message or "").strip().lower()
message_ok = (normalized_message == "") or (normalized_message == "success")
# 最终: status_code_ok AND message_ok → 进入断言检查 → PASSED/FAILED
```

### 4.2 新逻辑（三个独立维度）

```python
# 维度一：HTTP 状态码（始终检查）
status_code_ok = self.resp_status_code == expected_status

# 维度二：errCode（可选）
err_code_ok = True
if enable_err_code_judgment:
    err_code_ok = err_code.strip() in success_err_codes  # 匹配任一成功值

# 维度三：message（可选）
message_ok = True
if enable_message_judgment:
    normalized_msg = str(response_message or "").strip().lower()
    message_ok = (normalized_msg == "") or (normalized_msg in success_messages)

# 三维度全部通过 → 进入断言检查 → PASSED
# 任一不通过 → FAILED（附带具体不通过原因）
```

### 4.3 失败消息增强

```
# errCode 不匹配时：
'errCode 不满足成功条件(期望为 [0, 200]), 实际返回: 1001'

# message 不匹配时：
'message 不满足成功条件(期望为 [success, ok]), 实际返回: 系统错误'

# 状态码不匹配时（不变）：
'期望状态码: 200, 实际: 500; message: 系统错误'

# 多维度不匹配时，用分号拼接
```

def parse_success_list(config_value: str) -> frozenset[str]:

    """

def evaluate_result_judgment(
    *,
    status_code: int,
    expected_status: int,

## 七、配置解析链路

```
config.py
  SUCCESS_ERR_CODES = "0,200"
      ↓
report_server_config.py
  SUCCESS_ERR_CODES_SET: frozenset = parse_success_list("0,200") → {"0", "200", ""}
      ↓
executor.py 构造时从模块级常量读取（全局默认）
      ↓
execute_test() 检查 api.get("x_success_err_codes") → 有值则覆盖全局
      ↓
调用 judgment_utils.evaluate_result_judgment(...) 使用最终参数
```

## 八、与手动标记功能的关系

**本方案不删除、不替代现有的手动标记成功/失败功能**。两者共存：

| 能力 | 自动判定（本方案新增） | 手动标记（保留） |
|------|----------------------|-----------------|
| 时机 | 执行时自动判定 | 执行后人工修正 |
| 入口 | 全局配置 + 集合扩展字段 + Web 界面 | 报告详情页"标记成功/标记失败/恢复自动结果" |
| 适用场景 | 批量自动化、减少手动操作 | 个别接口的特殊情况修正 |
| 优先级 | 先自动判定 → 生成初始 status | 手动覆盖 → 修改已生成的 status |

执行流程：自动判定生成初始 status → 用户可在报告详情页手动修正 → 修正记录写入 `manual_judgement` + `judgement_history`。

## 九、Web 界面配置（运行时直接设置）

除了全局 config.py 和集合 JSON 扩展字段外，新增 **Web 界面**让用户在执行任务时直接在页面上配置判定条件，无需修改代码文件。

### 9.1 配置入口

| 页面 | 配置位置 | 说明 |
|------|---------|------|
| 首页上传执行表单 | 新增"判定条件"折叠面板 | 上传 collection 执行时可选配置 |
| ad-hoc 直接新增页 | 新增"判定条件"区域 | 录入接口时同步配置判定规则 |
| 报告详情页 | 只读展示当前判定条件 | 展示本次执行使用的判定参数 |

### 9.2 界面设计

首页上传区域新增折叠面板（默认收起）：

```
┌─ 判定条件配置（可选，点击展开）─────────────────────────┐
│                                                         │
│  ☑ 启用 errCode 判定                                    │

### 9.3 数据流

```
Web 界面填写 → POST /api/run-postman (或 /api/run-ad-hoc-tests)
    ↓ 请求体新增 judgment_config 字段
    {

        "success_err_codes": "0,200",

### 9.4 优先级最终定义（四层）

```
Web 界面本次任务配置 > 集合接口级 x_* 字段 > 全局 config.py > 内置默认值
```

## 十、兼容性保障

| 场景 | 行为 |
|------|------|
| 不配置任何新参数 | **完全保持当前行为**（状态码 + message 空/success） |
| 仅配置 `ENABLE_ERR_CODE_JUDGMENT=True` | 新增 errCode 检查，message 检查不变 |
| 仅配置 `SUCCESS_MESSAGES="ok"` | message 为 "ok" 时通过（空字符串仍通过，因为 `message_ok = (msg == "") or (msg in set)`） |
| 集合内某接口配了 `x_success_err_codes="0,200"` | 仅该接口使用自定义值，其余使用全局 |
| ad-hoc 运行 | 通过 `normalize_adhoc_case` 透传新字段 |
| 旧版 collection 无 `x_*` 字段 | 使用全局配置，无影响 |
