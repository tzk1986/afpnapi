# 编辑界面模式切换方案（方案讨论阶段）

**文档状态**：已决策收口（可实施）  
**涉及范围**：人工用例编辑 / 详情界面编辑  
**关联约束**：开发阅读文档.md - 约束 2.2、2.4  
**风险等级**：中（涉及 UI 交互、数据转换、状态管理）

---

## 1. 需求分析

### 1.1 用户需求

1. **模拟 Postman 编辑体验**：支持多种参数编辑方式
2. **模式切换**：可从 JSON 方式切换到关键字/表单方式
3. **双向转换**：两种方式的数据无损互转
4. **编辑便利性**：部分字段不便用 JSON 编辑时有替代方案

### 1.2 当前痛点

| 问题 | 表现 | 影响 |
|------|------|------|
| JSON 唯一方式 | Headers/Params/Body 均为 JSON 文本框 | 新手不熟悉 JSON 格式 |
| 参数众多时复杂 | 大量参数手写 JSON 易出错 | 容易输入语法错误 |
| 无视觉反馈 | 编辑过程无字段提示 | 易遗漏必填参数 |
| 切换成本高 | 要修改参数需全部重写 | 编辑效率低 |

## 2. 方案设计

### 2.1 两种编辑模式

#### 模式 A：JSON 编辑模式（当前）
- **适用场景**：快速导入已有 JSON、批量编辑、高级用户
- **特点**：文本框输入，完全自由格式
- **验证**：实时 JSON 语法检查

#### 模式 B：关键字表单模式（新增）
- **适用场景**：逐字段添加、快速构建、新手用户
- **特点**：表单化 KV 对输入、可视化编辑
- **验证**：字段级验证（类型、格式）

#### 两种模式的数据承载

```
模式 A：JSON 文本
{
  "key1": "value1",
  "key2": 123,
  "key3": { "nested": "value" }
}

        ↓↑ 双向转换（parseJson / stringifyJson）
        
模式 B：表单行列表
┌──────────────┬──────────────┬──────┐
│     Key      │    Value     │ 操作 │
├──────────────┼──────────────┼──────┤
│ key1         │ value1       │ × ✎  │
│ key2         │ 123          │ × ✎  │
│ key3.nested  │ value        │ × ✎  │
└──────────────┴──────────────┴──────┘
[+ 添加字段] [批量编辑]
```

#### 2.3.3 数据转换规则

**JSON → 表单**
```javascript
function jsonToForm(jsonStr) {
  try {
    const obj = JSON.parse(jsonStr || '{}');
    const rows = [];
    function flatten(obj, prefix = '') {
      for (const [k, v] of Object.entries(obj)) {
        const key = prefix ? `${prefix}.${k}` : k;
        if (typeof v === 'object' && v !== null) {
          flatten(v, key);  // 递归扁平化嵌套
        } else {
          rows.push({ key, value: String(v), type: typeof v });
        }
      }
    }
    flatten(obj);
    return rows;
  } catch (e) {
    return [];  // JSON 格式错误返回空
  }
}
```

**表单 → JSON**
```javascript
function formToJson(rows) {
  const obj = {};
  for (const { key, value } of rows) {
    if (!key.trim()) continue;  // 跳过空 key
    const keys = key.split('.');
    let ref = obj;
    for (let i = 0; i < keys.length - 1; i++) {
      if (!(keys[i] in ref)) ref[keys[i]] = {};
      ref = ref[keys[i]];
    }
    const finalKey = keys[keys.length - 1];
    // 尝试解析值的类型
    ref[finalKey] = tryParseValue(value);
  }
  return JSON.stringify(obj, null, 2);
}

function tryParseValue(str) {
  // 优先级：null → boolean → number → string
  if (str === 'null') return null;
  if (str === 'true') return true;
  if (str === 'false') return false;
  if (!isNaN(str) && str !== '') return Number(str);
  return str;
}
```

## 3. 核心流程

### 3.1 初次打开编辑器

```
打开编辑器
├─ 读取 existingCase（如果是编辑）
├─ 初始化模式为 JSON
├─ 显示 JSON 文本框（Headers / Params / Body）
├─ 隐藏表单区域
└─ 显示模式切换器
```

### 3.2 切换到表单模式

```
点击"表单编辑"按钮
├─ 触发事件：switchEditMode('form')
├─ 尝试解析当前 JSON
│  ├─ 成功：jsonToForm() → 渲染表单行
│  └─ 失败：显示错误提示 "JSON 格式错误，无法切换" 
├─ 隐藏 JSON 文本框
├─ 显示表单行列表
├─ 绑定表单事件（增/删/改行）
└─ 当前模式标记为 'form'
```

### 3.3 切换回 JSON 模式

```
点击"JSON 编辑"按钮
├─ 触发事件：switchEditMode('json')
├─ 尝试将当前表单行转 JSON（formToJson()）
├─ 更新 JSON 文本框内容
├─ 隐藏表单行
├─ 显示 JSON 文本框
├─ 当前模式标记为 'json'
└─ 后续修改同步到 JSON 文本框
```

### 3.4 发送请求时

```
点击"发送请求"
├─ 校验基础信息（Name / URL）
├─ 读取当前激活模式数据
│  ├─ 模式 JSON：直接解析 JSON 文本框
│  └─ 模式表单：表单行 → JSON → 解析
├─ 提取 Headers / Params / Body
├─ 调用 /api/proxy-request
└─ 显示响应面板
```

## 4. 风险识别与规避

### 4.1 数据一致性风险

| 风险 | 原因 | 规避方案 |
|------|------|--------|
| JSON 和表单不同步 | 用户同时修改两个模式 | 同一时间仅显示一个模式，隐藏未激活模式 |
| 嵌套对象解析错误 | 深层嵌套格式处理不当 | `flatten/unflatten` 采用点号路径（`a.b.c`），支持递归 |
| 类型丢失 | Value 从 JSON 转文本再解析 | `tryParseValue()` 智能推断（null/bool/number/string） |
| 环形引用崩溃 | JSON.stringify 无法处理环形 | 编辑器中禁止手写环形，依赖原始 Postman 导出的合法数据 |

### 4.2 UI 状态管理风险

| 风险 | 原因 | 规避方案 |
|------|------|--------|
| 编辑器崩溃 | 大量表单行渲染性能问题 | 限制单个参数组最多 200 行，超过显示滚动条 |
| 模式切换闪屏 | 频繁显示隐藏导致 DOM 重排 | 使用 `visibility: hidden` 而非 `display: none` |
| 丢失焦点 | 切换模式后光标位置丢失 | 记录焦点位置，切换后恢复 |
| 表单行 ID 冲突 | 动态添加删除行导致 ID 重复 | 使用 UUID 或时间戳为每行分配唯一 ID |

### 4.3 数据转换风险

| 风险 | 原因 | 规避方案 |
|------|------|--------|
| JSON 解析失败 | 用户输入格式错误 | 在切换前先尝试 `JSON.parse()` 并捕获异常，显示具体错误行号 |
| 循环引用 | 对象自引用 | 限制递归深度 <= 10 层，超过显示警告 |
| 超大数据 | Body 包含数 MB 二进制 | 限制 Body 字段展示长度 10000 字符，超过提示 "数据过大，推荐直接编辑 JSON" |
| 特殊字符编码 | Unicode / 转义符处理不当 | JSON 使用标准编码，表单采用原始文本，转换时保留原始编码 |

### 4.4 交互风险

| 风险 | 原因 | 规避方案 |
|------|------|--------|
| 误删字段 | 表单表单中一键删除整行 | 提供行级"撤销"或全局"重置"功能，记录操作栈 (5 步内) |
| 多行同名 Key | 表单允许重复 Key 导致冲突 | 表单中新增时做去重提示，提示 "该 Key 已存在，覆盖原值" |
| 切换时丢数据 | 用户从表单切回 JSON 前未保存 | 切换前弹确认框 "切换模式可能丢失当前编辑，是否继续？" |
| 关闭编辑器后丢失 | 未保存直接关闭 | 若有修改提示 "您有未保存的修改，是否保存？" |

### 4.5 约束冲突风险

| 冲突点 | 当前约束 | 新方案影响 | 规避 |
|-------|---------|---------|------|
| details.json 安全 | Header 中敏感字段替换为 `***` | 表单编辑时敏感字段可见 | 在表单字段上标记 `[敏感]` 徽章，保存时仍替换 |
| 模板外置约束 | 所有 UI 改动在 `templates/` | 表单 JS 逻辑应内置还是外置 | 表单渲染 JS 和转换函数直接写在模板中，不拆分外置 JS 文件 |
| 路由函数签名一致性 | 函数签名与路由参数必须同步 | 发送请求时需区分模式来源 | `sendManualRequest()` 内部透明处理模式，API 接收格式不变 |

### 4.6 性能风险

| 风险 | 原因 | 规避方案 |
|------|------|--------|
| 大表单卡顿 | 200+ 行表单频繁渲染 | 使用虚拟滚动（只渲染可见行）或分页显示 |
| JSON.parse 阻塞 | 解析超大 Body 主线程卡 | Body 超 1MB 时提示 "建议直接编辑 JSON"，不做自动转换 |
| 切换延迟 | 转换耗时超过 100ms | 显示加载动画，转换逻辑异步化 |

## 8. 决策点与建议

### 8.1 关键决策

| 决策点 | 方案 A | 方案 B | **推荐** |
|--------|--------|--------|---------|
| Body 编辑类型 | 简化子集 | 完整 Postman 六类型 | **完整 Postman 六类型** |
| 大表单渲染 | 虚拟滚动（复杂） | 简单分页 + 滚动（简单） | **简单分页** |
| 模式切换确认 | 无确认 | 有数据时弹确认 | **有确认** |
| 特殊字符处理 | Base64 编码 | 保留原始 + 转义提示 | **原始 + 提示** |

### 8.2 实施优先级

1. **P0 必做**：JSON ↔ 表单双向转换、模式切换器 UI、基础字段表单
2. **P1 应做**：表单行增删、Body 类型选择、数据校验提示
3. **P2 可做**：操作栈撤销、虚拟滚动优化、快捷键支持

### 8.3 交付里程碑
