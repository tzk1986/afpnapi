# 请求/响应对比视图实现方案

**版本基线**：v1.19.1  
**目标版本**：v1.20.0（MINOR 升级，新增前端 UX 功能）  
**预计工时**：4-5 小时  
**优先级**：中（前端 UX 增强）

---

## 一、功能概述

### 1.1 目标
在 Collection 编辑器中支持并排对比两次请求的完整报文（URL、Headers、Body、Response），高亮差异部分，便于排查接口行为变化。

### 1.2 使用场景
- 同一接口两次执行结果对比（如回归测试前后）
- 不同参数配置下的请求/响应差异分析
- 调试时对比预期与实际响应

### 1.3 核心交互
1. 用户在历史记录或执行结果中选择两次请求
2. 点击"对比"按钮
3. 打开双栏对比视图
4. 差异部分高亮显示（新增/删除/修改）
5. 支持同步滚动、折叠/展开

## 二、技术方案

### 2.1 数据存储

**方案选择**：内存存储（推荐）vs 持久化存储

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 内存存储 | 实现简单、无 IO 开销、自动清理 | 刷新页面丢失 | ✅ 采用 |
| 持久化存储 | 跨会话保留 | 需额外存储管理、隐私风险 | ❌ 不采用 |

**实现**：
```javascript
// state 中新增字段
state.compareHistory = [
  {
    id: 'req_001',
    timestamp: '2026-06-26T10:30:00Z',
    request: { url, method, headers, body },
    response: { status, headers, body }
  },
  // ... 最多保留 20 条
];
```

**风险控制**：
- 限制历史记录数量（最多 20 条），超出时 FIFO 淘汰
- 单条记录大小限制（response body > 1MB 时截断）
- 页面刷新时自动清空（无需持久化）

### 2.2 Diff 算法

**方案选择**：行级 diff（推荐）vs 字符级 diff vs JSON 结构 diff

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 行级 diff | 性能好、可读性强 | 细粒度不足 | 文本/JSON/代码 ✅ |
| 字符级 diff | 精确 | 性能差、视觉混乱 | 短文本 |
| JSON 结构 diff | 语义化 | 实现复杂 | 纯 JSON |

**采用方案**：行级 diff + JSON 美化（先 JSON.stringify 再 diff）

**库选择**：
- 使用 `diff` 库（MIT 许可，< 10KB gzip）
- 备选：自实现 LCS（最长公共子序列）算法

**风险控制**：
- 大文本（> 10000 行）时降级为简单对比，避免卡顿
- 二进制响应（图片/PDF）不展示 diff，仅显示元信息

### 2.3 UI 布局

**方案选择**：双栏并排（推荐）vs 单栏切换 vs 内联 diff

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 双栏并排 | 直观、同步滚动 | 窄屏显示受限 | ✅ 采用 |
| 单栏切换 | 节省空间 | 需频繁切换 | ❌ 不采用 |
| 内联 diff | 紧凑 | 可读性差 | ❌ 不采用 |

**布局结构**：
```
┌─────────────────────────────────────────────┐
│ 对比视图标题栏（关闭按钮、同步滚动开关）      │
├──────────────────┬──────────────────────────┤
│ 请求 A 信息栏    │ 请求 B 信息栏             │
│ (时间/状态/URL)  │ (时间/状态/URL)           │
├──────────────────┼──────────────────────────┤
│                  │                          │
│ 请求/响应内容    │ 请求/响应内容             │
│ (差异高亮)       │ (差异高亮)                │
│                  │                          │
├──────────────────┴──────────────────────────┤
│ 差异统计（+5 行 / -3 行 / ~2 行修改）        │
└─────────────────────────────────────────────┘
```

**Tab 切换**：
- URL & Method
- Request Headers
- Request Body
- Response Status & Headers
- Response Body

### 2.4 差异高亮

**配色方案**（参考 GitHub diff）：
- 新增行：`background: #e6ffed; color: #22863a;`
- 删除行：`background: #ffeef0; color: #cb2431;`
- 修改行（删除 + 新增）：分别用上述两色

**行号显示**：
- 左右两栏独立行号
- 新增行在对侧显示空白占位

**风险控制**：
- 高亮仅应用于行级，不深入字符级（避免性能问题）
- 超长行（> 500 字符）截断并提示"行过长，已折叠"

## 三、实现步骤

### Phase 1：数据层（1 小时）
1. `state` 新增 `compareHistory` 数组
2. `sendCurrentRequest()` 中请求成功后保存到历史
3. 限制历史记录数量（FIFO 淘汰）
4. 提供 `getCompareHistory()` 读取接口

**验收标准**：
- ✅ 每次请求自动保存到历史
- ✅ 历史记录不超过 20 条
- ✅ 刷新页面后历史清空

### Phase 2：UI 入口（0.5 小时）
1. 执行结果区域新增"历史"下拉按钮
2. 下拉列表显示最近 20 次请求（时间 + URL + 状态码）
3. 支持勾选两条记录
4. 勾选后显示"对比"按钮

**验收标准**：
- ✅ 下拉列表正常显示历史
- ✅ 最多勾选 2 条
- ✅ 勾选 2 条后"对比"按钮可用

### Phase 3：Diff 计算（1 小时）
1. 引入 `diff` 库（或自实现 LCS）
2. 实现 `diffLines(oldText, newText)` 函数
3. JSON 对象先 `JSON.stringify(obj, null, 2)` 再 diff
4. 返回差异结构：`[{ type: 'added'|'removed'|'unchanged', value: string }]`

**验收标准**：
- ✅ 文本 diff 正确识别新增/删除/未变行
- ✅ JSON 对象 diff 前先美化
- ✅ 大文本（> 10000 行）降级处理

### Phase 4：对比视图渲染（2 小时）
1. 创建对比模态框（全屏 overlay）
2. 双栏布局（CSS Grid 或 Flexbox）
3. 渲染 diff 结果（行级高亮）
4. 实现同步滚动（scroll 事件联动）
5. Tab 切换（URL/Headers/Body 等）
6. 差异统计栏

**验收标准**：
- ✅ 双栏并排显示
- ✅ 差异行高亮正确
- ✅ 同步滚动可开关
- ✅ Tab 切换流畅
- ✅ 关闭按钮正常工作

### Phase 5：边界处理与优化（0.5 小时）
1. 空响应处理（显示"(empty)"占位）
2. 二进制响应处理（显示元信息，不 diff）
3. 超长行截断
4. 键盘快捷键（ESC 关闭、←→ 切换 Tab）
5. 响应式布局（窄屏时提示"请在宽屏下使用"）

**验收标准**：
- ✅ 空响应不报错
- ✅ 二进制响应友好提示
- ✅ 超长行可展开/折叠
- ✅ ESC 键关闭对比视图

## 四、风险控制清单

### 4.1 已识别风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| TreeWalker DOM 修改导致遍历中断 | 搜索功能失效 | 高 | 两阶段处理（先收集再修改）✅ 已修复 |
| 大文本 diff 性能问题 | 页面卡顿 | 中 | 行数 > 10000 时降级为简单对比 |
| 内存泄漏（历史记录未清理） | 内存占用增长 | 低 | FIFO 淘汰 + 页面刷新清空 |
| 同步滚动死循环 | 滚动卡顿 | 中 | 加锁机制（scrolling 标志位） |
| 二进制响应 diff 失败 | 页面崩溃 | 低 | 类型检测 + 降级显示 |

### 4.2 历史问题复盘

**问题 1：JSON 搜索子串匹配不完整**
- 原因：TreeWalker 遍历时修改 DOM，导致后续节点被跳过
- 修复：改为两阶段（先收集匹配节点，再统一高亮）
- 本次应用：diff 渲染时同样采用两阶段（先计算 diff，再批量渲染）

**问题 2：版本号遗漏更新**
- 原因：多处版本引用分散，更新时遗漏
- 修复：统一 grep 检查 + 文档版本同步清单
- 本次应用：功能完成后执行版本同步检查清单（见 §5.3）

**问题 3：输入长度验证缺失**
- 原因：未对所有用户输入做长度限制
- 修复：BaseHandler 统一验证方法
- 本次应用：对比历史记录时验证 URL/Headers 长度（防 DoS）

### 4.3 测试策略

**单元测试**（新增 `tests/test_compare_history.py`）：

## 六、降级策略

如时间不足，按以下优先级裁剪：

1. **必做**：数据层 + UI 入口 + 基础 diff（Phase 1-3）
2. **应做**：对比视图渲染（Phase 4）
3. **可延后**：边界处理与优化（Phase 5）
4. **可放弃**：同步滚动、键盘快捷键

**最小可用版本**（MVP）：
- 保存历史记录
- 选择 2 条打开对比
- 简单文本 diff（无高亮）
- 单栏切换（非双栏）
