# 日志平台映射建议（Prometheus / ELK）

发布日期：2026-05-13
文档定位：提供结构化日志字段到 Prometheus 指标与 ELK 检索字段的统一映射口径，降低接入与看板建设成本。

## 1. 设计原则

1. 统一以 `event` 作为核心维度，其他字段作为标签或过滤条件。
2. Prometheus 只保留低基数标签，避免高基数字段直接作为标签。
3. ELK 保留原始字段，按 `event`、`status_code`、`job_id`、`report_name` 建议建立索引模板与常用查询。

## 2. 字段到指标名对照表

| 日志字段 | Prometheus 指标名 | 类型 | 推荐标签（labels） | ELK 字段建议 | 说明 |
|---|---|---|---|---|---|
| `event` | `postman_log_events_total` | Counter | `event`, `level`, `service` | `event.keyword` | 事件总量主指标 |
| `level` | `postman_log_events_total` | Counter 标签 | `level` | `level.keyword` | 建议聚合 `INFO/WARNING/ERROR` |
| `logger` | `postman_log_events_total` | Counter 标签（可选） | `logger` | `logger.keyword` | logger 基数可控时开启 |
| `status_code` | `postman_http_requests_total` | Counter | `method`, `path_tpl`, `status_class` | `status_code` | `status_class` 建议按 `2xx/4xx/5xx` 聚合 |
| `duration_ms` | `postman_http_request_duration_ms` | Histogram | `method`, `path_tpl` | `duration_ms` | 用于 P50/P95/P99 统计 |
| `request_id` | 不建议做指标标签 | - | - | `request_id.keyword` | 高基数，仅用于 Trace 检索 |
| `job_id` | `postman_job_runs_total` | Counter | `event`, `result` | `job_id.keyword` | 指标只统计结果，不直接打 `job_id` 标签 |
| `report_name` | `postman_report_ops_total` | Counter | `event`, `scope` | `report_name.keyword` | 导出/删除等报告操作统计 |
| `status` | `postman_test_results_total` | Counter | `status`, `method` | `status.keyword` | 用于通过率、失败率 |
| `response_time_ms` | `postman_test_response_time_ms` | Histogram | `status`, `method` | `response_time_ms` | 用于接口执行耗时分布 |
| `error_count`（health） | `postman_log_error_count_window` | Gauge | `window_seconds` | `log_alert.error_count` | 来自 `/health.log_alert` 快照 |
| `current`（health） | `postman_log_error_rate_per_min` | Gauge | `window_seconds` | `log_alert.current` | 当前 ERROR 速率 |
| `threshold`（health） | `postman_log_error_rate_threshold_per_min` | Gauge | `window_seconds` | `log_alert.threshold` | 告警阈值 |
| `alert_triggered`（health） | `postman_log_alert_triggered` | Gauge | `metric` | `log_alert.alert_triggered` | 建议布尔映射 `false=0/true=1` |

> `path_tpl` 建议使用路由模板（如 `/api/report-results`），避免把完整 URL path 直接作为高基数标签。

## 3. Prometheus 建议采集口径

## 3.1 指标命名

1. 统一前缀：`postman_`。
2. Counter 以 `_total` 结尾。
3. Histogram 使用 `_bucket/_sum/_count` 标准族。
4. 从 `/health` 映射来的快照建议用 Gauge。

## 3.2 日志转指标建议

1. 事件计数：按 `event` 聚合写入 `postman_log_events_total`。
2. HTTP 请求：从 `event=http.access.logged` 生成请求总量与耗时直方图。
3. 测试结果：从 `event=test.run.executed` 生成通过/失败计数与响应时延。
4. 作业执行：从 `job.run.started/completed/failed` 生成生命周期指标。
5. 健康告警：定时抓取 `/health`，将 `log_alert` 映射到 Gauge 指标。

## 3.3 示例 PromQL

```promql
# 5 分钟 ERROR 日志速率
sum(rate(postman_log_events_total{level="ERROR"}[5m]))

# HTTP 5xx 比例
sum(rate(postman_http_requests_total{status_class="5xx"}[5m]))
/
sum(rate(postman_http_requests_total[5m]))

# 接口执行失败率
sum(rate(postman_test_results_total{status=~"FAILED|ERROR"}[5m]))
/
sum(rate(postman_test_results_total[5m]))

# HTTP P95 耗时（毫秒）
histogram_quantile(0.95, sum(rate(postman_http_request_duration_ms_bucket[5m])) by (le))
```

## 4. ELK 映射建议

## 4.1 索引字段建议

1. 聚合字段：`event.keyword`、`level.keyword`、`logger.keyword`、`status.keyword`。
2. 检索字段：`request_id.keyword`、`job_id.keyword`、`report_name.keyword`。
3. 数值字段：`duration_ms`、`response_time_ms`、`status_code`、`log_alert.current`、`log_alert.threshold`。

## 4.2 常用 KQL 查询示例

```kql
# 任务失败事件
event.keyword : "job.run.failed"

# HTTP 请求错误（4xx/5xx）
event.keyword : "http.access.logged" and status_code >= 400

# 指定任务链路追踪
job_id.keyword : "<job_id>"

# 告警触发快照
log_alert.alert_triggered : true
```

## 4.3 看板建议

1. 事件趋势：按 `event.keyword` 的每分钟计数折线图。
2. 错误热力：按 `event.keyword + level.keyword` 的热力图。
3. HTTP 质量：`status_code` 分布 + `duration_ms` 分位图。
4. 测试质量：`status.keyword` 占比饼图 + `response_time_ms` 趋势。
5. 告警态势：`log_alert.current` 与 `log_alert.threshold` 双轴趋势。

## 5. 告警规则建议（Prometheus + ELK）

1. ERROR 速率告警（Prometheus）：`sum(rate(postman_log_events_total{level="ERROR"}[5m])) > 0.2` 持续 10 分钟。
2. HTTP 5xx 比例告警（Prometheus）：`5xx_ratio > 0.05` 持续 5 分钟。
3. 任务失败突增（ELK Watcher/Kibana Alert）：`event=job.run.failed` 10 分钟计数超过历史 P95。
4. 健康阈值告警：`postman_log_alert_triggered == 1` 持续 3 个采集周期。

## 6. 落地顺序建议

1. 先接入 `event` 主指标与 `http.access.logged` 两条链路。
2. 再补 `test.run.executed` 和 `job.run.*` 生命周期指标。
3. 最后接入 `/health.log_alert`，作为轻量兜底告警信号。
4. 持续治理高基数字段，避免将 `request_id/job_id/report_name` 直接作为 Prometheus 标签。
